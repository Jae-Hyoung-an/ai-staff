## 2.8 벤더 소속 기사 대상 물량 분배 (P0)

### 기본 정책

| 정책 | 설명 |
|------|------|
| **오더 배정 범위** | 벤더 소속 기사에게는 해당 벤더가 연결된 zone의 오더만 배정 |
| **원가 산출 기준** | 벤더가 연결된 Zone에 적용된 요금제 서버의 `기사 원가`(50m당 금액, 픽업비용, 배송완료 비용) 기준으로 원가를 계산하여 기사에게 제안 |
| **배차 우선도 조정** | 벤더에게 분배된 세트/슬롯별 잔여 물량이 많을수록 해당 벤더 소속 기사에 대한 배차 우선도 상향 (목표 충족 우선) |
| **물량 초과 후 처리** | 분배된 세트/슬롯 물량을 모두 수행한 이후에도 동일한 원가 기준으로 오더 배차 지속 (차단 없음) |
| **벤더 상태 조건** | 벤더 상태가 운영중(Active)일 때만 소속 기사에게 오더 배정. 비운영 벤더(대기/운영중지/해지) 소속 기사는 배정 대상에서 제외 |
| **데이터 참조 관계** | 기사 → 벤더 → Zone(세트분배 관리) → 관리정책(권역관리) + 요금제 서버(기사 원가) 순으로 설정값 참조 |

---

### 데이터 참조 흐름

```
기사(벤더 소속)
  └─ 벤더 (상태: 운영중 필수)
       ├─ Zone 연결 (세트분배 관리 2.7)
       │    ├─ 벤더별 세트 수 (분배 물량)
       │    └─ 적용 기사원가 (요금제 서버 조회: 50m당, 픽업, 배송완료)
       └─ 관리정책 연결 (권역관리 2.3)
            └─ 세트 물량 구성 (7x5 매트릭스 → 슬롯별 목표 오더 수)
```

---

### 시스템 동작 시나리오

> **시나리오 1: 벤더 소속 기사에게 오더 배정**
> 1. 배차 엔진이 오더 배정 대상 기사를 탐색한다.
> 2. 벤더 소속 기사의 경우, 해당 벤더의 상태가 운영중인지 확인한다.
> 3. 벤더 상태가 운영중이 아닌 경우(대기/운영중지/해지) 해당 벤더 소속 기사를 배정 대상에서 제외한다.
> 4. 벤더 상태가 운영중인 경우, 해당 벤더가 연결된 zone을 확인한다.
> 5. 오더의 zone과 벤더 연결 zone이 일치하는 경우에만 해당 기사를 배정 후보로 포함한다.

> **시나리오 2: 원가 계산 및 제안**
> 1. 배차 엔진이 벤더 소속 기사에게 오더를 제안할 때, 벤더가 연결된 Zone의 적용 기사원가(요금제 서버)를 조회한다.
> 2. 요금제 서버의 기사 원가(50m당 금액, 픽업비용, 배송완료 비용)를 기준으로 해당 오더의 원가를 계산한다.
> 3. 계산된 원가를 기사에게 제안한다.

> **시나리오 3: 잔여 물량 기반 배차 우선도 조정**
> 1. 배차 엔진이 현재 시각의 요일/시간대를 기준으로 해당 슬롯을 식별한다.
> 2. 해당 슬롯의 목표 오더 수(관리정책 세트 물량 구성)와 벤더의 분배 세트 수를 참조한다.
> 3. 현재까지 수행된 오더 수를 차감하여 잔여 물량을 산정한다.
> 4. 잔여 물량이 많은 벤더의 소속 기사에게 배차 우선도를 높여 목표 충족을 우선한다.

> **시나리오 4: 물량 초과 후에도 배차 지속**
> 1. 벤더에게 분배된 세트/슬롯의 목표 물량을 모두 수행한다.
> 2. 이후에도 동일한 원가 기준(요금제 서버 기사 원가)으로 오더가 배차된다.
> 3. 물량 초과를 이유로 배차가 차단되지 않는다.

## So What / Next Step

- 요금제 서버(기사 원가)와 2.7 세트분배(벤더별 세트 수), 2.3 권역관리(세트 물량 구성)의 설정값이 배차 엔진에서 실시간으로 참조되므로, **설정 변경 시 배차 엔진 반영 타이밍**을 명확히 해야 운영 혼선을 방지할 수 있습니다. 특히 기사 원가는 요금제 서버에서 별도 관리되므로, **배차 엔진이 요금제 서버 API를 직접 호출하는 연동 방식** 설계가 필요합니다.
- 물량 초과 후에도 배차가 지속되므로, **초과 물량에 대한 모니터링/알림** 기능을 추후 고려하면 운영 가시성을 높일 수 있습니다.
