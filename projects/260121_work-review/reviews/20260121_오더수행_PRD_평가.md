## 👎 아쉬운 점 (신랄하게)

### 1. 성공 지표 완전 부재 🚨

**문제**: 이 기능이 성공했는지 판단할 기준이 **단 하나도 없다**.

> *"좋은 PRD는 비즈니스 의사결정을 문서화하고, 나쁜 PRD는 의사결정을 개발자에게 떠넘긴다."*

774줄의 문서에서 KPI, 목표 수치, 측정 방법에 대한 언급이 **0건**이다.

**구체적 문제**:
- 수락률을 높이고 싶다면서 **목표 수락률이 몇 %인지 없음**
- 기사가 수행에 집중하게 만든다면서 **집중도를 어떻게 측정할지 없음**
- 데이터 정합성이 중요하다면서 **정합성 기준이 무엇인지 없음**

**PRD 기준 위반**:
> "성공 지표가 핵심/보조/가드레일로 계층화되어 있는가?" → ❌ 아예 없음

---

### 2. 사용자 스토리 부재 🚨

**문제**: "누가, 왜 이 기능을 사용하는가"에 대한 맥락이 **없다**.

문서 전체가 "~한다", "~가능하다"의 기능 나열이다. 기사가 **왜** 이런 기능이 필요한지, 어떤 **페인포인트**를 해결하는지 설명이 없다.

**나쁜 예시 (현재 PRD)**:
```
"기사는 신규 오더 제안 여부를 선택할 수 있다."
```

**좋은 예시 (있어야 할 내용)**:
```
[배달 기사]로서, [체력 관리와 수입 극대화를 위해] 
[원하는 시점에만 오더를 받고 싶다].

현재 문제: 프렌즈 앱에서는 오더 제안을 거부할 수 없어 
원치 않는 오더에 계속 노출됨 → 피로도 증가
```

**PRD 기준 위반**:
> "사용자 스토리로 '누가, 왜' 사용하는지 설명되는가?" → ❌ 설명 없음

---

### 3. 모호한 정책 표현 (안티패턴) 🚨

**문제**: "논의 필요", "N회", "N미터" 같은 **미확정 값이 10개 이상** 존재한다.

| 위치 | 모호한 표현 | 문제점 |
|------|------------|--------|
| 배너 | "최대 동시 노출 가능 배너의 수: n개 (논의 필요)" | 개발 시작 불가 |
| 상점도착 | "초기값 논의 필요 (100 미터)" | 괄호 안 힌트가 본문보다 구체적 |
| 배차취소 | "일별로 정해진 횟수만큼만 배차 취소가 가능하다" "초기값 논의 필요 (N회)" | 핵심 정책이 미정 |
| 자동Off | "일정 시간동안" | 몇 분? 몇 시간? |

**PRD 기준 위반**:
> "각 조건(if)에 대한 결과(then)가 명확한가?" → ❌
> "숫자/기준이 구체적인가? (적절히 → 3회, 빠르게 → 30분 이내)" → ❌

**문서 상태가 "리뷰중"이라지만**, 2025년 12월 5일 초안 작성 후 **6주가 지났는데도** 핵심 정책값이 미정인 것은 문제다.

---

### 4. 문서 구조 혼란 🚨

**문제**: PRD 본문(270줄) + 요약(80줄) + **회의 녹취록(400줄)**이 한 문서에 섞여 있다.

774줄 중 **절반 이상이 회의 받아쓰기**다. 이건 PRD가 아니라 회의록이다.

```
녹는다를 담은 내용이라고 보입니다. 그래서 잠시만요.
...
형님 네? 기존에는, 기존 앱이랑 프렌즈 앱에서는 3점 도착 취소를 할 수 있었거든요?
...
네 그러면 여기서 마쳐도 괜찮을까요?
```

**문제점**:
- 개발자가 읽을 때 **어디가 확정 정책인지 구분 불가**
- 회의에서 나온 의견 vs 최종 결정이 혼재
- 문서 유지보수 불가능

**PRD 기준 위반**:
> "구현 상세(API, UI 인터랙션)가 별도 문서로 분리되었는가?" → 회의록조차 분리 안 됨

---

### 5. 예외 처리 정책 미흡

**문제**: Happy Path는 상세하지만, **Edge Case와 Error Flow 처리가 부실**하다.

| 케이스 | 현재 상태 | 문제점 |
|--------|----------|--------|
| GPS 오류 | "bypass 할 수 있도록 한다" | bypass 조건, 로그 처리 미정의 |
| 묶음오더 취소 | "제가 오늘 한번 다시 생각하고 말씀드리겠습니다" | 회의에서 보류 후 미해결 |
| 반납오더 | "조금 더 고민하고 말씀드리겠습니다" | 회의에서 보류 후 미해결 |
| 과적오더 | "제한 대상에서 어떻게 제외를 하지?" | 질문만 있고 답변 없음 |

**PRD 기준 위반**:
> "예외 상황의 비즈니스 처리 방향이 정의되었는가?" → ❌ 미정의

---

### 6. 우선순위 단일화

**문제**: 모든 기능이 **P0 (Must have)** 하나로만 분류되어 있다.

12개 기능이 전부 P0다. 이건 우선순위가 아니라 **우선순위 포기**다.

| 현재 | 문제점 |
|------|--------|
| 배너 표시: P0 | 배너 없으면 앱 출시 불가? |
| 내비게이션 연결: P0 | 내비 연결 없으면 출시 불가? |
| 묶음오더 표시: P0 | MVP에 묶음오더가 필수? |

**PRD 기준 위반**:
> "우선순위(MoSCoW)가 명확히 정의되고 합의되었는가?" → ❌ P0만 존재

---

### 7. 비기능적 요구사항 부재

**문제**: 성능, 보안, 규제 요구사항이 **없다**.

| 영역 | 현재 상태 |
|------|----------|
| 성능 | 응답 시간 기준 없음 |
| 보안 | 개인정보(고객 주소, 전화번호) 처리 정책 없음 |
| 규제 | 위치정보 수집 동의, 개인정보보호법 준수 언급 없음 |

기사 위치를 계속 추적하고, 고객 주소/전화번호를 다루는 앱인데 **보안/규제 언급이 0건**이다.

---

### 8. DoR (Definition of Ready) 미충족

개발 시작 가능 기준 체크:

| 조건 | 충족 여부 |
|------|----------|
| 사용자 스토리와 시나리오가 정의되어 있다 | ❌ |
| 비즈니스 정책이 구체적 숫자/조건으로 명시되어 있다 | ❌ (논의필요 다수) |
| 성공 지표(핵심/보조/가드레일)가 정의되어 있다 | ❌ |
| 우선순위(MoSCoW)가 합의되어 있다 | ❌ (P0만 존재) |
| 디자인 문서 링크가 연결되어 있다 | ❌ |
| 이해관계자 리뷰가 완료되어 있다 | ⚠️ (리뷰중) |

**결론**: 이 PRD로는 **개발을 시작할 수 없다**.

```markdown
## 성공 지표

### 핵심 지표 (Core)

### 보조 지표 (Usage)

### 가드레일 지표 (Guardrail)
- 배차 취소율: 현재 수준 **유지** (5% 이하)
- 기사 앱 관련 CS 문의: **증가 없음**
- 앱 크래시율: **0.5% 이하**
```

### 6. 예외 케이스 명확화 (권장)

```markdown
## Edge Cases

### GPS 오류 시 상점 도착 처리
| 조건 | 처리 |
|------|------|
| GPS 신호 없음 | "GPS 확인 필요" 안내 후 **30초 대기** |
| 30초 후에도 신호 없음 | "강제 도착 처리" 버튼 활성화 |
| 강제 도착 처리 시 | 로그에 `force_arrival=true` 기록 |
| 강제 도착 처리 비율 | 월간 **5% 초과 시** 운영팀 리뷰 |

### 묶음오더 부분 취소
| 조건 | 처리 |
|------|------|
| 2건 묶음 중 1건 취소 | 남은 1건 단독 배송료로 **재계산** |
| 재계산 결과 | 기존 묶음 배송료 **이상 보장** |
| 기사 취소 시 | 취소 건만 제외, 나머지 수행 순서 유지 |
```

## 결론

이 PRD는 **기능 나열 문서**이지, **제품 요구사항 문서**가 아니다.

- "무엇을 만든다"는 있지만
- "왜 만드는가"가 없고
- "성공했는지 어떻게 아는가"도 없다

**6주간 리뷰 중 상태**인데도 핵심 정책값이 "논의 필요"로 남아있고, 회의록이 본문과 뒤섞여 있다. 이 상태로는 개발팀이 "이거 어떻게 해요?"라고 계속 물어볼 수밖에 없다.

> *"좋은 PRD는 비즈니스 판단을 문서화하고, 나쁜 PRD는 판단을 개발자에게 떠넘긴다."*

현재 이 PRD는 후자에 가깝다.
