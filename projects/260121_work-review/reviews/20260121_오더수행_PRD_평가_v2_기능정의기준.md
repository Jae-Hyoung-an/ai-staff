## 항목별 상세 평가

### 1. 정책 명확성 (12/25) ⚠️

**잘된 점:**
- 영향범위(신규앱/인트라/매니저PC) 구분 명확
- 상태 변경 조건 대부분 정의
- 픽업 전/후 취소 구분

**아쉬운 점:**

| 위치 | 모호한 표현 | 문제 |
|------|------------|------|
| 배너 | "최대 n개 (논의 필요)" | 개발 불가 |
| 상점도착 | "초기값 논의 필요 (100미터)" | 괄호 힌트가 본문보다 구체적 |
| 배차취소 | "일별 정해진 횟수" "N회" | 핵심 정책 미정 |
| 자동Off | "일정 시간동안" | 몇 분인지 불명확 |
| GPS bypass | "bypass 할 수 있도록" | 조건/로깅 미정의 |

**기준**: 개발자가 추가 질문 없이 구현 가능해야 함 → ❌ 불충족

---

### 2. 시나리오 완성도 (10/20) ⚠️

**잘된 점:**
- Happy Path (정상 수행 플로우) 상세
- 기본적인 상태 전이 정의

**아쉬운 점:**

| 시나리오 유형 | 상태 |
|-------------|------|
| Happy Path | ✅ 있음 |
| Edge Case | ⚠️ 일부만 (묶음오더 "고민 중") |
| Error Flow | ❌ 거의 없음 |

**구체적 미비 사항:**
- GPS 오류 시 처리: "bypass" 언급만, 조건 없음
- 네트워크 오류 시 처리: 없음
- 결제 실패 시 처리: 간략
- 묶음오더 부분 취소: "고민하고 말씀드리겠습니다" (미결)
- 반납오더 처리: "고민하고 말씀드리겠습니다" (미결)

---

### 3. 우선순위 관리 (8/15) ⚠️

**잘된 점:**
- P0로 핵심 기능 식별 시도
- 영향범위별 구분

**아쉬운 점:**
- **12개 기능이 전부 P0** → 우선순위가 아님
- Must/Should/Could/Won't 구분 없음
- 일정 지연 시 어떤 기능을 뺄지 판단 불가

| 기능 | 현재 | 실제 중요도 (추정) |
|------|------|------------------|
| 배너 표시 | P0 | Should |
| 내비게이션 연결 | P0 | Should |
| 묶음오더 UI | P0 | Could |
| 오더 제안 On/Off | P0 | **Must** (차별점) |
| 배송 완료 처리 | P0 | **Must** (필수) |

---

### 4. 문서 구조 (5/15) ❌

**문제:**
- 총 774줄 중 **400줄 이상이 회의 녹취록**
- PRD 본문과 회의 내용이 구분 없이 혼재
- 확정 정책 vs 논의 중 의견 구분 불가

**회의록 예시 (문서에 혼재):**
```
녹는다를 담은 내용이라고 보입니다. 그래서 잠시만요.
...
형님 네? 기존에는, 기존 앱이랑 프렌즈 앱에서는 3점 도착 취소를 할 수 있었거든요?
...
네 그러면 여기서 마쳐도 괜찮을까요?
```

**결과:**
- 개발자가 "이게 확정인가요?" 질문 반복
- 정책 찾기 위해 700줄+ 스크롤 필요
- 리뷰어 피로도 증가

---

### 5. 비기능 요구사항 (5/15) ❌

**현재 상태:**

| 영역 | 정의 여부 |
|------|----------|
| 성능 (응답 시간) | ❌ 없음 |
| 보안 (개인정보 처리) | ❌ 없음 |
| 규제 (위치정보법 등) | ❌ 없음 |

**문제:**
- 고객 주소/전화번호 처리 정책 없음
- 기사 위치 수집 동의 절차 없음
- API 응답 시간 기준 없음

**기능 정의 PRD라도 필요한 이유:**
- 개발자가 "몇 초 안에 응답해야 하나요?" 질문
- QA가 "개인정보 마스킹 필요한가요?" 질문
- 보안팀이 "위치정보 수집 동의 받나요?" 질문

### 6. 변경 이력 (8/10) ✅

**잘된 점:**

**아쉬운 점:**

- 이해관계자 합의 여부 불명확
