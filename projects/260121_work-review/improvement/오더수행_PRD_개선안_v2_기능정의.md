## 2. 기능 정책

### 2.1 오더 제안 정책

| 정책 | 규칙 |
|------|------|
| 기본 상태 | **Off** (앱 실행 시) |
| 제안 대상 | operation type = NVP인 상점 오더만 |
| 자동 Off | On 상태에서 **30분간** 액션 없으면 자동 Off |
| 자동 Off 시 | "오더 제안이 Off되었습니다" 팝업 |
| 안전운행 안내 | On 전환 시 **1일 1회** 안전운행 팝업 |
| 최대 동시 수행 | **3건** |

#### 오더 제안 카드 정보

| 영역 | 표시 정보 |
|------|----------|
| 지도 | 상점 위치 마커 |
| 바텀시트 | 상점명, 원가, 직선거리, 배송수단, 프로모션 태그 |
| 푸시 | 오더 알림 (소리/진동) |

### 2.2 상점 도착 정책

| 정책 | 규칙 |
|------|------|
| 도착 가능 반경 | 상점 좌표 기준 **150m 이내** |
| 반경 외 터치 | "상점 근처에서 도착 처리해주세요" 토스트 |
| GPS 오류 | 60초 대기 후 **[강제 도착]** 버튼 활성화 |
| 강제 도착 시 | 로그: `force_arrival=true`, `gps_status`, `timestamp` |

### 2.3 배차 취소 정책

| 정책 | 규칙 |
|------|------|
| 픽업 전 취소 | 기사 직접 취소 가능 |
| 픽업 후 취소 | **VCC 통해서만** 가능 |
| 일일 취소 횟수 | **최대 5회** |
| 횟수 리셋 | 매일 00:00 |
| 초과 시 | "VCC에 문의해주세요" + 전화 버튼 |
| 취소 시 안내 | "배차 취소 시 오더 배정에 불이익이 있을 수 있습니다" |

**취소 사유 (필수 선택)**:
1. 상점 영업 종료/품절
2. 개인 사정
3. 배달 불가 지역
4. 기타 (직접 입력)

### 2.4 배송 완료 정책

| 오더 유형 | 처리 방식 |
|----------|----------|
| 선불 오더 | 사진 촬영 → 배송 완료 |
| 후불 오더 | 결제 완료 → 자동 배송 완료 |
| 사진 필수 오더 | 올리브영 등 (상점 설정) - 스킵 불가 |
| 위치 검증 오더 | 올리브영 등 - bypass 가능, 로그 기록 |

**배송 완료 후**:
- M캐시 입금 (금액 표시)
- 완료 바텀시트 **3초 후** 자동 닫힘
- "아직 배달할 오더가 남아있어요!" (잔여 오더 있으면)

### 2.5 묶음오더 정책 (Could Have)

| 정책 | 규칙 |
|------|------|
| 표시 방식 | 동일 상점 오더끼리 묶음 표시 |
| 부분 취소 | 개별 오더 단위 취소 가능 |
| 취소 후 배송료 | 남은 오더 단독 배송료로 재계산 (**기존 묶음 배송료 이상 보장**) |

### 2.6 권한 정책

| 사용자 | 권한 |
|--------|------|
| NVP 기사 | 수락/거절, 상태 변경, 취소 (픽업 전) |
| 인트라 담당자 | 배너 등록, 오더 상태 강제 변경 |
| 매니저 PC/M | NVP 기사 **조회만** 가능, 배차/변경 **불가** |
| VCC | 픽업 후 취소, 긴급 상황 대응 |

## 3. 시나리오

### 3.1 Happy Path: 단건 오더 수행

```
1. 앱 실행 → 메인화면
2. [오더 제안 On] 터치
3. (안전운행 팝업 - 1일 1회)
4. 오더 제안 푸시 수신
5. 지도에 상점 마커, 바텀시트에 상점 정보
6. [수락] → 오더 배정
7. 지도에 상점까지 직선, 조리완료 잔여시간
8. 상점 150m 내 진입 → [상점 도착] 활성화
9. [상점 도착] → 픽업번호, 상품내역 표시
10. [픽업 완료] → 도착지 정보 표시
11. 도착지 도착 → [사진 촬영] → [배송 완료]
12. M캐시 입금, 완료 바텀시트 (3초 후 닫힘)
```

### 3.2 Edge Cases

#### EC1: GPS 오류
| 상황 | 처리 |
|------|------|
| 150m 밖에서 도착 터치 | 토스트: "상점 근처에서 도착 처리해주세요" |
| GPS 신호 없음 30초 | 팝업: "GPS 확인 필요" |
| GPS 신호 없음 60초 | **[강제 도착]** 버튼 활성화 |
| 강제 도착 선택 | 로그 기록 후 도착 처리 진행 |

#### EC2: 배차 취소 횟수 초과
| 상황 | 처리 |
|------|------|
| 5회 사용 완료 | 팝업: "오늘 취소 가능 횟수를 모두 사용했습니다" |
| 추가 취소 필요 | "VCC에 문의해주세요" + VCC 전화 버튼 |

#### EC3: 자동 Off
| 상황 | 처리 |
|------|------|
| On 후 30분간 수락/거절 없음 | 자동 Off 전환 |
| 자동 Off 시 | 팝업: "오더 제안이 Off되었습니다" |

#### EC4: 묶음오더 부분 취소 (Could Have)
| 상황 | 처리 |
|------|------|
| 2건 묶음 중 1건 취소 | 해당 오더만 취소 |
| 배송료 | 남은 오더 기준 재계산, 기존 묶음 이상 보장 |
| 수행 순서 | 취소 건 제외, 나머지 순서 유지 |

### 3.3 Error Flows

#### EF1: 오더 수락 실패
| 상황 | 처리 |
|------|------|
| 네트워크 오류 | 토스트: "네트워크 연결을 확인해주세요" + 재시도 |
| 이미 배정됨 | 토스트: "이미 배정된 오더입니다" + 자동 닫힘 |
| 서버 오류 | 팝업: "잠시 후 다시 시도해주세요" + 고객센터 버튼 |

#### EF2: 결제 실패 (후불)
| 상황 | 처리 |
|------|------|
| 카드 결제 실패 | "결제 실패. 다른 결제 수단을 시도해주세요" |
| 3회 연속 실패 | VCC 연결 유도 |

#### EF3: 사진 업로드 실패
| 상황 | 처리 |
|------|------|
| 업로드 실패 | "사진 업로드 실패. 재시도 중..." |
| 3회 실패 | "네트워크 확인 후 다시 시도해주세요" |

## 4. 비기능 요구사항

### 4.1 성능
| 항목 | 기준 |
|------|------|
| 앱 실행 → 메인화면 | **3초 이내** |
| 오더 제안 푸시 수신 | 서버 발송 후 **5초 이내** |
| 상태 변경 API 응답 | **2초 이내** |
| 지도 렌더링 | **1초 이내** |

### 4.2 보안
| 항목 | 기준 |
|------|------|
| 고객 전화번호 | **안심번호** (050-xxxx-xxxx) |
| 고객 상세주소 | 수행 순서 도달 전 **읍면동까지만** 표시 |
| 배송 완료 사진 | 암호화 저장, VCC만 조회 |
| 위치 정보 | 수행 중에만 수집, 종료 시 중단 |

### 4.3 규제
| 항목 | 기준 |
|------|------|
| 위치정보 | 위치정보법 준수, 동의 후 수집 |
| 개인정보 | 개인정보보호법 준수, 최소 수집 |

## 5. 예외 상황 정책

| 상황 | 처리 |
|------|------|
| 도착지 주소 변경 | 노티 + 순서 재계산 (현재 다음 목적지는 고정) |
| 인트라 오더 추가 | 노티 + 순서 재계산 |

| 주소확인필요 오더 | 핀 위치 대신 **고객 입력 주소**로 안내 |
