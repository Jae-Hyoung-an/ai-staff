## 1. 개요

### 1.1 배경 및 문제 정의

**현재 상황**:
- NVP(New Vroong Platform) 런칭을 위해 신규 기사앱 개발 필요
- 기존 프렌즈 앱은 오더 제안 거부 불가, 전체 경로 노출 등으로 기사 피로도 높음

**핵심 문제**:

| 문제 | 영향 | 근거 |
|------|------|------|
| 원치 않는 오더 강제 노출 | 기사 피로도 증가, 이탈 | 프렌즈 앱 기사 VOC |
| 전체 배송 경로 표시 | 판단 과부하, 집중력 저하 | 쿠팡/배민 벤치마크 대비 |
| 상점 도착 조기 처리 | 데이터 오염, 조리시간 예측 불가 | 내부 데이터 분석 |
| 배차 취소 무제한 | 어뷰징, 배차 효율 저하 | 운영팀 피드백 |

### 1.2 목표

> **핵심 목표**: 기사가 오더 수행에만 집중할 수 있는 앱

| 목표 | 설명 |
|------|------|
| **수행 집중** | 바로 다음 목적지만 표시, 정보 과부하 제거 |
| **자율성 부여** | 오더 제안 On/Off로 기사 주도권 확보 |
| **데이터 정합성** | 위치 기반 검증으로 정확한 수행 데이터 확보 |
| **어뷰징 방지** | 배차 취소 제한, 강제 도착 로깅 |

### 1.3 성공 지표 ⭐

#### 핵심 지표 (Core)
| 지표 | 현재 | 목표 | 측정 방법 |
|------|------|------|----------|
| 오더 수락률 | 65% | **75%** | 제안 대비 수락 비율 |
| 오더 당 평균 수행 시간 | 35분 | **30분** | 배정→완료 시간 |
| 기사 1일 평균 수행 건수 | 18건 | **22건** | 완료 건수/활성 기사 |

#### 보조 지표 (Usage)
| 지표 | 목표 | 측정 방법 |
|------|------|----------|
| 신규 오더 제안 On 유지 시간 | 평균 **4시간 이상** | On 상태 누적 시간 |
| 상점 도착 정상 처리율 | **95% 이상** | 반경 내 도착 / 전체 도착 |
| 바텀시트 확장 사용률 | **40% 이상** | 상세 정보 확인 비율 |

#### 가드레일 지표 (Guardrail)
| 지표 | 기준 | 초과 시 대응 |
|------|------|-------------|
| 배차 취소율 | **5% 이하 유지** | 취소 사유 분석, 정책 조정 |
| 기사앱 관련 CS 문의 | **증가 없음** | UI/UX 개선 검토 |
| 앱 크래시율 | **0.5% 이하** | 긴급 핫픽스 |
| GPS 강제 처리 비율 | **5% 이하** | GPS 로직 개선 |

| As a... | I want to... | So that... |

```
[앱 실행] → [메인화면 진입] → [배너 확인(있으면)] → [오더 제안 On]

[오더 제안 수신] → [수락] → [상점으로 이동]
    ↓
[상점 도착 처리] → [음식 픽업] → [도착지로 이동]
    ↓
[사진 촬영] → [배송 완료] → [M캐시 입금 확인]
    ↓
[다음 오더 대기 or Off]
```

## 4. 사용자 시나리오 ⭐

### 4.1 Happy Path

#### 시나리오: 단건 오더 수행
```
1. 기사가 앱 실행, 메인화면 진입
2. [오더 제안 On] 터치
3. 신규 오더 제안 푸시 수신
4. 지도에 상점 위치 표시, 바텀시트에 상점명/원가/거리 표시
5. [수락] 터치 → 오더 배정
6. 지도에 상점까지 직선 표시, 조리완료 잔여시간 표시
7. 상점 150m 내 진입 → [상점 도착] 활성화 → 터치
8. 바텀시트에 픽업번호, 상품내역 표시
9. [픽업 완료] 터치 → 도착지 정보 표시
10. 도착지 도착 → [사진 촬영] → [배송 완료]
11. M캐시 입금, 완료 바텀시트 표시 (3초 후 자동 닫힘)
```

### 4.2 Edge Cases

#### Case 1: GPS 오류로 상점 도착 처리 불가
| 조건 | 처리 |
|------|------|
| 상점 150m 밖에서 도착 터치 | "상점 근처에서 도착 처리해주세요" 안내 |
| 30초간 GPS 신호 없음 | "GPS 확인 필요" 팝업 |
| 60초 경과 후에도 신호 없음 | **[강제 도착 처리]** 버튼 활성화 |
| 강제 도착 처리 시 | 로그에 `force_arrival=true` 기록 |

#### Case 2: 배차 취소 횟수 초과
| 조건 | 처리 |
|------|------|
| 하루 5회 취소 사용 완료 | "오늘 취소 가능 횟수를 모두 사용했습니다" |
| 추가 취소 필요 시 | "VCC에 문의해주세요" + VCC 전화 버튼 |
| 횟수 리셋 | 매일 00:00 |

#### Case 3: 묶음오더 부분 취소 (Could Have)
| 조건 | 처리 |
|------|------|
| 2건 묶음 중 1건 취소 요청 | 개별 오더 단위로 취소 가능 |
| 취소 후 배송료 | 남은 오더 단독 배송료로 재계산, **기존 묶음 배송료 이상 보장** |
| 수행 순서 | 취소 건 제외, 나머지 순서 유지 |

### 4.3 Error Flows

#### Error 1: 오더 제안 수락 실패
| 조건 | 처리 |
|------|------|
| 네트워크 오류 | "네트워크 연결을 확인해주세요" + 재시도 버튼 |
| 이미 다른 기사가 수락 | "이미 배정된 오더입니다" + 자동 닫힘 |
| 서버 오류 | "잠시 후 다시 시도해주세요" + 고객센터 연결 |

#### Error 2: 결제 처리 실패 (후불건)
| 조건 | 처리 |
|------|------|
| 카드 결제 실패 | "결제 실패. 다른 결제 수단을 시도해주세요" |
| 3회 연속 실패 | VCC 연결 유도, 상황 보고 |

## 5. 비즈니스 정책 ⭐

### 5.1 핵심 비즈니스 규칙

#### 오더 제안 정책
| 정책 | 규칙 |
|------|------|
| 기본 상태 | **Off** (앱 실행 시) |
| 제안 대상 | operation type이 NVP인 상점의 오더만 |
| 자동 Off | On 상태에서 **30분간** 수락/거절 액션 없으면 자동 Off + 안내 팝업 |
| 안전운행 안내 | On 전환 시 **1일 1회** 안전운행 팝업 |
| 최대 동시 수행 | **3건** |

#### 상점 도착 정책
| 정책 | 규칙 |
|------|------|
| 도착 가능 반경 | 상점 좌표 기준 **150m 이내** |
| 반경 외 시도 | "상점 근처에서 도착 처리해주세요" 안내 |
| GPS 오류 대응 | 60초 대기 후 **강제 도착 처리** 버튼 활성화 |
| 강제 처리 로그 | `force_arrival=true`, `gps_status`, `timestamp` 기록 |

#### 배차 취소 정책
| 정책 | 규칙 |
|------|------|
| 픽업 전 취소 | 기사 직접 취소 가능 |
| 픽업 후 취소 | **VCC를 통해서만** 가능 |
| 일일 취소 횟수 | **최대 5회** |
| 횟수 리셋 | 매일 00:00 |
| 취소 사유 | 4가지 프리셋 + 직접 입력 |
| 취소 시 안내 | "배차 취소 시 오더 배정에 불이익이 있을 수 있습니다" |

**취소 사유 프리셋**:
1. 상점 영업 종료/품절
2. 개인 사정 (체력/긴급 상황)
3. 배달 불가 지역
4. 기타 (직접 입력)

#### 배송 완료 정책
| 정책 | 규칙 |
|------|------|
| 선불 오더 | 사진 촬영 → 배송 완료 (촬영 스킵 가능, 단 필수 오더 제외) |
| 후불 오더 | 결제 완료 → 자동 배송 완료 |
| 사진 필수 오더 | 올리브영 등 특정 화주사 오더 (상점 설정) |
| 위치 검증 오더 | 올리브영 등 선완료 방지 대상 (bypass 가능, 로그 기록) |

### 5.2 권한 및 접근 정책

| 사용자 | 권한 |
|--------|------|
| NVP 기사 | 오더 수락/거절, 상태 변경, 취소 (픽업 전) |
| 인트라 담당자 | 배너 등록/관리, 오더 상태 변경 (최후 수단) |
| 매니저 PC/M (지점장/오퍼레이터) | NVP 기사 위치/오더 **조회만** 가능, 배차/상태변경 **불가** |
| 어드민 계정 (매니저PC 로그인 시) | 매니저 PC와 동일 권한 적용. NVP 오더 **수정/상태변경/추가배차/배차취소 불가**, 단 오더 상태값 기준 **대기(접수 계열)** 오더는 조회 가능 |
| VCC | 픽업 후 취소 처리, 긴급 상황 대응 |

### 5.3 예외 처리 정책

| 예외 상황 | 처리 방향 |
|----------|----------|
| 도착지 주소 변경 | 주소 업데이트 + 수행 순서 재계산 (단, 현재 다음 목적지는 고정) |
| 인트라 오더 추가 배정 | 노티 + 수행 순서 재계산 (현재 다음 목적지 고정) |
| 인트라 오더 취소 | 노티 + 해당 오더 목록 제거 + 순서 재계산 |
| 스타벅스 오더 | 고객 문자 전송 **불가** (화주사 정책) |
| 주소확인필요 오더 | 핀 위치 대신 **고객 입력 주소**로 안내, baseAddress 표시 |

## DoR (Definition of Ready) 체크리스트

| 조건 | 충족 |
|------|------|
| 사용자 스토리와 시나리오가 정의되어 있다 | ✅ |
| 비즈니스 정책이 구체적 숫자/조건으로 명시되어 있다 | ✅ |
| 성공 지표(핵심/보조/가드레일)가 정의되어 있다 | ✅ |
| 우선순위(MoSCoW)가 합의되어 있다 | ✅ |
| 디자인 문서 링크가 연결되어 있다 | ⚠️ 작업 중 |
| 이해관계자 리뷰가 완료되어 있다 | ⚠️ 진행 중 |
