## 전제 조건

| 정책 | 설명 |
|------|------|
| **AI배차 대상** | AI배차를 통해서는 '원오더'만 배차를 시도한다. 화주사 과적 오더는 AI배차 대상에서 제외한다. |
| **과적 오더 갯수 제한** | 없음. 원오더에 종속되는 과적 오더의 갯수에 제한을 두지 않는다. |
| **적용 대상 기사** | 플러스앱 NVP 기사 및 프렌즈에게 적용한다. 프렌즈가 배차 대상이 되면 프렌즈의 기존 과적 묶음 정책(All or Not 규칙 등)을 따른다. 이륜 기사는 일반 오더로 취급한다. |
| **NVP 오더 제안 타이밍** | NVP 오더는 오더 생성 즉시 제안하지 않고, 기사가 오더를 수락하여 바로 이동했을 때 픽업 요청 시간을 준수할 수 있는 시점에 제안한다. |
| **묶음 유지 원칙** | 원오더가 취소되지 않는 한 묶음은 그대로 유지한다. 배차 취소, 접수 전환 등의 상황에서도 묶음 관계가 해제되지 않는다. |
| **VCC 배차 제외** | 묶음 오더가 존재할 경우 원오더와 종속된 과적 오더 모두 VCC 배차 대상에서 제외된다. 묶음이 해제되면 각각 개별 오더로 VCC 배차 대상이 된다. |

## 케이스 0: AI배차 과적 오더 제외

### 유저 시나리오

> 시스템은 화주사 과적 오더를 AI배차 대상에서 제외하고, 원오더만 배차를 시도한다.

### 개발 요구사항

| Priority | 기능 카테고리 | 유저 시나리오 | 개발 요구사항 |
|----------|-------------|-------------|-------------|
| P0 | 배차 엔진 | 시스템은 화주사 과적 오더를 AI배차 후보에서 제외한다 | **- 화주사 과적 오더 배차 제외**<br>- 화주사 과적 오더(홈플러스/스타벅스 규칙에 따라 원오더에 종속 생성된 오더)는 AI배차 대상에서 제외한다.<br>- 원오더만 AI배차를 통해 기사에게 제안한다.<br>- 원오더를 배차받은 기사에게 화주사 과적 오더는 자동으로 강제 배정한다.<br><br>**- VCC 배차 대상 제외**<br>- 묶음 오더가 존재할 경우 원오더+종속 과적 오더 모두 VCC 배차 대상에서 제외한다.<br>- VCC 배차 관리 목록에서 미노출한다.<br>- 묶음이 해제(원오더 취소)되면 각각 개별 오더로 VCC 배차 대상이 된다. |

## 케이스 1: 배정 전 과적 오더 존재 시 묶음 제안

### 유저 시나리오

> 오더 생성 이후, 기사에게 배정되기 전 시점에 이미 원오더에 대한 화주사 과적 오더가 생성되어 있을 경우:
> 1. 기사에게 오더 제안 시, 과적건이 포함된 오더임을 안내한다.
> 2. 원가는 원오더 기사 원가 + 과적 오더 기사 원가를 합산하여 표기한다.
> 3. 한 번의 수락으로 원오더 + 과적 오더가 모두 자동으로 배차된다.

### 개발 요구사항

| Priority | 기능 카테고리 | 유저 시나리오 | 개발 요구사항 |
|----------|-------------|-------------|-------------|
| P0 | 배차 엔진 | 시스템은 원오더 제안 시점에 존재하는 과적 오더를 묶음으로 구성한다 | **- 묶음 구성 시점**<br>- 원오더의 제안 시점 이전까지 생성된 화주사 과적 오더를 대상으로 묶음을 구성한다.<br>- 묶음 대상 오더 갯수에 제한을 두지 않는다 (원오더 1건 + 과적 오더 N건). |
| P0 | 오더 제안 (플러스앱) | 기사는 과적 묶음 오더 제안 시 과적 포함 여부와 합산 원가를 확인할 수 있다 | **- 제안 바텀시트 표시**<br>- [과적] 배지를 표시하여 과적 묶음 오더임을 안내한다.<br>- 총 원가: 원오더 기사 원가 + 과적 오더들의 기사 원가를 모두 합산하여 표기한다.<br>- 하단 수락 버튼에 총 오더 수를 표시한다 (예: "N건 한번에 수락").<br>- 제안 타이머는 기존과 동일하게 30초를 적용한다. |
| P0 | 배차 엔진 | 기사가 묶음 제안을 수락하면 원오더와 과적 오더가 한번에 배차된다 | **- 수락 시 배차 처리**<br>- 기사가 수락 버튼을 터치하면, 원오더 수락 처리와 동시에 종속된 과적 오더를 동일 기사에게 자동 강제 배정한다.<br>- 과적 오더의 상태는 원오더와 동일하게 '배정' 상태로 전환한다. |
| P0 | 배차 엔진 | 기사가 묶음 제안을 거절하면 원오더와 과적 오더 모두 거절 처리된다 | **- 거절/무응답 처리**<br>- 거절 또는 30초 무응답 시, 원오더와 과적 오더 모두 해당 기사에 대한 제안이 거절 처리된다.<br>- 묶음 관계는 유지되며, 다른 기사에게 묶음 오더로 재제안한다.<br>- 무응답 거절 카운트는 1회로 카운트한다 (원오더 기준). |

## 케이스 2: 배정 후 과적 오더 생성 시 자동 강제 배정

### 유저 시나리오

> 기사가 원오더를 이미 수락한 이후 화주사 과적 오더가 생성되었을 경우:
> 1. 원오더를 수행하고 있는 기사에게 자동으로 해당 과적 오더를 강제 배정한다.
> 2. 기사에게는 수행 중인 오더에 과적건이 추가되었음을 안내한다 (N건 추가, N원 추가 등).

### 개발 요구사항

| Priority | 기능 카테고리 | 유저 시나리오 | 개발 요구사항 |
|----------|-------------|-------------|-------------|
| P0 | 배차 엔진 | 시스템은 원오더 배정 후 생성된 과적 오더를 동일 기사에게 자동 강제 배정한다 | **- 자동 강제 배정 조건**<br>- 원오더가 '배정', '상점도착', '픽업완료' 상태인 경우, 해당 원오더에 종속된 신규 과적 오더가 생성되면 동일 기사에게 자동 강제 배정한다.<br>- 과적 오더의 상태는 원오더의 현재 상태와 동일하게 설정한다.<br>- 기사의 활동 상태(시작/휴식/종료)와 무관하게 강제 배정한다.<br><br>**- 원오더 완료 후 과적 생성**<br>- 원오더가 '배달완료' 상태인 경우 과적 오더가 생성되면, 과적 오더를 '배달완료' 상태로 동일 기사에게 강제 배정한다.<br>- 해당 과적 오더의 배송비를 기사에게 추가 지급하고, M캐시에 입금 처리한다. |
| P0 | 앱 내 안내 (플러스앱) | 기사는 수행 중 과적 오더 추가 시 안내를 받을 수 있다 | **- 인앱 알림**<br>- 수행 중 과적 오더가 추가 배정되면, 앱 내에서 과적 오더 추가를 안내한다.<br>- 안내 내용: 추가된 과적 건수(+N건), 추가 배송비(+N원).<br>- 수행 목록에서 과적 오더 Chip 카운트가 증가하여 반영된다.<br><br>**- 원오더 완료 후 과적 추가 시**<br>- 오더 원가 증액 안내와 함께 추가 원가가 지급되었음을 안내한다. |

### 케이스 1+2 복합: 이미 묶음 배차 이후 추가 과적 생성

> 케이스 1로 원오더와 과적 오더를 제안 시부터 안내받아 같이 배차받은 상태에서, 추가로 화주사 과적 오더가 생성되면 케이스 2도 함께 작동한다.

| Priority | 기능 카테고리 | 유저 시나리오 | 개발 요구사항 |
|----------|-------------|-------------|-------------|
| P0 | 배차 엔진 | 묶음 배차 이후 추가 과적 오더가 생성되면 동일 기사에게 자동 강제 배정한다 | **- 처리 방식**<br>- 케이스 2와 동일하게 처리한다.<br>- 추가 과적 오더를 동일 기사에게 강제 배정하고, 앱 내에서 추가 배정을 안내한다. |

## 케이스 3: 오더 수행 간소화 (상태 동기화)

### 유저 시나리오

> 화주사 과적 오더를 원오더와 같이 배차받은 상황에서:
> 1. 기사는 원오더 1건만 상태를 변경하면, 종속된 화주사 과적 오더도 동일하게 상태가 변경된다.
> 2. N건의 과적 오더가 배정되었더라도 상점도착/픽업/배달완료 처리를 건건이 하지 않아도 된다.

### 개발 요구사항

| Priority | 기능 카테고리 | 유저 시나리오 | 개발 요구사항 |
|----------|-------------|-------------|-------------|
| P0 | 오더 수행 (플러스앱) | 기사가 원오더의 상태를 변경하면 종속 과적 오더도 동일하게 상태가 변경된다 | **- 상태 동기화 대상 액션**<br>- 상점 도착: 원오더 상점 도착 처리 시, 종속 과적 오더도 상점 도착으로 변경한다.<br>- 픽업 완료: 원오더 픽업 완료 처리 시, 종속 과적 오더도 픽업 완료로 변경한다.<br>- 배달 완료: 원오더 배달 완료 처리 시, 종속 과적 오더도 배달 완료로 변경한다.<br><br>**- 과적 오더 상태 변경 불가**<br>- 플러스앱 수행 목록에는 원오더만 표시되며, 종속 과적 오더는 개별 오더로 수행 목록에 노출되지 않는다.<br>- 따라서 기사는 과적 오더에 대해 취소를 포함한 어떠한 상태 변경도 앱에서 직접 수행할 수 없다.<br>- 과적 오더의 상태는 오직 원오더의 상태 변경에 연동되어 자동으로만 변경된다. |
| P0 | 오더 수행 (플러스앱) | 기사는 수행 목록에서 과적 묶음 오더를 확인할 수 있다 | **- 수행 목록 표시**<br>- 수행 목록에는 원오더만 표시한다. 과적 오더는 [과적] 태그와 과적 오더 Chip(N = 과적 오더 개수)으로 원오더에 부속 표시한다.<br>- 과적 오더는 수행 목록에 개별 오더로 노출되지 않으므로, 기사가 과적 오더를 직접 선택하거나 상태를 변경할 수 없다.<br>- 과적 오더 추가 시 Chip 카운트가 증가한다.<br>- 수행 Task는 원오더 기준으로만 생성된다 (과적 오더별 개별 Task 미생성). |
| P0 | 배달 완료 (플러스앱) | 원오더 배달 완료 시 과적 오더도 한번에 완료 처리된다 | **- 배달 완료 처리**<br>- 원오더 배달 완료 사진 촬영 시, 촬영한 사진을 종속 과적 오더에도 모두 적용한다.<br>- 스타벅스 과적 묶음의 경우, 배달 완료 메시지는 원오더 1건만 발송한다.<br><br>**- 배송비 지급**<br>- 원오더 배달 완료 시 원오더 배송비 + 종속 과적 오더들의 배송비를 각각 M캐시에 입금한다.<br>- 배달내역에는 원오더와 과적 오더가 개별 건으로 표시되며, 각 오더별 배송비를 확인할 수 있다. |

## 케이스 4: 묶음 해제 및 미배차 시 처리

### 4-1. 묶음 해제 조건: 원오더 취소 시에만 해제

> 원오더가 취소되지 않는 한 묶음은 그대로 유지한다. 배차 취소, 접수 전환 등의 상황에서도 묶음 관계가 해제되지 않는다.

| Priority | 기능 카테고리 | 유저 시나리오 | 개발 요구사항 |
|----------|-------------|-------------|-------------|
| P0 | 배차 엔진 | 원오더가 취소되면 과적 오더와의 묶음이 해제된다 | **- 오더 취소 시 묶음 해제**<br>- 원오더가 취소되면, 종속 과적 오더는 묶음 해제되어 일반 오더로 전환된다.<br>- 과적 오더는 개별적으로 상태 변경 및 배차가 가능하다.<br>- 묶음 해제된 과적 오더는 VCC 배차 대상이 된다.<br>- 묶음 해제된 과적 오더는 2륜 기사앱, 프렌즈 셀프배차 목록에 일반 오더로 노출된다. |

### 4-2. 배차 취소 시: 묶음 유지 + 재제안

> 기사가 원오더를 배차 취소할 경우, 묶음 관계를 유지한 채 다른 플러스앱 기사 및 프렌즈에게 묶음 오더로 제안을 진행한다.

| Priority | 기능 카테고리 | 유저 시나리오 | 개발 요구사항 |
|----------|-------------|-------------|-------------|
| P0 | 배차 엔진 | 기사가 원오더를 배차 취소하면 묶음을 유지한 채 다른 기사에게 재제안한다 | **- 배차 취소 시 묶음 유지**<br>- 기사가 원오더를 배차 취소하면, 원오더와 종속 과적 오더의 묶음 관계를 유지한다.<br>- 묶음 오더를 다른 플러스앱 NVP 기사 및 프렌즈에게 묶음 오더로 재제안한다.<br>- 배차 취소 카운트는 원오더 기준 **1건**으로만 카운트한다.<br><br>**- 프렌즈 배차 시**<br>- 프렌즈가 배차 대상이 되면 프렌즈의 기존 과적 묶음 정책(All or Not 규칙, 책임배송 미적용 등)을 따른다. |

### 4-3. 접수 상태 전환 시: 묶음 유지 + 독점 제안 + 셀프배차

> 원오더의 픽업 요청시간까지 배차가 안되어 접수 상태로 변경되더라도, 묶음 상태를 유지하며 독점 제안을 지속한다.

| Priority | 기능 카테고리 | 유저 시나리오 | 개발 요구사항 |
|----------|-------------|-------------|-------------|
| P0 | 배차 엔진 | 접수 상태 전환 시에도 묶음을 유지하며 독점 제안을 지속한다 | **- 접수 상태 묶음 유지**<br>- 원오더의 픽업 요청시간까지 배차가 안되어 '접수' 상태로 전환되더라도, 과적 오더와의 묶음 관계를 유지한다.<br>- 다른 일반 오더와 다르게 동시제안으로 전환하지 않고, 지속적으로 묶음 오더로 독점 제안을 진행한다.<br><br>**- 2륜 기사앱 / 프렌즈 셀프배차 미노출**<br>- 묶음 오더는 접수 상태로 변경되더라도 2륜 기사앱의 전투배차 목록에 노출되지 않는다.<br>- 묶음 오더는 프렌즈 셀프배차 목록에도 노출되지 않는다. |
| P0 | 셀프배차 (플러스앱) | 접수 상태의 묶음 오더는 셀프배차 목록에 원오더만 표시되며, 수락 시 종속 오더도 함께 배차된다 | **- 셀프배차 목록 노출**<br>- 접수 상태의 묶음 오더는 플러스앱 셀프배차 목록에 표시한다.<br>- 원오더만 노출하되, [과적] 태그를 표시하여 과적 묶음 오더임을 안내한다.<br>- 원가는 원오더 원가 + 종속 과적 오더들의 원가를 모두 합산하여 표기한다.<br>- 과적 오더가 추가 생성되면 합산 원가를 실시간으로 업데이트한다.<br><br>**- 셀프배차 수락 시**<br>- 원오더를 셀프배차로 배정받을 경우, 종속된 과적 오더도 모두 동일 기사에게 자동 강제 배정된다. |

## 기존 정책 영향 정리

| 정책 항목 | 기존 정책 | 과적 묶음 적용 시 변경사항 |
|-----------|----------|------------------------|
| **최대 동시 수행 건수** | 3건 (논의 중) | 원오더만 카운트. 과적 오더는 미포함. 과적 오더 갯수 제한 없음. |
| **배차 취소 횟수** | 일 3회 | 원오더 기준 1건만 카운트. 종속 과적 오더는 별도 미카운트. |
| **무응답 거절 → 자동 Off** | 연속 3회 | 묶음 제안 거절은 1회로 카운트 (원오더 기준). |
| **셀프배차** | 접수 상태 NVP 오더 노출 | **플러스앱**: 묶음 상태에서도 셀프배차 노출 (원오더만 + [과적] 표시 + 합산 원가, 실시간 갱신. 수락 시 종속 오더 함께 배차). **프렌즈**: 묶음 오더는 셀프배차 미노출. 묶음 해제 시 각각 별개 일반 오더로 양쪽 모두 노출. |
| **동시제안** | SLA 위험 시 동시제안 진행 | 묶음 오더는 **동시제안 미진행**. 접수 상태에서도 독점 제안만 지속. |
| **2륜 기사앱 노출** | 접수 상태 시 전투배차 목록 노출 | 묶음 오더는 접수 상태에서도 **2륜 기사앱에 미노출**. |
| **VCC 배차** | VCC 배차 관리 목록 노출 | 묶음 오더 존재 시 원오더+과적 오더 모두 **VCC 배차 대상 제외, 목록 미노출**. 묶음 해제 시 각각 개별 VCC 배차 대상. |
| **배송비/할증** | 원오더 할증 동일 적용 | 과적 오더에 미적용 할증: 프로모션, SLA, 부스트 오더, VCC 추가 할증. 나머지는 원오더와 동일 적용. |
| **배달내역** | 개별 오더로 표시 | 과적 오더 완료 후 배달내역에 개별 건으로 표시. 각 오더별 배송비 확인 가능. |
| **배달 완료 사진** | 배달 완료 시 촬영 | 원오더 촬영 사진을 종속 과적 오더에도 모두 적용. |

## 엣지 케이스 & 예외 처리

### E1. 과적 오더 일부만 취소

| 항목 | 정책 |
|------|------|
| **상황** | 과적 묶음 중 일부 과적 오더만 인트라에서 취소 처리 |
| **처리** | 취소된 과적 오더만 묶음에서 제거. 나머지 과적 오더와 원오더의 묶음 관계는 유지. |

### E2. 과적 오더 전체 취소

| 항목 | 정책 |
|------|------|
| **상황** | 원오더에 연결된 과적 오더가 모두 취소 |
| **처리** | 원오더는 일반 오더로 전환. 과적 묶음 관계 해제. 기사는 원오더만 수행 계속. 원오더는 VCC 배차 대상 및 2륜 노출 대상이 된다. |

### E3. 묶음 해제 후 재묶음 가능 여부

| 항목 | 정책 |
|------|------|
| **상황** | 원오더 취소로 묶음 해제된 과적 오더가 다시 묶음으로 복구될 수 있는지 |
| **처리** | 불가. 원오더가 취소되어 묶음 해제되면 별개 오더로 처리. 재묶음 미지원. |

### E4. 원오더 인트라 상태 역방향 변경

| 항목 | 정책 |
|------|------|
| **상황** | 인트라에서 원오더 상태를 역방향으로 변경 (예: 배달완료 → 픽업) |
| **처리** | 원오더 상태 변경 시 종속 과적 오더도 동일하게 역방향 변경. 기존 오더수행 정책서의 인트라 상태 변경 정책(Section 10)을 동일 적용. |

### E5. 과적 묶음 상태에서 기사 Off 전환

| 항목 | 정책 |
|------|------|
| **상황** | 과적 묶음 오더 수행 중 기사가 오더 제안 Off로 전환 |
| **처리** | 기배정 오더에는 영향 없음 (기존 정책 동일). 과적 묶음 오더 수행을 계속한다. 추가 과적 오더 생성 시에도 강제 배정은 정상 작동. |

### E6. 2시간 미수행 자동 Off

| 항목 | 정책 |
|------|------|
| **상황** | 과적 묶음 오더 수행 중이므로 2시간 미수행 자동 Off 조건 해당 여부 |
| **처리** | 현재 수행 중인 오더가 있으면 자동 Off 미적용 (기존 정책 동일). 과적 묶음 오더도 '수행 중'으로 판단. |

### E7. 셀프배차 목록 노출 중 과적 오더 추가 생성

| 항목 | 정책 |
|------|------|
| **상황** | 셀프배차 목록에 묶음 오더가 노출 중일 때 추가 과적 오더가 생성되는 경우 |
| **처리** | 합산 원가를 실시간으로 업데이트한다. 수락 시 추가 생성된 과적 오더도 함께 배차된다. |

### E8. 배차 취소 후 프렌즈 제안 시 프렌즈 정책 적용

| 항목 | 정책 |
|------|------|
| **상황** | 플러스앱 기사가 배차 취소 후, 묶음 오더가 프렌즈에게 제안되는 경우 |
| **처리** | 프렌즈가 배차 대상이 되면 프렌즈의 기존 과적 묶음 정책을 모두 따른다 (All or Not 규칙, 책임배송 미적용, 상태 동기화 등). |

## So What / Next Step

- 이 기능이 적용되면 화주사 과적 오더로 인한 **기사 2인 현장 도착 문제가 원천 차단**된다. 과적 오더는 원오더 기사에게만 배정되므로, 현장 클레임과 화주사 이탈 리스크를 해소할 수 있다.
- **묶음 유지 원칙 강화**로 인해, 배차 취소/접수 전환 시에도 과적 오더가 분리되지 않으므로 과적 오더 미수행 리스크가 추가로 해소된다.
- 접수 상태에서도 **독점 제안 유지 + 셀프배차 노출**로 이중 배차 시도가 가능하여 배차 지연을 최소화할 수 있다.
- 기사 관점에서는 과적 오더를 건건이 처리할 필요 없이 **원오더 1건 수행만으로 전체 완료**되므로 수행 편의성이 크게 개선된다.
- **인트라 상태 변경 제한**(TBD #1)은 빠르게 결정하여 개발 착수 전 확정해야 한다.
- 홈플러스 측에 계도 요청이 필요하다: 원오더 발생 후 **5분 이내 과적 오더 생성** + **픽업 요청시각을 충분히 길게 설정**하도록 안내하여 케이스 1(배정 전 묶음 제안) 비율을 높여야 한다.
- 배포 후 케이스 1(배정 전 묶음) vs 케이스 2(배정 후 강제 배정) 비율을 모니터링하여, 화주사별 과적 오더 생성 패턴에 따른 후속 개선 방향을 도출해야 한다.
