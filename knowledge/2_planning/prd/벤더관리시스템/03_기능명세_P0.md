## 2.1 벤더존 지점 타입 추가 + 존 관리에서 매핑 지점 제한 (P0 / 선행)

### 기본 정책

| 정책 | 설명 |
|------|------|
| **추가 목적** | A존/B존 상점을 더 상세하게 구분하기 위해, 지점 운영방식에 벤더존(VendorZone, NVP-B) 타입을 추가 |
| **지점 타입명** | NVP-B → **벤더존(VendorZone)** 으로 명명 |
| **운영 구조** | 벤더존 지점은 존(Zone)과 **1:1 매핑**을 전제로 운영 |
| **존 관리 연결 정책** | 존 관리(Zone Management)에서 지점/존 매핑 시 **벤더존 타입 지점만** 매핑 가능 |
| **검증** | 저장 시 서버단에서 **벤더존 여부 재검증** (위반 시 저장 불가) |

### 기능 명세

| Priority | 화면/메뉴 | 기능 카테고리 | 유저 시나리오 | 개발 요구사항 |
|----------|----------|-------------|-------------|-------------|
| P0 | 인트라 > 지점 관리 > 지점 타입 | 지점 타입 추가 | RM이 벤더 운영을 위한 지점 타입을 추가/관리한다 | 벤더존(VendorZone) 추가. 벤더존 지점은 존과 1:1 매핑 전제 |
| P0 | 인트라 > 존 관리 > 지점/존 매핑 | 선택 제한 | RM이 벤더 운영을 위해 존/지점을 매핑한다 | 벤더존 지점만 노출 및 선택 가능. 벤더존이 아닌 지점은 목록에서 숨김 처리 |
| P0 | 인트라 > 존 관리 > 지점/존 매핑 | 저장 검증 | RM이 매핑을 저장한다 | 저장 시점에 벤더존 여부 재검증, 위반 시 에러 표시 |
| P0 | 인트라 > 지점 관리 > 지점 타입 변경 | 저장 검증 | 기존 벤더존 지점이 non 벤더존 지점 변경 시 존 연결 여부 체크 | 존 연결돼 있을 경우 변경 불가 |
| P0 | 인트라 > 존 관리 > 지점 변경 | 변경 불가 | 존에 매핑된 지점을 변경할 수 없다 | 존과 매핑되어 있는 지점은 다른 지점으로 교체 불가 |

## 2.2 벤더 관리 — 인트라 신규 메뉴 (P0)

### 기본 정책

| 정책 | 설명 |
|------|------|
| **GNB 위치** | 부릉 인트라 > 세트 관리 (신규 메뉴) > 벤더 관리 (신규 메뉴) |
| **관리 단위** | 벤더 단위 (벤더 기본 정보 + 상태 + 연결 존 + 벤더장(VO) + 소속 기사 + 이력) |
| **상태 체계** | STANDBY(대기) → ACTIVE(운영중) ↔ INACTIVE(운영중지) → TERMINATED(해지) |
| **상태 변경 정책** | 모든 상태 변경 시 변경 사유 입력 필수. 변경 이력 로그(사유 포함) 저장 |
| **벤더장(VO) 운영 구조** | 1벤더 1벤더장 운영. VO 지정/변경/해제는 인트라에서 RM이 수행 |

### 사용자 시나리오

> **시나리오 1: 벤더 신규 등록 후 운영 준비**
> 1. RM이 벤더 계약을 완료한다.
> 2. RM이 인트라에서 벤더를 신규 등록한다.
> 3. 등록 완료 시 벤더 상태는 STANDBY(대기)로 저장된다.
> 4. 운영 준비가 되면 상태를 ACTIVE로 변경한다.

> **시나리오 2: 벤더장(VO) 지정 및 권한 회수**
> 1. RM은 벤더장으로 지정할 후보 기사가 플러스 기사로 등록되어 있는지 확인한다.
> 2. RM이 벤더 상세의 VO 탭에서 벤더장을 지정한다.
> 3. 벤더존 지점 소속 기사만 벤더장으로 지정할 수 있다.
> 4. 지정 즉시 해당 기사는 벤더 포털 접근 권한을 부여받는다.
> 5. VO 변경 또는 해제 시 기존 VO의 접근 권한은 즉시 회수된다.

> **시나리오 3: 사업자명/대표자명 변경 발생 시 신규 벤더로 재등록**
> 1. 사업자명 또는 대표자명이 변경되면 기존 벤더를 TERMINATED(해지)로 변경한다.
> 2. 신규 사업자등록증 기준으로 신규 벤더를 등록한다.

### 기능 명세

| Priority | 화면/메뉴 | 기능 카테고리 | 유저 시나리오 | 개발 요구사항 |
|----------|----------|-------------|-------------|-------------|
| P0 | 인트라 > 벤더 관리 | 리스트 조회 | RM이 등록된 벤더 전체를 조회한다 | **컬럼**: 벤더코드, 사업자명(벤더명), 상태, 존, 벤더장, 등록일<br>**필터**: 상태, 존, 벤더명<br>**정렬**: 등록일 내림차순 |
| P0 | 인트라 > 벤더 관리 > 신규 등록 | 벤더 등록 | RM이 새 벤더를 등록한다 | **필수 입력**: 사업자등록번호(중복 체크), 사업자명, 대표자명, 이메일, 연락처, 예금주/은행/계좌번호<br>**선택 입력**: 계약 시작일/종료일, 사업자등록증/계약서/통장 사본/신분증 이미지<br>등록 완료 시 STANDBY 상태 |
| P0 | 인트라 > 벤더 관리 > 상세 | 상태 변경 | RM이 벤더 상태를 변경한다 | **허용 전이**: STANDBY→ACTIVE/TERMINATED, ACTIVE→INACTIVE/TERMINATED, INACTIVE→ACTIVE/TERMINATED<br>STANDBY→ACTIVE: VO 지정 필수<br>INACTIVE: 세트 할당 회수, 기사 소속 유지<br>TERMINATED: 소속 기사 전원 해제, 세트 회수, VO 해제 |
| P0 | 인트라 > 벤더 관리 > 상세 > VO 탭 | VO 지정 | RM이 벤더장(VO)을 지정/해제한다 | 벤더존 지점 소속 벤더 타입 기사 중 선택<br>타 벤더에 지정된 VO는 선택 불가<br>지정 즉시 벤더 포털 접근 가능<br>ACTIVE 상태에서는 VO 해제 불가 (교체만 가능) |
| P0 | 인트라 > 벤더 관리 > 상세 > 소속 기사 탭 | 기사 소속 등록 | RM이 기존 플러스 기사를 벤더에 소속시킨다 | 핸드폰번호(본인인증 번호)로 기사 검색<br>조건: 플러스앱 기사 + 벤더 타입 + 벤더존 지점 소속 + 타 벤더 미소속 |
| P0 | 인트라 > 벤더 관리 > 상세 > 소속 기사 탭 | 기사 소속 해제 | RM이 벤더 소속 기사를 내보낸다 | 해제 시 세트모드에서 일반 모드 전환<br>출근 중이어도 해제 가능 |
| P0 | 인트라 > 벤더 관리 > 상세 > 이력 탭 | 변경 이력 조회 | RM이 벤더의 변경 이력을 확인한다 | 상태 변경, 지점 연결, VO 지정, 기사 등록/해제, 정보 수정 등 모든 이벤트. 최신순 |

## 2.4 벤더 포털 — 로그인 + 소속 기사 관리 (P0)

### 기본 정책

| 정책 | 설명 |
|------|------|
| **인증 방식** | Plus App 기사 계정 기반 ID/PW 인증 |
| **접근 권한** | RM이 VO로 지정한 기사만 로그인 가능. VO 해제 시 즉시 로그인 불가 + 기존 세션 만료 |
| **동시 접속** | 동시 접속 허용 |
| **자동 로그아웃** | 60분 이상 활동이 없을 경우 자동 로그아웃 |
| **기사 추가** | 벤더 포털 내에서 기사 직접 가입/승인/추가 기능은 제공하지 않음 (인트라에서 RM이 소속 등록) |

### 기능 명세

| Priority | 화면/메뉴 | 기능 카테고리 | 유저 시나리오 | 개발 요구사항 |
|----------|----------|-------------|-------------|-------------|
| P0 | 벤더 포털 > 로그인 | 로그인 | VO가 벤더 포털에 로그인한다 | Plus App 기사 계정 기반 ID/PW 인증<br>최초 로그인 시 비밀번호 변경 기능 제공 |
| P0 | 벤더 포털 > 소속 기사 관리 | 기사 목록 조회 | VO가 소속 기사 전체 목록을 확인한다 | **컬럼**: NO, 이름, 연락처, 상태(배달중/대기중/오프라인), 모드(세트/일반), 가입일<br>**요약 지표**: 현재 활성 기사 수 / 최대 가능 수, 출근 중 기사 수 / 전체 기사 수 |
| P0 | 벤더 포털 > 소속 기사 관리 | 모드 전환 | VO가 세트/일반 모드를 전환한다 | 활성화 ↔ 비활성화 토글 전환<br>최대 인원 = 세트당 최대 활성화 기사수 × 할당 세트 수 |

## 2.6 벤더 포털 — 라이더 현황/기사 관리 (P0)

### 기본 정책

| 정책 | 설명 |
|------|------|
| **접근 권한** | 해당 벤더의 VO만 접근 가능 |
| **모드 체계** | 세트/일반(활성 상태) 모드 전환 지원 |
| **세트 가능 인원** | 최대 인원 = (벤더에 할당된 세트 수 × 세트당 최대 활성화 기사수). 화면 상단에 "세트 가능: 현재/최대"로 표시 |
| **해제 정책** | 출근 중이어도 해제 가능. 해제 시 일반 기사 전환 |

### 사용자 시나리오

> **시나리오 1: VO가 특정 기사의 활성 상태를 ON/OFF로 전환한다**
> 1. VO가 기사 관리 > 실시간 수행 현황에서 특정 기사의 활성 상태 토글을 조작한다.
> 2. 활성 상태가 즉시 반영된다.
> 3. ON인 경우, 슬롯 잔여자리가 있으면 세트 물량을 수행하고 잔여자리가 없으면 오더 수행 불가.
> 4. OFF인 경우, 슬롯에 잔여자리가 있어도 오더 수행 불가.

### 기능 명세

| Priority | 화면/메뉴 | 기능 카테고리 | 개발 요구사항 |
|----------|----------|-------------|-------------|
| P0 | 벤더 포털 > 현황 관리 > 라이더 현황 | 라이더 현황 조회 | 현재 온라인/전체 등록기사, 세트, 일반<br>실시간 오프라인 라이더: 이름/전화번호<br>과거 일자 조회 가능 (최대 2달) |
| P0 | 벤더 포털 > 기사 관리 > 실시간 수행 현황 | 라이더 수행 현황 | **컬럼**: 우선순위(출근 순서), 이름/연락처, 상태, 거절/무시, 취소, 완료, 슬롯별 수행건, 활성상태<br>**활성상태**: Default=ON, ON=세트수행(잔여자리 없으면 오더 수행 불가), OFF=오더 수행 불가. off 처리 시 수동 on 전환 전까지 유지 |
| P0 | 벤더 포털 > 기사 관리 | 라이더 수행 정책 | 벤더 소속 기사는 벤더에 매핑된 존의 오더만 수행<br>배차/수락 시 서버에서 벤더 소속/ACTIVE/존 일치 재검증<br>벤더 INACTIVE/TERMINATED 시 일반 기사 정책으로 전환 |

## 2.9 플러스앱 — 벤더 소속 알림 (P0)

### 기본 정책

| 정책 | 설명 |
|------|------|
| **소속 변경 주체/방식** | RM이 인트라에서 소속 등록/변경/해제. VO는 소속 해제만 가능. 기사 앱에서 별도 동의 절차 없음 |
| **소속 변경 안내** | 벤더 소속 상태가 변경된 경우, 변경 즉시 알림 노출 |
| **기사 활동 형태** | 기사 활동 형태(Activity Type)에 "벤더" 타입을 추가 |
| **모드 결정** | 출근 시 모드 자동 판정 (벤더/일반) |
| **V캐시** | 벤더 기사 전용 가상 포인트. 정산 예정 금액 표시, 출금 불가. 현금 결제 처리를 위해 마이너스 허용 |

### 모드 자동 결정 우선순위

```
1) 벤더 미소속 → 일반 기사
2) INACTIVE 벤더 소속 → 일반 기사
3) 세트 할당 없음 → 일반 기사
4) 세트 자리 남음 → 벤더 기사
5) 세트 자리 다 참 → 일반 기사
```

### 기능 명세
