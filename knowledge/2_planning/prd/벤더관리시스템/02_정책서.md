## 핵심 원칙

| # | 원칙 | 설명 | 비고 |
|---|------|------|------|
| 1 | **벤더 1:1 지점 연결** | 벤더 1개는 B-type 지점 1개만 연결. 1개 지점에는 여러 벤더 연결 가능 | 운영 복잡도 제한 |
| 2 | **장애 시 배송 지속** | Vendorset 장애 시 모든 기사를 DP-FA로 처리, 프렌즈 배차로 배송 지속 | 서비스 연속성 |
| 3 | **1벤더 1벤더장** | 벤더당 VO 1명 제한. RM이 인트라에서 직접 지정 | 운영 구조 단순화 |
| 4 | **이력 추적** | 벤더 상태, 정책 버전 등 모든 변경 사항은 누가, 언제, 무엇을 바꿨는지 기록 | 추적 가능한 구조 |

## 전략적 배경 — 세트 개념의 본질

세트는 단순한 물량 단위가 아니라 **퀄리티 관리와 통제력 유지 수단**입니다.

| 항목 | 설명 |
|------|------|
| **정의** | 협력사에 물량을 위탁하는 최소 단위. 요일/구간별 물량으로 구성 |
| **수행 기준** | 해당 물량 수행 성공 여부 + 거절률 + 퀄리티 등을 정산 기준으로 삼음 |
| **기사 요건** | 1세트 수행을 위해 최소 액티브 기사 N명 필요 |
| **탄력 운영** | 협력사마다 계약 세트 수를 탄력적으로 운영 |
| **통제 수단** | 소단위 물량의 모듈화를 통한 쉬운 교체 + 주단위 실적에 따른 빠른 상벌 적용 |

### 퀄리티 관리 — 이중 레버

| 레버 | 수단 | 예시 |
|------|------|------|
| **(1) 계약을 통한 관리** | 지급 물량 조정, 계약 해지, 신규 계약 | 1세트→2세트 증가, 계약 해지 후 대기 벤더 투입 |
| **(2) 정산을 통한 관리** | 지표 도달 여부에 따른 인센티브 지급 | 세트 물량 수행률, 제안 오더 수락률 |

### SET 운영 구조

| 항목 | 쿠팡이츠 상세 | 부릉 적용 시사점 |
|------|-------------|-----------------|

| **SET당 활성화 라이더** | 최소 3개 ~ 최대 7개 (예외적 최대 10개) | 부릉은 권역관리 관리정책에서 설정 (세트당 최대 활성화 기사수) |
| **SET 초과 시** | 협력사 소속 라이더가 일반 배달파트너로 일시 전환 | 부릉의 DP-F 전환과 동일 개념 |

### 시간대별 중요도 가중치 (관리수수료 산정)

| 구간 | 평일(월~금) | 주말(토,일) |
|------|-----------|-----------|
| Morning | 0.15 | 0.10 |
| Lunch Peak | 0.25 | 0.29 |
| Post Lunch | 0.15 | 0.13 |
| **Dinner Peak** | **0.30** | **0.35** |
| Post Dinner | 0.15 | 0.13 |

> Dinner Peak가 가장 높은 가중치. 부릉도 저녁 피크에 가중치를 높게 설정하는 것이 합리적

### 수수료/정산 구조

| 항목 | 쿠팡이츠 상세 |
|------|-------------|
| **관리수수료 (인센티브)** | SET당 15만~20만원. 계약조건 충족 SET 수 × 충족 비율 × SET당 금액 |
| **관리수수료 조건** | (1) 주간 달성률 90%+ (2) 일일 달성률 80%+ (3) 미달성 구간 4개 이하 (4) 수행률 90%+ |
| **거리 할증** | 직선거리 1.5km 초과 시 100m당 추가 (단건), 900m 초과 시 (멀티) |
| **멀티배달 환산** | 멀티배달 1건 = SET상 0.8건으로 환산 |
| **시스템 이용료** | 벤더포털 + 배달파트너앱: 건당 100원, 콜센터: 건당 100원 |

### SLA (서비스 품질 기준)

| 구분 | 항목 | 위반 시 조치 |
|------|------|-------------|
| 서비스/시스템 품질 | 허위 배달완료, 잦은 배정취소, 비정상운행, 미협의 운송수단 | 1차 개선요청 → 2차부터 관리수수료 SET 차감 |
| 배달 품질 | 배달미완료, 파손, 오픽업, 주문오배달, 메뉴누락 | 동일 |

## 기능별 기본 정책

### 1. 벤더 관리

| 정책 | 설명 |
|------|------|
| **벤더 상태** | STANDBY(대기) → ACTIVE(운영중) ↔ INACTIVE(운영중지) → TERMINATED(해지) |
| **등록 시 필수 입력** | 사업자등록번호, 사업자명, 대표자명, 이메일, 연락처, 예금주/은행/계좌번호 |
| **자동 생성** | 벤더 코드 (V + 6자리), 초대 코드 (영문+숫자 8자리) |
| **벤더-지점 관계** | 벤더존 지점과 매핑 (세트관리에서 zone에 벤더 연결) |
| **VO 지정** | RM이 인트라에서 벤더존 지점 소속 기사를 선택하여 VO로 지정. 벤더당 1명 제한 |
| **해지 시 연쇄 처리** | 소속 기사 전원 탈퇴(→DP-FA), 세트 할당 즉시 회수, VO 해제 |

### 5. 벤더 포털 (로그인 + 기사 관리)

| 정책 | 설명 |
|------|------|
| **인증** | Plus App 기사 계정 기반 인증 |
| **세션 정책** | 동시 접속 허용. 60분 이상 미활동 시 자동 로그아웃 |
| **모드 전환** | VO가 기사 활성/비활성 토글 전환 가능 |
| **SET 기반 활성 기사 수** | 최대 인원 = 세트당 최대 활성화 기사수 × 벤더 할당 세트 수 |
| **소속 해제** | 출근 중이어도 해제 가능 → 일반 기사 전환 |

### 8. 플러스앱 (기사 등록 + 모드 결정)

| 정책 | 설명 |
|------|------|
| **중복 소속 불가** | 한 명의 기사가 동시에 2개 이상 벤더에 ACTIVE 소속 불가 |
| **모드 자동 결정** | 출근 시 우선순위에 따라 DP-S/DP-F/DP-FA 자동 결정 |
| **세트 자리** | 세트당 최대 활성화 기사수 × 할당 세트 수 = 최대 동시 활성 기사 수 |
| **초과 시** | 오더 수행 불가 (활성 상태 ON 기준) |
| **V캐시** | 벤더 기사 전용 가상 포인트. 정산 예정 금액 표시, 출금 불가. 현금 결제 처리를 위해 마이너스 허용 |

## 시스템 자동 처리

| 자동 처리 | 시점 | 설명 |
|----------|------|------|
| 기사 모드 자동 결정 | 출근 시 | 세트 자리/벤더 상태/소속 상태에 따라 DP-S/DP-F/DP-FA 자동 결정 |
| 프렌즈 배송 즉시 정산 | 배송 완료 즉시 | DP-F/DP-FA 배송 완료 시 M캐시 실시간 입금 |
| 배차 제안 카운터 | 배차 이벤트 발생 시 | 기사별·슬롯별 제안/수락/거절/만료/시스템취소 자동 카운트 |
| 실적 카운터 | 배차 확정/배송 완료 시 | 기사별·슬롯별 배차 확정/배송 완료 자동 카운트 |
| 기사 활동 이력 추적 | 출근/퇴근/모드 변경 시 | 기사별 활성화/비활성화/모드 전환 이력 자동 기록 |
| 장애 시 폴백 | Vendorset 장애 시 | 모든 기사를 DP-FA로 처리, 프렌즈 배차로 배송 지속 |

## 배차 흐름 (DP-S 오퍼)

```
주문 발생
  → Dispatch Service가 Vendorset에 DP-S 후보 기사 목록 요청
  → 슬롯(요일/시간대) 판단 (주문 생성 시간 기준)
  → 해당 주차의 활성 DP-S 기사 목록 응답
  → Dispatch Service가 통합 풀에서 가중치 기반으로 기사 선정
       (DP-S 기사에게 높은 가중치 부여)
  → 기사에게 배차 오퍼 전달
       ├── 수락 → 배차 확정 (dispatchSource=SET)
       ├── 거절 → 다음 후보에게 재제안
       └── 30초 무응답(만료) → 다음 후보에게 재제안
  → DP-S 후보 소진 시 → Friends 기사에게 자동 전환
```

---

## 배송 수익금 흐름

```
세트 모드(DP-S) 배송 완료
  → 실적 카운터 증가 (배차 시점 기준으로 귀속)
  → V캐시에 정산 예정 금액 표시
  → 벤더장 → 기사에게 지급 (정산 시스템 도입 전까지 수동 관리)

프렌즈 모드(DP-F, DP-FA) 배송 완료
  → 즉시 정산
  → 기사 앱 M캐시에 실시간 입금
```

## 장애 대응

| 장애 시간 | 예상 영향 |
|----------|----------|
| ~5분 | 출근 시도 기사만 DP-FA, 실적 미미 |
| ~30분 | 다수 기사 DP-FA, 슬롯 1개 분량 누락 가능 |
| ~2시간 | 피크 시간대 전체 영향, 슬롯 다수 누락 |
| 4시간+ | 반일 운영 영향 |

> 복구 후 보정 절차(기사 모드 보정, M캐시 이중 정산 처리, DLQ 재처리)는 운영 절차 수립 필요 (TBD)
