# PRD 작성 가이드

> PRD(Product Requirements Document)를 효과적으로 작성하기 위한 나만의 가이드

---

## 개요

PRD는 **제품/기능의 요구사항을 정의하고, 이해관계자들과 합의된 내용을 문서화**하는 핵심 산출물이다.

---

## 핵심 내용

### 1. 문제 정의가 가장 중요하다

- "무엇을 만들 것인가"보다 **"왜 만드는가"**를 먼저 명확히
- 데이터로 문제의 크기와 영향도를 정량화
- 이 문제를 해결하지 않으면 어떤 일이 생기는지 명시

### 2. 성공 지표(KPI)를 미리 정의한다

- 이 기능이 성공했는지 어떻게 알 수 있는가?
- 측정 가능한 지표로 정의 (예: 배차 성공률 +2%p, 이탈률 -5%)
- A/B Test 계획과 연계

### 3. 범위(Scope)를 명확히 한다

- **포함하는 것 (In-Scope)**: 이번에 개발할 내용
- **포함하지 않는 것 (Out-of-Scope)**: 이번엔 안 하지만 나중에 할 수 있는 것
- 범위 논쟁을 줄이고 일정 리스크를 관리

### 4. 사용자 시나리오로 설명한다

- 기능 명세만 나열하지 않고, 사용자 관점의 흐름으로 작성
- "사용자가 ~하면, 시스템이 ~한다" 형식
- 엣지 케이스도 시나리오로 정리

### 5. 이해관계자 Align을 문서에 남긴다

- 논의 과정에서 결정된 사항과 그 이유를 기록
- 나중에 "왜 이렇게 했지?" 질문에 답할 수 있도록
- 변경 이력 관리

---

## 자주 쓰는 용어

| 용어 | 설명 |
|------|------|
| **PRD** | Product Requirements Document. 제품 요구사항 문서. 기획 의도, 기능 명세, 성공 지표 등을 포함 |
| **User Story** | 사용자 관점에서 기능을 설명하는 형식. "~로서, ~를 원한다. 왜냐하면 ~때문이다" |
| **Acceptance Criteria** | 기능이 완료되었다고 판단하는 기준. QA 및 개발 완료 조건으로 활용 |

## 나만의 체크리스트

PRD 작성 후 이 항목들을 점검:

- [ ] 문제 정의가 데이터로 뒷받침되는가?
- [ ] 성공 지표가 측정 가능한가?
- [ ] Scope가 명확한가?
- [ ] 개발팀이 이해할 수 있는 수준인가?
- [ ] 이해관계자와 Align 되었는가?
