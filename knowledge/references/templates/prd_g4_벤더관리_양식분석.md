## 1. 기존 PRD 양식과의 차이점

| 구분 | 기존 양식 (기능정의 중심) | G4 포맷 (벤더관리 PRD) |
|------|--------------------------|----------------------|
| **문서 단위** | 단일 문서 | **5개 문서로 분리** (Index, 용어집, 정책서, 기능명세, 디자인레퍼런스) |
| **정책 기술** | 기능명세 내 인라인 기술 | **정책서를 독립 문서로 분리** |
| **용어 관리** | 별도 관리 없음 | **용어집 독립 문서** (액터, 상태, 도메인 용어) |
| **기능명세 테이블** | 6컬럼 (우선순위, 기능카테고리, 유저시나리오, 개발요구사항, 레퍼런스, Data분석조건) | **5컬럼** (Priority, 화면/메뉴, 기능카테고리, 유저시나리오, 개발요구사항) |
| **Phase 관리** | 단일 Phase | **Phase 1/2/3 명확 구분 + 스펙아웃 추적** |
| **경쟁사 분석** | 별도 또는 없음 | **정책서 내 경쟁사 레퍼런스 통합** |
| **시나리오 표현** | 테이블 내 기술 | **Callout 블록으로 시나리오 강조** |
| **AS-IS/TO-BE** | 없음 | **비교 테이블로 현재 vs 미래 대비** |
| **지표** | 없음 또는 간략 | **결과 지표(Outcome) + 과정 지표(Process) 분리** |

## 2. 전체 문서 구조 (5개 문서)

```
벤더 관리 (랜딩 페이지)
├── 00 PRD Index       — 개요, Phase 로드맵, 지표, AS-IS/TO-BE, TBD
├── 01 용어집          — 용어집 + 액터 정의
├── 02 정책서          — 핵심 원칙, 전략적 배경, 기능별 정책, 시스템 자동처리, 배차/정산 흐름, 장애대응
├── 03 기능명세 P0     — Phase 1~2 P0 기능 명세 (상세 표)
└── 기타 레퍼런스      — 디자인 레퍼런스, 브레인스토밍, 상세 초안 등
```

### 문서별 역할과 대상

| 문서 | 내용 | 대상 |
|------|------|------|
| **00 PRD Index** | 개요, Phase 로드맵, 지표, AS-IS/TO-BE, TBD | 전체 이해관계자 |
| **01 용어집** | 용어집 + 액터 정의 | 전체 이해관계자 |
| **02 정책서** | 핵심 원칙, 전략적 배경, 기본 정책, 시스템 자동처리, 배차/정산 흐름, 장애대응 | PO / 개발 |
| **03 기능명세 P0** | Phase 1~2 P0 기능 명세 (상세 표) | PO / 개발 / QA |
| **04 디자인 레퍼런스** | 디자인 브레인스토밍 요약 + 경쟁사 레퍼런스 | PO / 디자인 |

## 3. 랜딩 페이지 양식

랜딩 페이지는 전체 PRD의 **진입점**. 간결하게 핵심만 요약.

### 구성 요소

1. **제목 + 한 줄 설명** (인용 블록)
2. **문서 구조 테이블** (어떤 문서가 있고, 누구를 위한 것인지)
3. **Phase별 기능 요약 테이블** (전체 기능 조감도)
4. **하위 문서 링크** (실제 문서로 이동)

### 양식 예시

```markdown
## 벤더관리 시스템 PRD
> B존 지역에 벤더(외부 배송 업체)를 등록·운영하는 시스템 구축
---
### 문서 구조
| 문서 | 내용 | 대상 |
|------|------|------|
| **00 PRD Index** | 개요, Phase 로드맵, 지표, AS-IS/TO-BE, TBD | 전체 이해관계자 |
| ... | ... | ... |
---
### Phase별 기능 요약
| Phase | 기능 | Priority |
|-------|------|----------|
| **Phase 1** | 1️⃣ 기능명 (위치) | **P0** |
| ... | ... | ... |
```

## 4. 00 PRD Index 양식

### 섹션 순서

```
1. 제목 + 원본 링크
2. 문서 구조 테이블
3. 개요 (인용 블록 + 본문)
4. 개발 단계 Overview (Phase별 목표/범위/일정)
5. Phase별 기능 요약 테이블
6. 스펙 아웃 항목 (취소선으로 표시)
7. AS-IS vs TO-BE 비교 테이블
8. 지표
   - 결과 지표 (Outcome)
   - 과정 지표 (Process)
9. 미정의(TBD) / 확인 필요 포인트
10. 디자인 브레인스토밍 (참조 링크)
11. 문서 변경 History
```

### 주요 테이블 양식

#### 개발 단계 Overview

| Phase | 목표 | 핵심 범위 | Priority | 예상 일정 |
|-------|------|----------|----------|----------|
| **Phase 1** | 벤더 운영 체계 구축 | ... | **P0** | Q1 |

#### Phase별 기능 요약

| Phase | 기능 | Priority |
|-------|------|----------|
| **Phase 1** | 1️⃣ 기능명 (위치 설명) | **P0** |

> 기능 앞에 이모지 번호(1️⃣ 2️⃣ ...)를 붙여 추적 가능하게 한다.

#### 스펙 아웃 항목

| 기능 | 원래 Phase | 제외 사유 |
|------|-----------|----------|
| ~~3️⃣ 기능명~~ | Phase 1 | 스펙 아웃 |

> 스펙 아웃된 기능은 **취소선(~~)** 처리하여 시각적으로 구분.

#### AS-IS vs TO-BE

| 항목 | 현재 상황 | 문제점 | TO-BE 변경 내용 |
|------|----------|--------|----------------|

#### 결과 지표 (Outcome)

| 지표명 | 정의 | 목표 | 측정 산식 | 측정 시점 |
|--------|------|------|----------|----------|

#### 과정 지표 (Process)

| 지표명 | 정의 | 목표 | 측정 산식 | 측정 시점 |
|--------|------|------|----------|----------|

| # | 항목 | 현재 상태 | 관련 섹션 |
|---|------|----------|----------|

## 5. 01 용어집 양식

### 섹션 순서

```
1. 제목 + 상위 문서 링크
2. 액터 정의 테이블
3. 용어집 (카테고리별)
   - 벤더/조직
   - 벤더 상태
   - 기사 소속 상태
   - 세트/물량
   - 지역/권역
   - 배차/배송
   - 정산/비용
   - 시스템/운영
   - 실적/지표
4. 약어 정리
```

### 주요 테이블 양식

#### 액터 정의

| 한국어 | 코드 | 정의 | 사용 채널 | 주요 행동 |
|--------|------|------|----------|----------|

#### 일반 용어

| 한국어 | 영어 | 정의 | 관련 기능 |
|--------|------|------|----------|

#### 상태 용어

| 한국어 | 코드 | 정의 | 할 수 있는 것 |
|--------|------|------|-------------|

#### 약어 정리

| 약어 | 풀네임 | 의미 |
|------|--------|------|

### 작성 특징

- 용어를 **도메인별로 카테고리화** (조직, 상태, 물량, 배차, 정산 등)
- 각 용어에 **관련 기능 번호**(1️⃣~)를 연결하여 추적 가능
- 상태값은 **코드(영문)**와 **한국어 의미**를 병기
- 약어를 별도 테이블로 정리하여 빠른 참조 지원

## 6. 02 정책서 양식

### 섹션 순서

```
1. 문서 목적 + 상위/관련 문서 링크
2. 목차 (자동 생성)
3. 핵심 원칙 테이블
4. 전략적 배경 (경쟁사 Reference)
   - 세트 개념의 본질
   - 퀄리티 관리 — 이중 레버
   - 정산 구조 — 인센티브 + 기사 원가
   - 경쟁사 계약 구조 레퍼런스 (쿠팡이츠)
5. 기능별 기본 정책 (기능 번호별)
   - 1️⃣ 벤더 관리
   - 2️⃣ 지점/권역관리
   - 5️⃣ 벤더 포털
   - 6️⃣ 단위물량/정산
   - 8️⃣ 플러스앱
6. 시스템 자동 처리 테이블
7. 배차 흐름 (의사코드 블록)
8. 배송 수익금 흐름 (의사코드 블록)
9. 장애 대응 테이블
```

#### 시스템 자동 처리

| 자동 처리 | 시점 | 설명 |
|----------|------|------|

#### 장애 대응

| 장애 시간 | 예상 영향 |
|----------|----------|

### 의사코드 블록 패턴

```
주문 발생
  → 서비스A가 서비스B에 요청
  → 조건 판단
  → 결과
       ├── 케이스 1 → 처리
       ├── 케이스 2 → 처리
       └── 케이스 3 → 처리
  → 후속 처리
```

### 작성 특징

- 정책과 기능명세를 분리하여 **"왜/어떤 규칙"**과 **"무엇을 만들 것인가"**를 구분
- 경쟁사 레퍼런스를 정책서에 통합 → 정책 결정의 근거 제시
- 의사코드 형태의 흐름도로 복잡한 프로세스를 시각화
- 장애 대응을 시간대별 영향으로 정량화

## 7. 03 기능명세 양식 (핵심)

### 섹션 구조 (기능 단위 반복)

```
## 2.N 기능명 — 부가 설명 (Priority / Phase)

### 기본 정책 (2컬럼 테이블)
### 사용자 시나리오 (Callout 블록)
### 기능 명세 (5컬럼 메인 테이블)
```

### 기본 정책 테이블

| 정책 | 설명 |
|------|------|

> 해당 기능에 적용되는 핵심 정책을 간결하게 요약. 정책서의 상세 내용 중 이 기능에 해당하는 것만 발췌.

### 사용자 시나리오 (Callout 블록)

> **시나리오 1: 시나리오 제목**
> 1. 액터가 ~한다.
> 2. 액터가 ~한다.
> 3. 시스템이 ~한다.
> 4. 결과가 ~이다.

> Callout(회색 배경) 블록으로 시나리오를 강조. 기능명세 테이블 앞에 배치하여 맥락 제공.

### 기능 명세 메인 테이블 (5컬럼)

| Priority | 화면/메뉴 | 기능 카테고리 | 유저 시나리오 | 개발 요구사항 |
|----------|----------|-------------|-------------|-------------|
| P0 | 앱/시스템 > 메뉴 > 하위메뉴 | 리스트 조회 | RM이 ~한다 | (상세 스펙) |

#### 컬럼별 작성 규칙

**Priority**: P0 / P1 / P2

**화면/메뉴**: 계층 구조로 표기
- `인트라 > 세트 관리 > 벤더 관리`
- `벤더 포털 > 소속 기사 관리`
- `플러스앱 > 출근`

**기능 카테고리**: 해당 화면 내 세부 기능
- `리스트 조회`, `벤더 등록`, `상태 변경`, `VO 지정`
- `로그인`, `모드 전환`, `기사 소속 해제`

**유저 시나리오**: 한 문장 형태
- `RM이 등록된 벤더 전체를 조회한다`
- `VO가 소속 기사의 전체 목록을 확인한다`
- `기사가 출근 시 모드가 자동으로 결정된다`

**개발 요구사항**: 구조화된 볼드 키워드 + 불릿 리스트

```markdown
**컬럼**
  • 벤더코드
  • 벤더명
  • 상태

**필터**
  • 상태(대기/운영중/운영중지/해지)
  • 존

**기본 정렬**
  • 등록일 내림차순
```

### 개발 요구사항에 자주 쓰이는 키워드 카테고리

| 키워드 | 용도 | 예시 |
|--------|------|------|
| **컬럼** | 목록/테이블에 표시할 항목 | 벤더코드, 벤더명, 상태 |
| **필터** | 조회 조건 | 상태, 존, 벤더명 |
| **기본 정렬** | 기본 정렬 기준 | 등록일 내림차순 |
| **필수 입력** | 반드시 입력해야 하는 필드 | 사업자등록번호, 사업자명 |
| **선택 입력** | 선택적 입력 필드 | 계약 시작일/종료일 |
| **수정 불가** | 읽기 전용 필드 | 사업자등록번호 |
| **검증** | 저장/실행 시 체크할 조건 | NVP 지점 여부 재검증 |
| **허용 전이** | 상태 변경 가능한 경로 | ACTIVE → INACTIVE / TERMINATED |
| **변경 사유** | 변경 시 필수 입력 | 모든 상태 변경 시 사유 입력 필수 |
| **연쇄 처리** | 상태 변경 시 후속 동작 | 소속 기사 전원 INACTIVE, 세트 회수 |
| **예외** | 예외 상황 처리 | 출근 중이어도 해제 가능 |
| **인증** | 인증/접근 관련 | Plus App 기사 계정 기반 ID/PW 인증 |
| **UI** | UI 요소 설명 | 확인 모달 제공 |
| **이력** | 로그/추적 | 해제 이력 로그 저장 |
| **처리** | 시스템 동작 | 동의 → ACTIVE(소속 확정) |
| **표시** | 화면 노출 | 모드 Badge 상시 표시 |
| **갱신** | 자동 업데이트 | 1분 주기 자동 갱신 |
| **제한** | 제약 조건 | 최대 인원 = 세트당 5명 × 할당 세트 수 |
| **요약 지표** | 대시보드 요약 | 현재 활성 기사 수 / 최대 가능 수 |
| **기타** | 보조 설명 | 등록 완료 시 STANDBY 상태 |

## 10. 양식 적용 체크리스트

새 PRD 작성 시 아래 항목을 확인:

### 랜딩 페이지
- [ ] 한 줄 설명(인용 블록)이 있는가?
- [ ] 문서 구조 테이블이 있는가?
- [ ] Phase별 기능 요약이 있는가?

### 00 Index
- [ ] 개요(왜 만드는가)가 명확한가?
- [ ] Phase별 목표/범위/일정이 있는가?
- [ ] 스펙 아웃 항목이 취소선 처리되었는가?
- [ ] AS-IS vs TO-BE 비교가 있는가?
- [ ] 결과/과정 지표가 측정 산식과 함께 정의되었는가?
- [ ] TBD 항목이 정리되었는가?
- [ ] 변경 History가 기록되었는가?

### 01 용어집
- [ ] 액터 정의가 채널/행동과 함께 정의되었는가?
- [ ] 용어가 도메인별로 카테고리화되었는가?
- [ ] 상태값에 코드(영문)가 병기되었는가?
- [ ] 약어 정리 테이블이 있는가?

### 02 정책서
- [ ] 핵심 원칙이 테이블로 정리되었는가?
- [ ] 경쟁사 레퍼런스와 시사점이 있는가?
- [ ] 기능별 기본 정책이 정리되었는가?
- [ ] 시스템 자동 처리가 시점별로 정리되었는가?
- [ ] 핵심 흐름이 의사코드로 시각화되었는가?
- [ ] 장애 대응이 시간대별로 정의되었는가?

### 03 기능명세
- [ ] 기능 단위로 "기본 정책 → 시나리오 → 기능 명세" 순서인가?
- [ ] 사용자 시나리오가 Callout 블록에 있는가?
- [ ] 5컬럼 테이블(Priority, 화면/메뉴, 기능카테고리, 유저시나리오, 개발요구사항)인가?
- [ ] 개발 요구사항이 볼드 키워드 + 불릿 리스트로 구조화되었는가?
- [ ] TBD 항목이 기능명세 하단에도 있는가?
