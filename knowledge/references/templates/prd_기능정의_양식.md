# PRD 기능정의 양식 가이드

> 안재형 PO의 PRD 문서 분석 결과 (2026-01-21)
> NVP 통합기사앱 프로젝트의 최근 1개월 내 PRD 문서들을 분석하여 정리

---

## 1. 문서 개요

### 이 문서의 목적
- **기능정의에 초점을 맞춘 PRD** 작성 가이드
- 문제정의/지표정의는 별도 문서로 관리 (이 문서 범위 외)
- 개발팀이 바로 작업 가능한 수준의 상세 스펙 제공

### 문서 특성
| 구분 | 설명 |
|------|------|
| **목적** | 개발팀에게 "무엇을 만들어야 하는지" 명확히 전달 |
| **독자** | 개발자, QA, 디자이너 |
| **핵심** | 유저 시나리오 → 개발 요구사항 매핑 |

---

## 2. 문서 구조

PRD는 **3개의 주요 섹션**으로 구성됨:

```
1. 문서 변경 History
2. 프로덕트 영향범위
3. PRD (메인 테이블)
```

### 2.1 문서 변경 History

| 변경 일자 | 변경한 사람 | 변경 내용 | 비고 (참고사항) |
|-----------|------------|----------|----------------|
| 2025-12-01 | @작성자 | 초안 작성 | |
| 2025-12-17 | @작성자 | PRD 리뷰 피드백 반영 | |

**작성 원칙:**
- 최초 작성 시 "초안 작성" 기록
- 이후 변경 시마다 변경 내용 요약 기록
- 리뷰 피드백 반영, 디자인 반영, 정책 변경 등 기록

### 2.2 프로덕트 영향범위

| 프로덕트 구분 | 내용 | 비고 |
|--------------|------|------|
| 신규 앱 | - 메인 화면에서 ~를 확인 가능하다<br>- 기사는 ~를 선택할 수 있다 | |
| 인트라 | - 담당자는 ~를 관리할 수 있다 | |
| 매니저 PC/M | - 사용자는 ~를 확인 가능하다 | |

**작성 원칙:**
- 영향받는 프로덕트별로 변경사항 요약
- 한 문장으로 "누가 무엇을 할 수 있다" 형태
- 상세 내용은 PRD 테이블에서 기술

### 2.3 PRD 메인 테이블

| 우선순위 | 기능 카테고리 | 유저 시나리오 | 개발 요구사항 | 레퍼런스 | Data 분석 조건 |
|---------|--------------|--------------|--------------|---------|---------------|
| P0 | 메인 화면 | 기사는 ~를 확인 가능하다 | (상세 스펙) | (이미지/링크) | (이벤트 정의) |

## 3. 컬럼별 작성 가이드

### 3.1 우선순위

| 레벨 | 의미 | 기준 |
|------|------|------|
| **P0** | Must Have | 출시 필수, 핵심 기능 |
| **P1** | Should Have | 출시 후 빠른 시일 내 추가 |
| **P2** | Nice to Have | 여유 있을 때 추가 |

### 3.2 기능 카테고리

- Epic 또는 기능 영역 단위로 분류
- 예시: `메인 화면 (신규앱)`, `로그인`, `오더 수행`, `인트라`

### 3.3 유저 시나리오

**형식:** `[주체]는 [행동/목표]를 [할 수 있다/한다]`

**좋은 예시:**
- "기사는 신규 오더 제안 여부를 선택할 수 있다"
- "메인 화면 진입 시 노출할 배너가 있을 경우 최우선적으로 표시한다"
- "담당자는 신규 앱에 노출할 배너를 등록/관리할 수 있다"

**특징:**
- 한 문장으로 완결
- 사용자 관점에서 기술
- "무엇을"에 집중 (how가 아닌 what)

### 3.4 개발 요구사항 (핵심)

**구조적 작성 패턴:**

```markdown
**- [대주제를 볼드로 작성]**
  - 세부 사항 1
  - 세부 사항 2
    - 더 상세한 조건
    - 예외 케이스

**- [다음 대주제]**
  - 세부 사항
```

**작성 원칙:**

| 원칙 | 설명 | 예시 |
|------|------|------|
| **계층적 구조** | 대주제 → 세부사항 → 조건 | 볼드 + 들여쓰기 활용 |
| **조건문 명시** | ~일 경우, ~할 때 분기 | "로그인 불가 계정의 경우 올바른 계정 종류가 아님을 안내한다" |
| **예외 처리** | 에러 상황, 엣지 케이스 | "일정 거리 밖에서 시도할 경우 불가 안내 팝업을 노출한다" |
| **수치 명시** | 구체적 값 제시 | "초기값: 100m (논의 필요)" |
| **상태 전이** | 상태값 변화 흐름 | "대기 → 표시 → 종료 상태로 전환된다" |
| **논의 필요 표시** | 미결정 사항 마킹 | 빨간색 하이라이트 또는 (논의 필요) |

**실제 작성 예시:**

```markdown
**- 배너 창은 앱 종료 후 메인화면 진입 시 팝업 형태로 표시한다.**
  - 유저가 배너 이미지를 터치할 경우 배너 등록시 설정한 특정 링크 혹은 앱 상 특정 화면으로 이동 가능하다.
  - 유저는 배너 팝업을 닫을 수 있으며, 원할 경우 오늘 하루 다시보지 않기를 설정할 수 있다.
  - 앱 종료가 아닌 백그라운드로 내린 이후 앱 재진입하여 메인 화면에 진입한 경우에는 배너를 다시 노출하지 않는다.

**- 배너 팝업 상 표시하는 배너 이미지는 한개 이상일 수 있다.**
  - 다수의 배너 이미지를 순서대로 슬라이드 형태로 노출한다.
  - 최대 동시 노출 가능 배너의 수: n개 (논의 필요)
```

### 3.5 레퍼런스

- 인트라 화면 캡처 이미지
- 기존 시스템 참조 화면
- API Response 예시 (코드 블록)

### 3.6 Data 분석 조건

- 이벤트 트래킹 조건
- 로그 수집 필요 항목
- (대부분 PRD 완성 후 별도 정의)

## 5. 작성 체크리스트

PRD 완성 전 확인:

- [ ] 모든 유저 시나리오에 대응하는 개발 요구사항이 있는가?
- [ ] 조건 분기(~일 경우)가 빠짐없이 기술되었는가?
- [ ] 예외 케이스/에러 상황 처리가 명시되었는가?
- [ ] 수치값이 구체적으로 제시되었는가? (미정이면 "논의 필요" 표기)
- [ ] 상태 전이 흐름이 명확한가?
- [ ] 인트라/어드민 기능도 포함되었는가?
- [ ] 개발자가 추가 질문 없이 구현 가능한 수준인가?

## 6. 주요 패턴 및 표현

### 자주 사용되는 표현

| 패턴 | 용도 | 예시 |
|------|------|------|
| `~의 경우에는` | 조건 분기 | "로그인 불가 계정의 경우에는..." |
| `~할 수 있다` | 기능 가능 여부 | "기사는 설정을 변경할 수 있다" |
| `~를 표시한다` | UI 노출 | "잔여 시간을 핀 상단에 표시한다" |
| `~로 전환된다` | 상태 변경 | "대기 상태에서 표시 상태로 전환된다" |
| `~이 불가하다` | 제약 조건 | "픽업 이후에는 직접 배차취소가 불가하다" |
| `(논의 필요)` | 미결정 사항 | "최대 n개 (논의 필요)" |
| `(초기값: N)` | 기본값 명시 | "(초기값: 100m)" |

### 복잡한 로직 기술 시

```markdown
**- 로그인 조건:**
  - 로그인 가능 계정은 관리 지점의 Operation type이 NVP일 경우에만 가능하다.
    - 로그인 불가 계정의 경우 올바른 계정 종류가 아님을 안내한다.
  - 기사 계정의 ID/PW가 정확히 입력되어야 한다.
    - 정확히 입력되지 않은 경우에는 안내를 진행한다.
  - 비밀번호가 만료되지 않은 계정이어야 한다.
    - 만료 정책은 2륜 기사와 동일하게 적용한다.
    - 만료된 비밀번호 입력 시 비밀번호를 갱신할 수 있는 화면으로 이동한다.
```

## 7. 회의록 통합

PRD 문서 하단에 관련 회의록을 포함할 수 있음:

- 기능 설계 회의 내용
- 주요 결정사항 요약
- 액션 아이템 목록

**예시 구조:**
```
## 정리 필요한 정책
(미결 사항 기록)

## 회의록
### [회의 제목] - YYYY-MM-DD
- 주요 결정사항
- 액션 아이템
```
