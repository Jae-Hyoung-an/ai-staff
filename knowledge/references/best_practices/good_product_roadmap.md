# 좋은 프로덕트 로드맵의 원칙

## 핵심 요약

좋은 프로덕트 로드맵은 **명확한 출시물 정의**와 **비즈니스 임팩트 연결**을 동시에 갖춘다.

> *"로드맵은 '무엇이 언제 나오는가'가 아니라, '왜 이것을 만들고 어떤 가치를 기대하는가'를 전달하는 문서다."*

---

## 로드맵의 역할과 범위

### 로드맵이 다뤄야 할 것
- **출시 프로덕트**: 각 단계에서 고객에게 전달되는 구체적인 결과물
- **비즈니스 목표**: 해당 출시물이 달성하려는 비즈니스 임팩트
- **시간 프레임**: 언제 출시되는가 (단, 과도하게 구체적인 날짜는 지양)
- **우선순위 근거**: 왜 이 순서로 진행하는가
- **의존성**: 선후 관계가 있는 항목들의 연결

### 로드맵이 다루지 않아야 할 것 (별도 문서로 분리)

| 내용 | 담당 문서 |
|------|----------|
| 상세 기능 명세, 정책 | PRD |
| 스프린트별 작업 분배 | 스프린트 플랜 |
| 기술적 구현 방식 | 개발 설계 문서 |
| 출시 후 변경 내역 | 릴리즈 노트 |
| 개별 작업 항목 | 백로그 |

### 유사 문서와의 구분

| 문서 | 초점 | 시간 범위 | 주요 독자 |
|------|------|----------|----------|
| **로드맵** | 전략적 방향, 비즈니스 임팩트 | 분기~연간 | 경영진, 사업팀, 전체 조직 |
| **백로그** | 구현할 기능/작업 목록 | 지속 관리 | PM, 개발팀 |
| **스프린트 플랜** | 이번 스프린트 작업 | 1~4주 | 개발팀 |
| **PRD** | 개별 기능의 상세 요구사항 | 기능 단위 | PM, 개발, 디자인, QA |
| **릴리즈 노트** | 출시된 변경 사항 | 출시 후 | 고객, 내부 전체 |

### 로드맵 유형별 활용

목적과 대상에 따라 로드맵의 유형을 구분하여 커뮤니케이션 효과를 높일 수 있다.

| 유형 | 초점 | 적합한 대상 | 상세도 |
|------|------|------------|--------|
| **전략 로드맵** | 비즈니스 목표, 고차원 이니셔티브 | 경영진, 이사회 | 낮음 (방향 중심) |
| **포트폴리오 로드맵** | 여러 제품/팀의 릴리스 현황 | 사업부 리더, 여러 팀 협업 시 | 중간 |
| **출시 로드맵** | 구체적 기능, 출시 일정 | 영업, 마케팅, CS | 높음 (기능 중심) |
| **기술 로드맵** | 인프라, 기술 부채 해결 | 개발팀, CTO | 높음 (기술 중심) |
| **시장 기반 로드맵** | 경쟁 대응, 시장 트렌드 반영 | 경영진, 전략팀 | 중간 |

> 💡 **팁**: 하나의 마스터 로드맵을 관리하되, 대상에 맞는 "뷰(View)"를 제공하는 것이 효율적이다.

## 1. 단계별 명확한 출시 프로덕트 정의

### 원칙
> 각 마일스톤에서 "무엇이 고객에게 전달되는가"가 구체적이어야 한다. 추상적인 목표가 아닌 실체가 있는 출시물로 정의한다.

### 왜 중요한가
- 비즈니스 관계자가 "이번에 뭐가 나오는 거야?"에 즉답 가능
- 개발팀이 목표 범위를 명확히 이해
- 일정 지연 시 "무엇이 빠지는지" 판단 가능

### 출시물 정의 체크리스트
- [ ] 고객이 경험할 수 있는 변화가 명시되어 있는가?
- [ ] 출시 형태가 구체적인가? (신규 기능, 개선, 플랫폼 등)
- [ ] 해당 출시로 해결되는 고객 문제가 연결되어 있는가?
- [ ] "무엇이 포함되지 않는가"(Scope Out)가 명확한가?

### 좋은 출시물 정의 vs 나쁜 출시물 정의

| 나쁜 예시 ❌ | 좋은 예시 ✅ |
|-------------|-------------|
| "검색 개선" | "검색 결과 페이지에 필터 기능 추가 (카테고리, 가격대, 평점)" |
| "결제 고도화" | "간편결제 3종 추가 (카카오페이, 네이버페이, 토스)" |
| "사용자 경험 개선" | "상품 상세 페이지 로딩 속도 2초 이내로 단축" |
| "Phase 2 개발" | "판매자 대시보드 v1: 일별 매출, 주문 현황, 재고 알림" |
| "인프라 작업" | "동시 접속 10만 명 처리 가능한 서버 확장 완료" |

### 출시물 단위 구분

| 출시 유형 | 설명 | 예시 |
|----------|------|------|
| **신규 기능** | 없던 것을 새로 만듦 | 위시리스트 기능 출시 |
| **기능 개선** | 기존 기능의 사용성/성능 향상 | 검색 자동완성 추가 |
| **기술 기반** | 향후 기능의 토대 (사용자 경험 변화 제한적) | API 버전 업그레이드 |
| **운영 효율화** | 내부 프로세스 개선 | CS 응대 자동화 도구 |

## 2. 비즈니스 임팩트 연결

### 원칙
> 모든 출시물은 "왜 만드는가"에 대한 비즈니스 관점의 답을 갖고 있어야 한다.

### 왜 중요한가
- 경영진/사업팀이 "이게 왜 중요한 거야?"에 답할 수 있음
- 우선순위 논의 시 객관적 기준 제공
- 출시 후 성과 측정의 기준점 확보

### 비즈니스 임팩트 유형

| 임팩트 유형 | 설명 | 예시 |
|------------|------|------|
| **매출 증대** | 직접적 수익 증가 | 결제 전환율 +5% |
| **비용 절감** | 운영/지원 비용 감소 | CS 문의 20% 감소 |
| **고객 확보** | 신규 고객 유입 | 신규 가입 +30% |
| **고객 유지** | 이탈 방지, 재구매 | 재구매율 +10% |
| **시장 확장** | 새로운 시장/세그먼트 진입 | B2B 시장 진출 |
| **경쟁 대응** | 경쟁사 대비 열위 해소 | 경쟁사 수준 기능 확보 |
| **리스크 완화** | 법적/기술적 리스크 제거 | 개인정보보호법 준수 |

### 임팩트 연결 형식

```markdown
## Q2 출시: 간편결제 추가

### 출시물
- 카카오페이, 네이버페이, 토스 연동

### 비즈니스 임팩트
| 지표 | 현재 | 목표 | 기대 근거 |
|------|------|------|----------|
| 결제 전환율 | 68% | 75% | 간편결제 도입 타사 사례 평균 +7%p |
| 결제 포기율 | 32% | 25% | 결제 단계 간소화 효과 |

### 왜 지금인가
- 경쟁사 3사 모두 간편결제 지원 중
- 고객 VOC "결제가 불편하다" 상위 3위
```

### 임팩트 정의 체크리스트
- [ ] 측정 가능한 지표로 표현되었는가?
- [ ] 현재 수치(baseline)가 명시되었는가?
- [ ] 목표 수치와 기대 근거가 있는가?
- [ ] "왜 지금 해야 하는가"가 설명되었는가?

## 3. 시간 프레임워크

### 원칙
> 가까운 미래는 구체적으로, 먼 미래는 유연하게. 모든 일정에 확정 수준을 명시한다.

### Now-Next-Later 프레임워크

| 구분 | 시간 범위 | 확정 수준 | 상세도 |
|------|----------|----------|--------|
| **Now** | 현재 분기 | 높음 (Committed) | 출시물, 목표, 일정 명확 |
| **Next** | 다음 분기 | 중간 (Planned) | 출시물 확정, 일정은 대략적 |
| **Later** | 그 이후 | 낮음 (Exploring) | 방향성과 기대 임팩트 중심 |

### 확정 수준 표기

| 표기 | 의미 | 변경 가능성 |
|------|------|------------|
| **Committed** | 일정/범위 확정, 리소스 배정 완료 | 낮음 (변경 시 공지 필요) |
| **Planned** | 범위 확정, 일정은 조정 가능 | 중간 |
| **Exploring** | 방향성만 결정, 실행 여부 미정 | 높음 |
| **On Hold** | 보류 (사유 명시) | - |

### 시간 표기 원칙

| 나쁜 예시 ❌ | 좋은 예시 ✅ |
|-------------|-------------|
| "2026년 3월 15일 출시" | "Q1 2026 (Committed)" |
| "곧 출시" | "Q2 2026 (Planned)" |
| "하반기" | "H2 2026 (Exploring)" |
| "미정" | "2026 Later (Exploring) - 리소스 확보 시" |

### 왜 정확한 날짜를 피하는가
- 로드맵은 전략 문서, 프로젝트 일정표가 아님
- 날짜 변경 시 신뢰도 저하
- 유연성 확보로 시장 변화 대응 가능

## 4. 이해관계자 커뮤니케이션

### 원칙
> 같은 로드맵이라도 독자에 따라 강조점이 달라야 한다. 비즈니스 관계자는 "왜"와 "언제"에, 개발팀은 "무엇"과 "어떻게"에 관심이 많다.

### 관계자별 관심사와 커뮤니케이션 포인트

| 이해관계자 | 주요 관심사 | 로드맵에서 강조할 것 |
|-----------|------------|-------------------|
| **경영진** | 비즈니스 성과, 전략 정렬 | 매출/성장 임팩트, 경쟁 대응 |
| **영업팀** | 고객에게 약속할 수 있는 것 | 출시 시점, 고객 가치 |
| **마케팅팀** | 캠페인/런칭 타이밍 | 확정 일정, 차별화 포인트 |
| **CS팀** | 고객 문의 대응 준비 | 변경 사항, 교육 필요 시점 |
| **개발팀** | 기술적 실현 가능성, 리소스 | 범위, 의존성, 우선순위 |
| **디자인팀** | UX 방향, 작업 범위 | 사용자 문제, 영향 범위 |

### 비즈니스 관계자용 요약 형식 (Executive Summary)

```markdown
## 2026 상반기 로드맵 요약

### 핵심 메시지
"상반기에는 결제 경험 개선에 집중하여 전환율 10%p 향상을 목표합니다."

### 주요 출시 일정

| 시기 | 출시물 | 비즈니스 임팩트 | 확정 수준 |
|------|--------|----------------|----------|
| Q1 | 간편결제 3종 추가 | 결제 전환율 +7%p | Committed |
| Q1 | 결제 페이지 리뉴얼 | 결제 포기율 -15% | Committed |
| Q2 | 정기 결제 기능 | 구독 매출 신규 창출 | Planned |

### 주요 변경 사항 (vs 이전 로드맵)
- 추가: 정기 결제 (고객 요청 상위)
- 연기: 해외 결제 → H2로 이동 (리소스 재배치)
```

### 로드맵 리뷰 미팅 운영

| 미팅 유형 | 주기 | 참석자 | 목적 |
|----------|------|--------|------|
| **전사 공유** | 분기 | 전체 | 방향 정렬, 주요 변경 공유 |
| **사업팀 싱크** | 월간 | PM, 사업팀 | 우선순위 조율, 피드백 수렴 |
| **개발팀 플래닝** | 격주 | PM, 개발, 디자인 | 실행 계획 구체화 |

### 전사적 피드백 루프 구축

로드맵은 제품팀만의 산출물이 아니라 **전사적 협업의 결과물**이어야 한다.

| 피드백 소스 | 수집 방법 | 반영 방식 |
|------------|----------|----------|
| **고객 VOC** | CS팀 정기 리포트, NPS 조사 | 우선순위 조정 근거로 활용 |
| **영업팀** | 주간 싱크, 딜 파이프라인 리뷰 | 고객 요청 기능 반영 |
| **마케팅팀** | 캠페인 성과, 시장 트렌드 | 타이밍 조정 |
| **운영팀** | 운영 이슈 리포트 | 운영 효율화 항목 추가 |

```
[피드백 수집] → [우선순위 검토] → [로드맵 반영] → [변경 공유]
     ↑                                                    │
     └────────────────────────────────────────────────────┘
                     (지속적인 순환)
```

## 5. 리스크 및 종속성 관리

### 원칙
> 로드맵은 계획일 뿐, 실행 과정에서의 리스크와 종속성을 명시하여 이해관계자의 기대치를 관리한다.

### 왜 중요한가
- 비즈니스 관계자는 "계획대로 될 것인가?"에 민감함
- 사전에 리스크를 공유하면 지연 시 신뢰 유지 가능
- 종속성 파악으로 병목 현상 예방

### 종속성(Dependencies) 유형

| 종속성 유형 | 설명 | 예시 |
|------------|------|------|
| **기술적 종속성** | 선행 개발이 완료되어야 진행 가능 | "간편결제 인프라 → 정기 결제 기능" |
| **팀 간 종속성** | 다른 팀의 작업 완료 필요 | "디자인 시스템 완성 → UI 개편" |
| **외부 종속성** | 외부 파트너/API 연동 필요 | "PG사 계약 체결 → 결제 연동" |
| **비즈니스 종속성** | 경영 의사결정 필요 | "해외 진출 결정 → 다국어 지원" |

### 리스크 식별 및 표기

```markdown
### [출시물: 간편결제 추가]

#### 리스크 및 대응
| 리스크 | 영향도 | 발생 가능성 | 대응 방안 |
|--------|--------|------------|----------|
| PG사 계약 지연 | 높음 | 중간 | 대안 PG사 협의 병행 |
| 개발 리소스 부족 | 중간 | 낮음 | Q1 채용 완료 예정 |

#### 종속성
- **선행**: 결제 모듈 리팩토링 완료 (Q4 완료 예정)
- **후행**: 정기 결제, 해외 결제 기능
```

### 리스크 관리 체크리스트
- [ ] 기술적 난이도가 높은 항목이 식별되었는가?
- [ ] 외부 파트너/계약 관련 종속성이 파악되었는가?
- [ ] 리소스(인력, 예산) 제약이 반영되었는가?
- [ ] 각 리스크의 대응 방안이 준비되었는가?

## 6. 변경 관리와 Living Document

### 원칙
> 로드맵은 한 번 만들고 끝나는 문서가 아니라, 지속적으로 업데이트되는 "살아있는 문서"다.

### 변경 트리거

다음 상황에서는 반드시 로드맵을 업데이트해야 한다:

1. **우선순위 변경**: 비즈니스 환경 변화로 순서 조정
2. **범위 변경**: 출시물 내용 추가/축소
3. **일정 변경**: 예상 출시 시점 변경
4. **신규 항목 추가**: 새로운 이니셔티브 등장
5. **항목 제거/보류**: 더 이상 진행하지 않는 항목

### 변경 이력 기록

```markdown
## 변경 이력

| 날짜 | 변경 내용 | 변경 사유 | 담당자 |
|------|----------|----------|--------|
| 2026-01-20 | 해외 결제 Q2→H2 연기 | 간편결제 리소스 집중 | @jaehyoung |
| 2026-01-15 | 정기 결제 Q2 추가 | 고객 VOC 상위 & 구독 모델 사업 계획 | @jaehyoung |
| 2026-01-10 | 2026 H1 로드맵 v1.0 | 초기 작성 | @jaehyoung |
```

### 변경 커뮤니케이션 원칙

| 변경 규모 | 커뮤니케이션 방식 |
|----------|------------------|
| 소규모 (범위 미세 조정) | 문서 업데이트 + 관련자 노티 |
| 중규모 (일정 변경, 항목 추가/삭제) | 관련 팀 미팅 공유 |
| 대규모 (전략 방향 변경) | 전사 공지 + 경영진 승인 |

## 7. 로드맵 구조 템플릿

### 권장 구조

```markdown

## 메타 정보
- 작성자: 
- 최종 수정일: 
- 버전: 
- 검토 주기: 월간/분기

## Executive Summary
- 핵심 메시지 (1~2문장)
- 주요 출시물과 비즈니스 임팩트 요약 표

---

## Now (현재 분기)

### [출시물 1: 간편결제 추가]
| 항목 | 내용 |

| **출시 시점** | Q1 2026 (Committed) |
| **출시물** | 카카오페이, 네이버페이, 토스 연동 |
| **해결하는 문제** | 결제 단계 복잡 → 이탈 발생 |
| **비즈니스 임팩트** | 결제 전환율 68% → 75% (+7%p) |
| **성공 지표** | 결제 전환율, 결제 포기율 |
| **Scope Out** | 해외 결제 수단은 미포함 |

### [출시물 2: ...]

---

## Next (다음 분기)

### [출시물 3: 정기 결제]
| 항목 | 내용 |
|------|------|
| **출시 시점** | Q2 2026 (Planned) |
| **출시물** | 월/연 단위 자동 결제 기능 |
| **해결하는 문제** | 반복 구매 고객의 재결제 번거로움 |
| **비즈니스 임팩트** | 구독 매출 신규 창출 (월 X억 목표) |

- 백로그 링크
```

## 8. 피해야 할 안티패턴

| 안티패턴 | 설명 | 결과 |
|----------|------|------|
| **기능 나열만** | 비즈니스 임팩트 없이 기능 목록만 | "왜 이걸 하는데?" 질문 반복 |
| **날짜 과잉 확정** | 모든 항목에 정확한 날짜 | 변경 시 신뢰도 급락 |
| **추상적 목표** | "UX 개선", "성능 향상" | 완료 여부 판단 불가 |
| **업데이트 안 함** | 초기 버전 그대로 방치 | 실제와 괴리, 참조 가치 상실 |
| **모든 것 포함** | 백로그 전체를 로드맵에 | 전략적 우선순위 불명확 |
| **Scope Creep** | 매번 새 항목 추가만 | 핵심 출시물 지연 |
| **확정 수준 미표기** | Committed/Exploring 구분 없음 | 기대치 관리 실패 |
| **한 방향 소통** | 공유만 하고 피드백 수렴 안 함 | 현장과 괴리된 로드맵 |
| **개발팀 전용** | 비즈니스 관계자 이해 불가 | 조직 정렬 실패 |
| **리스크 무시** | 종속성/리스크 표기 없음 | 지연 시 신뢰 상실 |
| **단일 뷰만 제공** | 모든 대상에 같은 상세도 | 경영진 피로, 실무진 정보 부족 |
| **KPI 연결 누락** | 성공 지표 없이 출시만 목표 | 성과 측정 불가 |

## 9. 로드맵 작성 전 자문

1. 각 출시물이 **고객에게 어떤 변화**를 주는지 한 문장으로 설명할 수 있는가?
2. 각 출시물의 **비즈니스 임팩트**가 측정 가능한 지표로 연결되어 있는가?
3. **비즈니스 관계자**가 이 문서만 보고 "언제 뭐가 왜 나오는지" 이해할 수 있는가?
4. **확정 수준**(Committed/Planned/Exploring)이 명확히 표기되어 있는가?
5. **우선순위 근거**가 설명되어 있는가? (왜 이 순서인가?)
6. **리스크와 종속성**이 식별되고 대응 방안이 있는가?
7. 정기적인 **업데이트 주기**가 정해져 있는가?
8. **변경 이력**이 관리되고 있는가?
9. **전사 피드백**(영업, CS, 마케팅 등)이 반영되었는가?
10. 관련 **이해관계자들과 리뷰**가 완료되었는가?

## 좋은 로드맵 공식

```
좋은 로드맵 = 명확한 출시물 + 비즈니스 임팩트 연결 + 적절한 시간 프레임 + 지속적 업데이트
```

| 요소 | 질문 |
|------|------|
| 명확한 출시물 | 비개발자도 "뭐가 나오는지" 이해할 수 있는가? |
| 비즈니스 임팩트 | "왜 만드는가"에 대한 명확한 답이 있는가? |
| 시간 프레임 | 가까운 것은 구체적, 먼 것은 유연하게 관리되는가? |
| 지속적 업데이트 | 최신 상태를 신뢰할 수 있는가? |

---

*"좋은 로드맵은 전체 조직을 하나의 방향으로 정렬시키고, 나쁜 로드맵은 혼란과 불신을 만든다."*
