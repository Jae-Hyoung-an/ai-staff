# [Best Practice] 좋은 프로젝트 관리

> **핵심 메시지**: 프로젝트 관리의 핵심은 '관리'가 아니라 **'불확실성 속에서 가치를 전달하는 것'**이다.  
> 계획대로 되는 프로젝트는 없다. 변화에 적응하면서 목표를 달성하는 것이 진짜 역량이다.

---

## 핵심 요약

PM(Product Manager/Owner)에게 프로젝트 관리란 단순히 일정 관리가 아니다. **목표 정의 → 범위 관리 → 팀 조율 → 리스크 대응 → 가치 전달**의 전 과정을 리드하는 것이다. 애자일이든 워터폴이든 핵심은 같다: **작은 단위로 빠르게 학습하고, 고객에게 가치를 지속적으로 전달**하는 것.

---

## 1. 프로젝트 관리 기본

### 1.1 프로젝트의 3대 제약 (Triple Constraint)

```
                    범위 (Scope)
                        △
                       ╱ ╲
                      ╱   ╲
                     ╱     ╲
                    ╱   품질  ╲
                   ╱  (Quality) ╲
                  ╱_____________╲
               시간             비용
              (Time)          (Cost)

→ 하나를 변경하면 다른 것도 영향 받음
→ 모든 것을 동시에 만족할 수 없음
→ 우선순위 결정이 필수
```

### 1.2 PM의 역할

| 역할 | 설명 |
|------|------|
| **목표 정의** | 왜 이 프로젝트를 하는가? 성공 기준은? |
| **범위 관리** | 무엇을 하고 하지 않을 것인가? |
| **팀 조율** | 개발, 디자인, QA 등 협업 조율 |
| **커뮤니케이션** | 이해관계자와 지속적 소통 |
| **리스크 관리** | 문제 예측 및 선제적 대응 |
| **의사결정** | 트레이드오프 상황에서 결정 |

## 2. 애자일 프로젝트 관리

### 2.1 애자일 핵심 원칙

```
[애자일 선언 (요약)]

1. 프로세스보다 → 개인과 상호작용
2. 문서보다 → 작동하는 소프트웨어
3. 계약보다 → 고객과의 협력
4. 계획보다 → 변화에 대응

→ 왼쪽도 가치 있지만, 오른쪽에 더 높은 가치
```

### 2.2 스크럼 프레임워크

```
┌─────────────────────────────────────────────────────────┐
│                     SCRUM FRAMEWORK                     │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   [Product      [Sprint      [Sprint]   [Potentially   │
│    Backlog]  →   Planning] →    ↓    →   Shippable     │
│                              ╭─────╮     Increment]    │
│    우선순위      무엇을?     │Daily│                    │
│    정렬된       얼마나?     │Scrum│     작동하는       │
│    기능 목록    왜?         ╰─────╯     결과물         │
│                    ↑          │    ↘                   │
│                    │          ↓     ╲                  │
│                    │     ╭────────╮  ↘                 │
│                    ╰─────│Sprint  │   [Retro-          │
│                          │Review  │    spective]       │
│                          ╰────────╯                    │
│                           데모/검토    회고/개선        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 2.3 스프린트 구성

| 이벤트 | 목적 | 시간 (2주 스프린트 기준) |
|--------|------|------------------------|
| **Planning** | 스프린트 목표 및 작업 결정 | 2-4시간 |
| **Daily Scrum** | 진행 상황, 장애물 공유 | 15분 |
| **Review** | 결과물 데모, 피드백 | 1-2시간 |
| **Retrospective** | 프로세스 개선 논의 | 1-1.5시간 |

### 2.4 스프린트 플래닝 실무

**플래닝 전 준비 (PM)**

```
1. 백로그 정리 (Refinement)
   - 우선순위 정렬
   - 스토리 상세화
   - AC(Acceptance Criteria) 명확화

2. 스프린트 목표 초안
   - "이번 스프린트에서 검증할 것은..."

3. 의존성/리스크 파악
   - 다른 팀 작업과의 연계
   - 기술적 불확실성
```

**스토리 포인트 추정**

```
[피보나치 스토리 포인트]

  1  - 몇 시간 (매우 간단)
  2  - 하루 미만
  3  - 1-2일
  5  - 3-4일
  8  - 일주일 (복잡)
 13  - 스프린트 전체 (쪼개기 필요)
 21+ - 너무 큼, 반드시 분해

→ 시간이 아닌 "상대적 복잡도"로 추정
→ 팀 전체가 함께 추정 (Planning Poker)
```

## 3. 일정 관리

### 3.1 일정 산정 원칙

```
[개발자 추정 × 버퍼 = 현실적 일정]

개발자 추정: 3일
일반 버퍼 (×1.5): 4.5일
불확실성 높으면 (×2): 6일

→ "언제 되나요?" 대신
   "언제까지 90% 확률로 되나요?"
```

**버퍼가 필요한 이유**

```
• 요구사항 변경
• 기술적 예상치 못한 문제
• 코드 리뷰/테스트 피드백
• 팀원 부재 (휴가, 병가)
• 다른 긴급 업무 (버그 수정)
```

### 3.2 마일스톤 관리

```
┌────────────────────────────────────────────────────────────┐
│              📅 간편결제 프로젝트 마일스톤                   │
├──────────────┬──────────┬──────────┬────────────────────────┤
│   마일스톤   │   날짜   │   상태   │   설명                 │
├──────────────┼──────────┼──────────┼────────────────────────┤
│ M1: 설계 완료│  1/15    │   ✅    │ 기술 스펙, 화면 설계   │
│ M2: MVP 개발 │  2/1     │   🔄    │ 토스/카카오페이 연동   │
│ M3: QA 완료  │  2/15    │   ⬜    │ 테스트 및 버그 수정    │
│ M4: 출시     │  2/22    │   ⬜    │ 프로덕션 배포          │
└──────────────┴──────────┴──────────┴────────────────────────┘

✅ = 완료  🔄 = 진행중  ⬜ = 예정
```

    Week1  Week2  Week3  Week4  Week1  Week2  Week3
    ─────────────────────────────────────────────────

    ─────────────────────────────────────────────────
         M1▲                        M2▲  M3▲   M4▲

## 4. 리스크 관리

### 4.1 리스크 식별

| 리스크 유형 | 예시 |
|------------|------|
| **기술** | API 연동 실패, 성능 이슈 |
| **리소스** | 핵심 개발자 이탈, 병가 |
| **범위** | 추가 요구사항, 스펙 변경 |
| **외부** | 외부 서비스 장애, 규제 변경 |
| **일정** | 다른 프로젝트 지연 영향 |

### 4.2 리스크 매트릭스

```
                    발생 확률 높음
                         │
    ┌────────────────────┼────────────────────┐
    │                    │                    │
    │    [모니터링]       │    [최우선 대응]    │
    │    계획 B 준비      │    즉시 대응 필요   │
영향│                    │                    │
높음├────────────────────┼────────────────────┤
    │                    │                    │
    │    [수용]          │    [대응 계획]      │
    │    발생 시 처리     │    완화 방안 준비   │
영향│                    │                    │
낮음└────────────────────┼────────────────────┘
                         │
                    발생 확률 낮음
```

### 4.3 리스크 대응 전략

| 전략 | 설명 | 예시 |
|------|------|------|
| **회피** | 리스크 원인 제거 | 불확실한 기술 대신 검증된 기술 선택 |
| **완화** | 발생 확률/영향 감소 | 조기 프로토타입으로 기술 검증 |
| **전가** | 제3자에게 전가 | 보험, 외주 계약 조건 |
| **수용** | 리스크 감수 | 영향 낮으면 발생 시 대응 |

## 5. 팀 커뮤니케이션

### 5.1 커뮤니케이션 계획

| 대상 | 빈도 | 채널 | 내용 |
|------|------|------|------|
| **개발팀** | 매일 | 데일리 스크럼 | 진행, 장애물 |
| **디자인팀** | 주 2회 | 싱크 미팅 | 디자인 리뷰 |
| **경영진** | 주간 | 메일/보고 | 상태 업데이트 |
| **이해관계자** | 격주 | 데모 | 결과물 공유 |

### 5.2 효과적인 데일리 스크럼

```
[좋은 데일리]

각 팀원 공유 (1-2분):
1. 어제 한 일 (완료 중심)
2. 오늘 할 일 (목표 중심)
3. 장애물 (블로커 있으면)

PM 역할:
• 장애물 해결 지원
• 이슈 파악 및 조율
• 필요 시 후속 논의 예약
```

```
[나쁜 데일리]

❌ 15분 넘어감 (집중 분산)
❌ 디테일한 기술 논의 (별도 미팅으로)
❌ 보고 형식 (팀원 간 공유가 목적)
❌ PM만 말함 (팀원이 주체)
```

### 5.3 비동기 커뮤니케이션

```
[슬랙 채널 구조 예시]

#proj-간편결제-general    ← 일반 논의
#proj-간편결제-dev        ← 개발 관련
#proj-간편결제-standup    ← 데일리 봇
#proj-간편결제-alerts     ← 모니터링 알림

[문서화]
• Confluence/Notion: 결정사항, 스펙
• Jira: 태스크, 이슈 트래킹
• Figma: 디자인 협업
```

## 6. 회고 (Retrospective)

### 6.1 회고의 목적

```
"같은 실수를 반복하지 않고,
좋은 것은 더 잘하기 위해"

→ 비난이 아닌 개선에 집중
→ 사람이 아닌 프로세스에 집중
```

### 6.2 회고 포맷

**4Ls (Liked, Learned, Lacked, Longed for)**

```
┌─────────────────────┬─────────────────────┐
│      좋았던 것      │       배운 것       │
│      (Liked)        │      (Learned)      │
├─────────────────────┼─────────────────────┤
│ • 데일리 스크럼 효과│ • 조기 테스트 중요  │
│ • 팀 협업 원활      │ • 스펙 문서화 필수  │
├─────────────────────┼─────────────────────┤
│      부족했던 것    │       바라는 것     │
│      (Lacked)       │    (Longed for)     │
├─────────────────────┼─────────────────────┤
│ • QA 시간 부족      │ • 더 충분한 버퍼    │
│ • 디자인 싱크 지연  │ • 자동화 테스트     │
└─────────────────────┴─────────────────────┘
```

**Start-Stop-Continue**

```
[Start] 새로 시작할 것
• 매주 디자인 싱크 미팅
• 코드 리뷰 체크리스트

[Stop] 그만할 것
• 스펙 없이 개발 시작
• 긴 회의

[Continue] 계속할 것
• 데일리 스크럼
• 주간 데모
```

### 6.3 액션 아이템 관리

```
[회고 후 액션 아이템]

┌────────┬───────────────────┬──────────┬──────────┐
│   ID   │       액션        │   담당   │   기한   │
├────────┼───────────────────┼──────────┼──────────┤
│ ACT-01 │ QA 시간 +3일 확보 │ PM       │ 다음 플래닝│
│ ACT-02 │ 디자인 싱크 주2회 │ 디자이너 │ 즉시      │
│ ACT-03 │ 코드 리뷰 가이드  │ 개발리드 │ 2주 내    │
└────────┴───────────────────┴──────────┴──────────┘

→ 다음 회고에서 이행 여부 점검
```

## 7. 문제 상황 대응

### 7.1 일정 지연 시

```
[상황] 2주 예정 기능이 3주 걸릴 것으로 예상

[대응 옵션]
1. 범위 축소 (MVP로 출시)
2. 리소스 추가 (다른 팀원 투입)
3. 일정 연장 (이해관계자 합의)
4. 품질 타협 (권장하지 않음)

[커뮤니케이션]
• 빠르게 공유 (늦을수록 옵션 줄어듦)
• 원인과 대안 함께 제시
• 결정 요청 (어떤 옵션을 선택할지)
```

### 7.2 요구사항 변경 시

```
[상황] 중간에 새로운 기능 추가 요청

[대응]
1. 변경 영향 분석
   - 일정 영향: +2주
   - 기존 작업 영향: 설계 변경 필요

2. 트레이드오프 제시
   - 옵션 A: 추가 기능 포함, 출시 2주 연기
   - 옵션 B: 기존 범위 유지, 신규 기능 다음 릴리즈
   - 옵션 C: 다른 기능 제외하고 신규 기능 포함

3. 결정 요청
   - 의사결정자에게 옵션 제시
   - 결정 후 팀 및 이해관계자 공유
```

### 7.3 팀 갈등 시

```
[상황] 개발팀과 디자인팀 의견 충돌

[대응]
1. 각자 관점 경청
   - 왜 그렇게 생각하는지 이해

2. 공통 목표 확인
   - "우리가 달성하려는 것은?"

3. 객관적 기준 적용
   - 데이터, 사용자 피드백 기반 판단

4. 필요시 결정
   - 합의 안 되면 PM이 결정하고 책임
```

## 8. 피해야 할 안티패턴

### ❌ 안티패턴 1: 마이크로매니지먼트

```
문제:
• 모든 작업을 일일이 확인/지시
• 팀의 자율성 저해
• PM이 병목 현상

해결:
• 목표와 기대치만 명확히
• 팀을 신뢰하고 위임
• 결과로 판단
```

### ❌ 안티패턴 2: 문서만 쓰는 PM

```
문제:
• 스펙/문서만 쓰고 실행에 관여 안 함
• 현장 상황 파악 못 함
• "전달했으니 끝" 마인드

해결:
• 팀과 함께 문제 해결
• 장애물 직접 제거
• 진행 상황 상시 파악
```

### ❌ 안티패턴 3: 계획에 집착

```
문제:
• 상황 변해도 원래 계획 고수
• 유연성 부족
• 실패하면 "계획대로 했는데..."

해결:
• 계획은 가설, 실행하며 수정
• 목표는 유지, 방법은 유연하게
• 변화 시 빠르게 공유
```

### ❌ 안티패턴 4: 버퍼 없는 일정

```
문제:
• 낙관적 추정만으로 일정 수립
• 항상 일정 지연
• 크런치 모드 반복

해결:
• 불확실성 인정하고 버퍼 포함
• 최선/최악/예상 시나리오 제시
• 경험 기반 추정 보정
```
