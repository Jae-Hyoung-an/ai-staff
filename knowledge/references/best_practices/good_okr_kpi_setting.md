# [Best Practice] 좋은 OKR/KPI 설계

> **핵심 메시지**: 좋은 지표는 팀이 나아갈 방향을 제시하고, 성공을 객관적으로 판단할 수 있게 한다.  
> OKR은 '어디로 갈 것인가(방향)'를, KPI는 '잘 가고 있는가(속도)'를 알려주는 나침반이다.

---

## 핵심 요약

OKR(Objectives and Key Results)과 KPI(Key Performance Indicators)는 목적이 다르다. OKR은 **야심찬 목표와 그 달성 여부를 측정하는 결과 지표**이고, KPI는 **일상적인 비즈니스 건강 상태를 모니터링하는 지표**다. 둘을 적절히 조합하면 팀은 큰 그림을 보면서도 세부적인 실행을 관리할 수 있다.

---

## 1. OKR vs KPI 구분

### 1.1 개념 비교

| 구분 | OKR | KPI |
|------|-----|-----|
| **목적** | 도전적 목표 달성, 혁신 | 운영 효율, 건강 상태 모니터링 |
| **특징** | 야심찬, 질적 목표 + 정량적 결과 | 정량적, 지속적 측정 |
| **주기** | 분기별 설정 및 리뷰 | 상시 모니터링 |
| **달성 기준** | 70% 달성도 성공으로 간주 | 100% 달성이 기본 |
| **연결** | 전략적 방향성 | 일상 운영 |

### 1.2 언제 무엇을 쓸까?

```
OKR이 적합한 경우:
- 새로운 시장 개척, 신규 기능 출시
- 조직 문화 변화, 프로세스 혁신
- "한 번도 해보지 않은 것"에 도전

KPI가 적합한 경우:
- 서비스 안정성 모니터링
- 운영 효율 관리 (응답 시간, 처리량)
- "유지해야 하는 것"의 상태 확인
```

## 2. 좋은 OKR 작성법

### 2.1 OKR의 구조

```
Objective (목표)
├── Key Result 1 (핵심 결과)
├── Key Result 2
└── Key Result 3
```

### 2.2 좋은 Objective(목표)의 특징

| 특징 | 설명 | 예시 |
|------|------|------|
| **영감을 주는** | 팀에게 동기부여 | "고객이 사랑하는 결제 경험 만들기" |
| **명확한** | 해석의 여지가 없음 | "모바일 앱 사용자 경험 혁신" |
| **야심찬** | 쉽게 달성 불가 | "업계 최고의 배달 속도 달성" |
| **시간 제한** | 분기 등 기한 명시 | "Q1 내에 ~" |

**좋은 목표 vs 나쁜 목표**

| ❌ 나쁜 목표 | ✅ 좋은 목표 |
|------------|------------|
| 매출 늘리기 | 신규 고객이 첫 주문을 완료하기까지의 여정을 매끄럽게 만든다 |
| 버그 줄이기 | 고객이 결제 중 겪는 불편을 완전히 제거한다 |
| 열심히 하기 | 데이터 기반 의사결정 문화를 정착시킨다 |

### 2.3 좋은 Key Result(핵심 결과)의 특징

**SMART 원칙 적용**

| 원칙 | 설명 | 예시 |
|------|------|------|
| **S**pecific | 구체적 | "결제 전환율" (X "개선") |
| **M**easurable | 측정 가능 | "35% → 42%" |
| **A**chievable | 달성 가능 (도전적이지만) | 현재 대비 20% 개선 |
| **R**elevant | Objective와 연관 | 목표 달성에 직접 기여 |
| **T**ime-bound | 기한 명시 | "Q1 말까지" |

**좋은 KR vs 나쁜 KR**

| ❌ 나쁜 KR | ✅ 좋은 KR |
|----------|----------|
| 전환율 개선 | 결제 전환율 35% → 42% 달성 |
| 사용자 만족도 향상 | NPS 점수 45 → 60 달성 |
| 성능 최적화 | 앱 로딩 시간 3초 → 1.5초 단축 |
| 기능 출시 | 간편결제 기능 출시 후 채택률 30% 달성 |

### 2.4 OKR 계층 구조 (Alignment)

```
회사 OKR
    │
    ├── 팀 OKR (회사 OKR에 연계)
    │       │
    │       └── 개인 OKR (팀 OKR에 연계)
    │
    └── 다른 팀 OKR
```

**정렬(Alignment) 예시**

```
[회사 Objective]
"2026년 Q1, 국내 음식배달 시장 1위 탈환"

    ↓ 연계

[제품팀 Objective]
"신규 사용자의 첫 주문 경험을 완벽하게 만든다"

    ↓ 연계

[결제 스쿼드 Objective]
"결제 과정에서의 이탈을 최소화한다"
    - KR1: 결제 단계 이탈률 32% → 18%
    - KR2: 간편결제 도입 후 채택률 40% 달성
    - KR3: 결제 오류율 0.5% 이하 유지
```

## 3. 지표의 3요소 (Value-Usage-Guardrail)

단일 지표에 매몰되지 않고, 제품의 건강 상태를 다각도로 측정한다.

### 3.1 지표 유형별 정의

| 지표 유형 | 목적 | 예시 |
|----------|------|------|
| **Value (가치)** | 비즈니스/사용자에게 전달하는 핵심 가치 | 전환율, 매출, NPS |
| **Usage (사용)** | 기능/제품이 얼마나 활용되는지 | DAU, 기능 채택률, 재방문율 |
| **Guardrail (가드레일)** | 성장 과정에서 희생되면 안 되는 것 | 앱 안정성, CS 발생률, 취소율 |

### 3.2 지표 설계 예시

```
[기능] 간편결제 도입

Value 지표 (성공 측정)
├── 결제 전환율 +5%p
└── 인당 평균 주문 금액 유지 또는 상승

Usage 지표 (활용 측정)
├── 간편결제 선택률 40%
└── 간편결제 재사용률 80%

Guardrail 지표 (부작용 방지)
├── 결제 오류율 < 0.5%
├── 결제 관련 CS 증가율 < 10%
└── 앱 크래시율 증가 없음
```

### 3.3 Guardrail이 중요한 이유

```
⚠️ 경고 사례

목표: 결제 전환율 +10%p
달성 방법: 공격적인 할인 쿠폰 노출

결과:
- 전환율 +12%p 달성 ✅
- 하지만 취소율 +25% ❌
- 인당 마진 -30% ❌

→ Guardrail이 없었다면 "성공"으로 잘못 판단
```

## 4. 지표 위계 (Metric Hierarchy)

### 4.1 북극성 지표 (North Star Metric)

조직 전체가 집중해야 하는 단 하나의 핵심 지표

**특징**
- 비즈니스 성장과 직결
- 모든 팀의 노력이 기여
- 장기적으로 추적 가능

**예시**
- Netflix: 총 시청 시간
- Airbnb: 예약된 숙박 일수
- Slack: 주간 활성 사용자의 메시지 수

### 4.2 지표 위계 구조

```
                    ┌─────────────────┐
                    │  북극성 지표     │
                    │ (North Star)    │
                    └────────┬────────┘
                             │
         ┌───────────────────┼───────────────────┐
         ▼                   ▼                   ▼
    ┌─────────┐        ┌─────────┐        ┌─────────┐
    │ 획득    │        │ 활성화  │        │ 수익화  │
    │ (Acq)  │        │ (Act)   │        │ (Rev)   │
    └────┬────┘        └────┬────┘        └────┬────┘
         │                  │                  │
    ┌────┴────┐        ┌────┴────┐        ┌────┴────┐
    │ 가입률  │        │ 첫주문율│        │ 전환율  │
    │ CPI    │        │ 재방문율│        │ ARPU   │
    └─────────┘        └─────────┘        └─────────┘
```

### 4.3 지표 간 인과관계 맵핑

```
[목표] 월 매출 100억 달성

매출 = 주문 수 × 객단가

주문 수 = DAU × 전환율 × 주문 빈도

DAU = 신규 가입 + 기존 재방문

전환율 = 상품 조회율 × 장바구니 담기율 × 결제 완료율

→ 각 레벨에서 가장 개선 여지가 큰 지표에 집중
```

## 5. OKR/KPI 운영 프로세스

### 5.1 분기별 사이클

```
[분기 시작 - 2주 전]
├── 회사 OKR 확정
├── 팀 OKR 초안 작성
└── 이해관계자 리뷰

[분기 시작]
├── 팀 OKR 확정
├── 전체 공유
└── 실행 시작

[분기 중 - 매주]
├── 주간 Check-in (진행 상황 점검)
├── 필요시 이니셔티브 조정
└── 장애물 해결

[분기 말]
├── OKR 스코어링 (0.0 ~ 1.0)
├── 회고 및 Lessons Learned
└── 다음 분기 OKR 준비
```

### 5.2 주간 Check-in 템플릿

```
📊 주간 OKR Check-in

Objective: [목표명]
현재 Confidence: 🟢 On Track / 🟡 At Risk / 🔴 Off Track

KR1: [핵심 결과 1]
- 목표: 42%
- 현재: 38%
- 진행률: 65%
- 이번 주 액션: 간편결제 A/B 테스트 시작

KR2: [핵심 결과 2]
- 목표: 30%
- 현재: 22%
- 진행률: 50%
- 이번 주 액션: 온보딩 플로우 개선 배포

🚧 장애물:
- PG사 연동 지연으로 테스트 일정 영향

📌 다음 주 계획:
- A/B 테스트 결과 분석 및 의사결정
```

### 5.3 스코어링 가이드

| 점수 | 의미 | 판단 기준 |
|------|------|----------|
| 1.0 | Extraordinary | 목표를 크게 초과 달성 |
| 0.7 ~ 0.9 | 성공 | 도전적 목표 대비 양호한 달성 |
| 0.4 ~ 0.6 | 부분 성공 | 진전은 있으나 목표에 미달 |
| 0.0 ~ 0.3 | 미달 | 유의미한 진전 없음 |

> **Note**: OKR은 100% 달성이 목표가 아님. 70% 달성도 성공으로 간주.  
> 매번 100% 달성된다면 목표가 충분히 도전적이지 않은 것.

## 6. 실무 예시

### 6.1 제품팀 OKR 예시

```
[Objective]
신규 사용자가 첫 주문까지 도달하는 경험을 완벽하게 만든다

[Key Results]
KR1: 가입 후 7일 내 첫 주문 전환율 25% → 35%
KR2: 신규 사용자의 앱 평균 체류 시간 3분 → 5분
KR3: 첫 주문 완료 NPS 점수 45 → 60

[Guardrail]
- 신규 가입자 이탈률 기존 수준 유지
- 첫 주문 취소율 < 5%

## 7. 피해야 할 안티패턴

### ❌ 안티패턴 1: Task를 KR로 설정

```
나쁜 예: "간편결제 기능 출시"
문제: 출시 자체는 결과가 아닌 활동

좋은 예: "간편결제 출시 후 채택률 30% 달성"
→ 기능이 실제로 가치를 전달하는지 측정
```

### ❌ 안티패턴 2: 측정 불가능한 KR

```
나쁜 예: "사용자 경험 개선"
문제: 무엇이 개선인지 판단 불가

좋은 예: "주문 완료까지 평균 클릭 수 8회 → 5회"
→ 명확한 수치로 개선 여부 판단
```

### ❌ 안티패턴 3: 단일 지표 맹신

```
나쁜 예: 전환율만 보고 성공 판단
문제: 취소율, CS 증가 등 부작용 간과

좋은 예: Value + Usage + Guardrail 함께 확인
→ 전체 그림을 보고 판단
```

### ❌ 안티패턴 4: 너무 많은 OKR

```
나쁜 예: 팀당 OKR 10개
문제: 집중력 분산, 리소스 파편화

좋은 예: 팀당 Objective 2-3개, KR 3-5개
→ 선택과 집중
```

## 8. 체크리스트

### OKR 작성 시
- [ ] Objective가 영감을 주고 명확한가?
- [ ] KR이 측정 가능하고 기한이 있는가?
- [ ] 상위 OKR과 정렬되어 있는가?
- [ ] 도전적이되 달성 불가능하지 않은가?
- [ ] Task가 아닌 Outcome인가?

### KPI 설정 시
- [ ] Value / Usage / Guardrail 구분했는가?
- [ ] 지표 간 인과관계가 명확한가?
- [ ] 실시간 모니터링이 가능한가?
- [ ] 팀이 직접 영향을 줄 수 있는 지표인가?

### 운영 시
- [ ] 주간 Check-in 진행 중인가?
- [ ] 장애물을 조기에 파악하고 해결하는가?
- [ ] 분기 말 스코어링과 회고를 진행하는가?

- 승인일: [날짜]
```

### 9.2 KPI 설계 워크시트

```markdown
# [기능/프로젝트명] KPI 설계서

## 1. 기본 정보
- 기능/프로젝트명: [이름]
- 해결하려는 문제: [1문장]
- 목표 사용자: [타겟 정의]
- 출시 예정일: [날짜]

## 2. 성공 지표 정의

### Value Metric (핵심 가치)
"이 기능이 성공했다"를 판단하는 핵심 지표

| 지표 | 정의 | 현재 | 목표 | 측정 주기 |
|------|------|------|------|----------|
| [지표명] | [계산식/정의] | [N] | [N] | 일간/주간/월간 |

### Usage Metric (사용 지표)
기능이 얼마나 활용되는지 측정

| 지표 | 정의 | 목표 | 측정 주기 |
|------|------|------|----------|
| 채택률 | 기능 노출 유저 중 사용 비율 | [N]% | 주간 |
| 재사용률 | 사용자 중 재사용 비율 | [N]% | 주간 |

### Guardrail Metric (가드레일)
이 기능으로 인해 악화되면 안 되는 지표

| 지표 | 허용 기준 | 현재 | 모니터링 담당 |
|------|----------|------|--------------|
| [지표명] | < [기준값] | [N] | [담당자] |

## 3. 측정 계획
- 데이터 소스: [GA, 내부 로그, 등]
- 대시보드 링크: [URL]
- 리포팅 주기: [주간/월간]
- 의사결정 시점: [출시 후 N주]

## 4. 실험 설계 (해당 시)
- A/B 테스트 여부: [예/아니오]
- 실험 기간: [N주]
- 최소 표본 크기: [N명]
- 통계적 유의 수준: 95%
```

### 예시 1: 배달 플랫폼 제품팀 OKR

```markdown
# 제품팀 OKR - 2026 Q1

## 배경
- 회사 Objective: "국내 배달 시장 1위 탈환"
- 핵심 과제: 신규 고객 전환율 개선 및 첫 주문 경험 강화

---

## Objective 1: 신규 고객이 첫 주문을 "쉽고 빠르게" 완료하게 만든다

### Key Results

| KR | 지표 | 현재 | 목표 | 측정 방법 |
|----|------|------|------|----------|
| KR1 | 가입 후 24시간 내 첫 주문 전환율 | 18% | 28% | 앱 로그 분석 |
| KR2 | 첫 주문까지 평균 소요 시간 | 15분 | 8분 | 퍼널 분석 |
| KR3 | 첫 주문 고객 NPS | 42 | 55 | 인앱 설문 |

### Guardrail 지표
- 첫 주문 취소율: < 8%
- 앱 크래시율: 현재 수준(0.3%) 유지
- CS 문의율: < 5%

### 주요 이니셔티브
| 이니셔티브 | 담당 | 예상 기여 | 일정 |
|-----------|------|----------|------|
| 신규 가입 온보딩 플로우 간소화 | UX팀 | KR2에 -3분 | 1월 |
| 첫 주문 전용 프로모션 노출 최적화 | 그로스팀 | KR1에 +5%p | 2월 |
| 간편결제 원클릭 도입 | 결제팀 | KR2에 -2분 | 2월 |
```

### 예시 2: 라이더 운영팀 KPI

```markdown
# 라이더 운영 KPI 대시보드

## 북극성 지표
- **라이더 유효 가동 시간당 배달 완료 건수**: 3.2건 → 3.8건

## 핵심 KPI

### 공급 효율 (Value)
| 지표 | 현재 | 목표 | 트렌드 |
|------|------|------|-------|
| 시간당 배달 완료 건수 | 3.2건 | 3.8건 | ↑ |
| 콜 수락률 | 72% | 85% | ↑ |
| 평균 배달 시간 | 34분 | 28분 | ↓ |

### 운영 건강성 (Usage)
| 지표 | 현재 | 목표 | 트렌드 |
|------|------|------|-------|
| 일 평균 활성 라이더 수 | 12,000명 | 15,000명 | ↑ |
| 피크 시간 라이더 가동률 | 68% | 80% | ↑ |
| 신규 라이더 7일 리텐션 | 45% | 60% | ↑ |

### 품질 (Guardrail)
| 지표 | 기준 | 현재 | 상태 |
|------|------|------|------|
| 배달 사고율 | < 0.01% | 0.008% | 🟢 |
| 고객 클레임 발생률 | < 2% | 1.8% | 🟢 |
| 라이더 이탈률 (월간) | < 15% | 18% | 🔴 |
```

## 11. OKR 품질 검토 가이드

### AI가 OKR 초안 작성 시 자문 체크리스트

**Objective 검토**
- [ ] 영감을 주는가? (팀이 읽고 동기부여 되는가?)
- [ ] 명확한가? (해석의 여지 없이 이해되는가?)
- [ ] 솔루션이 아닌 결과인가? ("기능 출시"가 아닌가?)
- [ ] 상위 OKR과 연결되는가?
- [ ] 분기 내 유의미한 진전이 가능한가?

**Key Result 검토**
- [ ] 숫자로 측정 가능한가?
- [ ] 현재 값(baseline)이 있는가?
- [ ] 목표 값이 도전적이되 불가능하지 않은가?
- [ ] 팀이 직접 영향을 줄 수 있는가? (외부 의존성 최소화)
- [ ] Task가 아닌 Outcome인가?

**균형 검토**
- [ ] Value / Usage / Guardrail이 균형 있는가?
- [ ] 한 지표 개선이 다른 지표를 악화시킬 위험은 없는가?
- [ ] KR 개수가 적절한가? (Objective당 2-5개)

**실행 가능성 검토**
- [ ] 데이터 측정이 가능한가?
- [ ] 담당자가 명확한가?
- [ ] 이니셔티브가 KR 달성에 직접 기여하는가?
