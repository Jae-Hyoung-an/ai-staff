# [Best Practice] 좋은 우선순위 결정

> **핵심 메시지**: 우선순위 결정의 본질은 "무엇을 할 것인가"보다 **"무엇을 하지 않을 것인가"**를 정하는 것이다.  
> 제한된 리소스로 최대의 가치를 창출하기 위한 전략적 선택이다.

---

## 핵심 요약

모든 요청은 중요해 보인다. 하지만 모든 것을 할 수는 없다. 우선순위 결정은 **비즈니스 임팩트, 사용자 가치, 구현 비용**을 종합적으로 고려하여 **가장 큰 가치를 가장 적은 리소스로 달성할 수 있는 것**을 선택하는 과정이다. 감정이 아닌 데이터와 프레임워크에 기반해야 한다.

---

## 1. 우선순위 결정의 원칙

### 1.1 핵심 질문

```
1. 이것을 하면 비즈니스에 얼마나 기여하는가? (Impact)
2. 이것을 하지 않으면 무슨 일이 일어나는가? (Risk)
3. 이것을 하는 데 얼마나 걸리는가? (Effort)
4. 지금 해야 하는가, 나중에 해도 되는가? (Urgency)
```

### 1.2 우선순위 레벨 정의

| 레벨 | 정의 | 기준 |
|------|------|------|
| **P0** | Critical | 안 하면 서비스 장애/법적 문제 |
| **P1** | Must Have | 이번 릴리즈에 반드시 필요 |
| **P2** | Should Have | 중요하지만 없어도 출시 가능 |
| **P3** | Nice to Have | 있으면 좋지만 후순위 |
| **P4** | Later | 언젠가 검토 (백로그 보관) |

## 2. 우선순위 프레임워크

### 2.1 RICE 스코어

```
RICE = (Reach × Impact × Confidence) / Effort
```

| 요소 | 설명 | 측정 방법 |
|------|------|----------|
| **R**each | 영향받는 사용자 수 | 분기당 예상 도달 사용자 수 |
| **I**mpact | 개별 사용자에 대한 영향 | 0.25(최소) ~ 3(최대) |
| **C**onfidence | 추정의 확신도 | 100%(높음) ~ 50%(낮음) |
| **E**ffort | 필요 리소스 | 인월(person-months) |

**RICE 계산 예시**

```
[기능 A: 간편결제 도입]
Reach: 50,000명/분기
Impact: 2 (높음)
Confidence: 80%
Effort: 3 인월

RICE = (50,000 × 2 × 0.8) / 3 = 26,667

[기능 B: 다크모드]
Reach: 100,000명/분기
Impact: 0.5 (낮음)
Confidence: 100%
Effort: 1 인월

RICE = (100,000 × 0.5 × 1.0) / 1 = 50,000

→ RICE 스코어만 보면 B가 높지만,
  비즈니스 전략상 A가 더 중요할 수 있음
  (프레임워크는 참고용)
```

### 2.2 Impact-Effort Matrix

```
                     Effort 낮음
                          │
    ┌─────────────────────┼─────────────────────┐
    │                     │                     │
    │   [Big Bets]        │   [Quick Wins]      │
    │   신중하게 검토     │   즉시 실행         │
Impact│   ROI 높지만       │   빠른 성과         │
높음  │   리소스 큼        │                     │
    ├─────────────────────┼─────────────────────┤
    │                     │                     │
    │   [Money Pit]       │   [Fill-ins]        │
    │   회피              │   여유 있을 때      │
Impact│   투자 대비         │                     │
낮음  │   효과 낮음        │                     │
    └─────────────────────┼─────────────────────┘
                          │
                     Effort 높음
```

### 2.3 MoSCoW Method

| 분류 | 의미 | 비율 가이드 |
|------|------|------------|
| **M**ust Have | 없으면 출시 불가 | ~60% |
| **S**hould Have | 중요하지만 필수 아님 | ~20% |
| **C**ould Have | 있으면 좋음 | ~10% |
| **W**on't Have | 이번엔 안 함 | ~10% |

### 2.4 Kano Model

```
만족도
  ↑
  │         ╱ Delighter (감동)
  │        ╱
  │───────●─────── Performance (기대)
  │      ╱│
  │     ╱ │
  │    ╱  │
  ─────●──┼─────────────→ 기능 충족도
       │  │
       │  │
       │ Basic (기본)
       │
```

| 유형 | 특징 | 예시 |
|------|------|------|
| **Basic** | 없으면 불만, 있어도 당연시 | 결제 기능, 로그인 |
| **Performance** | 좋을수록 만족 | 배달 속도, 앱 성능 |
| **Delighter** | 기대 안 했는데 있으면 감동 | 서프라이즈 혜택 |

## 3. 우선순위 결정 프로세스

### 3.1 전체 프로세스

```
1. 백로그 수집
   └── 모든 요청/아이디어 한 곳에 정리

2. 기준 정립
   └── 이번 분기 전략 목표 확인

3. 초기 분류
   └── P0~P4 대략적 분류

4. 상세 평가
   └── 프레임워크 적용 (RICE 등)

5. 이해관계자 논의
   └── 트레이드오프 합의

6. 최종 확정
   └── 스프린트/로드맵 반영

7. 지속 검토
   └── 상황 변화 시 재평가
```

### 3.2 주요 의사결정 기준

**비즈니스 전략 정렬**

```
[질문] 이번 분기 핵심 전략은?
→ "신규 사용자 전환율 개선"

[평가]
기능 A: 신규 사용자 전환율에 직접 영향 → 높은 우선순위
기능 B: 기존 사용자 편의 개선 → 다음 분기로
```

**기회 비용 고려**

```
[상황]
기능 A: 3인월, 예상 매출 +5%
기능 B: 1인월, 예상 매출 +2%
기능 C: 1인월, 예상 매출 +1.5%

[분석]
A만 하면: +5% (3인월)
B+C 하면: +3.5% (2인월)
B+C 후 A: 총 +8.5% (5인월)

→ B, C 먼저 진행 후 A 착수 (시간 가치 고려)
```

## 4. 실무 적용

### 4.1 백로그 관리

```
┌────────────────────────────────────────────────────────────┐
│                    📋 제품 백로그                           │
├──────┬──────────────┬────────┬────────┬────────┬──────────┤
│ ID   │ 항목          │ 우선순위│ RICE   │ 담당   │ 상태     │
├──────┼──────────────┼────────┼────────┼────────┼──────────┤
│ F-01 │ 간편결제 도입 │ P1     │ 26,667 │ 결제팀 │ 진행중   │
│ F-02 │ 검색 개선     │ P1     │ 23,000 │ 탐색팀 │ 대기     │
│ F-03 │ 리뷰 시스템   │ P2     │ 15,000 │ 탐색팀 │ 백로그   │
│ F-04 │ 다크모드      │ P3     │ 50,000 │ 공통   │ 백로그   │
│ B-01 │ 결제 오류 수정│ P0     │ -      │ 결제팀 │ 핫픽스   │
└──────┴──────────────┴────────┴────────┴────────┴──────────┘
```

### 4.2 우선순위 회의 진행

**회의 아젠다**

```
1. 지난 결정 리뷰 (5분)
   - 진행 상황, 변경 필요 사항

2. 신규 요청 리뷰 (15분)
   - 각 항목 배경, 임팩트 설명
   - 초기 우선순위 제안

3. 트레이드오프 논의 (20분)
   - 리소스 제약 확인
   - 충돌하는 우선순위 조율

4. 최종 결정 (10분)
   - 합의된 우선순위 확정
   - 다음 단계 액션 정리
```

### 4.3 요청 거절 커뮤니케이션

**좋은 거절 방법**

```
[상황]
마케팅팀: "다음 주 프로모션을 위해 새 배너 시스템이 필요해요"

[나쁜 응답]
"지금 개발 리소스가 없어서 안 돼요"

[좋은 응답]
"프로모션 성공을 위해 새 배너가 필요하시다는 점 이해합니다.

현재 우선순위 상황을 공유드리면:
- P0: 결제 오류 긴급 수정 (서비스 영향)
- P1: 간편결제 (이번 분기 핵심 목표)

새 배너 시스템은 3인주 소요 예상입니다.

대안으로 제안드립니다:
1) 기존 배너 시스템 활용 (즉시 가능)
2) 2주 후 착수하여 다음 프로모션에 적용
3) P1을 조정하려면 경영진 승인 필요

어떤 방향이 좋을까요?"
```

```
│ 요청일: [날짜]                                             │

│ [한 줄 요약]                                               │
├────────────────────────────────────────────────────────────┤
│ 해결하려는 문제:                                           │

├────────────────────────────────────────────────────────────┤
│ 예상 임팩트:                                               │
│ [정량적 기대 효과: 전환율 +X%, 비용 절감 Y원 등]            │
├────────────────────────────────────────────────────────────┤
│ 긴급도:                                                    │
│ [ ] 즉시 (서비스 장애/법적 문제)                           │

│ [ ] 다음 분기 이후                                         │
├────────────────────────────────────────────────────────────┤
│ 대안/현재 해결 방법:                                       │
│ [현재 어떻게 대응하고 있는지]                               │
├────────────────────────────────────────────────────────────┤
│ PO 코멘트:                                                 │
│ [검토 의견, 추가 질문]                                     │
└────────────────────────────────────────────────────────────┘
```

### 5.2 요청 vs 니즈 구분

```
[요청]
"결제 화면에 쿠폰 목록 보여주세요"

[진짜 니즈 파악]
Q: "왜 쿠폰 목록이 필요하세요?"
A: "사용자가 쿠폰이 있는지 몰라서 사용 안 해요"

[진짜 문제]
"사용자가 보유 쿠폰을 인지하지 못함"

[대안 솔루션]
1. 쿠폰 목록 추가 (요청대로)
2. 쿠폰 보유 시 배지 표시 (더 간단)
3. 자동 쿠폰 적용 (더 나은 UX)

→ 요청이 아닌 문제를 해결
```

## 6. MVP 범위 결정

### 6.1 MVP 정의

```
MVP (Minimum Viable Product)
= 핵심 가설을 검증하기 위한 최소한의 기능 세트

❌ MVP ≠ 미완성 제품
❌ MVP ≠ 품질 낮은 제품
✅ MVP = 학습을 위한 최소 범위
```

### 6.2 범위 결정 프로세스

```
1. 핵심 가설 명확화
   "간편결제를 도입하면 전환율이 5%p 이상 상승할 것이다"

2. 가설 검증에 필수적인 것만 선별
   ✅ 토스/카카오페이 연동 (핵심)
   ❌ 네이버페이 (추가 검증 시)
   ❌ 결제 수단 순서 커스터마이징 (편의)

3. 출시 → 학습 → 확장
   MVP 출시 → 데이터 확인 → 확장 결정
```

### 6.3 범위 협상 시나리오

```
[상황]
원래 요청: 5가지 결제 수단 모두 연동 (8주)
리소스: 4주 내 출시 필요

[협상]
"5가지 모두 연동하면 8주 소요됩니다.
4주 내 출시하려면 범위 조정이 필요합니다.

제안:
Phase 1 (4주): 토스, 카카오페이 (점유율 70%)
Phase 2 (4주): 나머지 3개

Phase 1으로 가설 검증 후 Phase 2 진행하면
리스크도 줄이고 학습도 할 수 있습니다.
어떠신가요?"
```

## 7. 피해야 할 안티패턴

### ❌ 안티패턴 1: HiPPO (Highest Paid Person's Opinion)

```
현상: "대표님이 이거 하래요" → 무조건 P1

문제: 
- 전략 정렬 없이 리소스 낭비
- 팀의 주도성 상실

해결:
- 배경과 임팩트 확인 후 판단
- 데이터 기반 대안 제시
- 트레이드오프 투명하게 공유
```

### ❌ 안티패턴 2: 급한 것 = 중요한 것

```
현상: 큰 소리로 요청하면 우선순위 상승

문제:
- 전략적으로 중요한 것이 뒤로 밀림
- 반응적(Reactive) 조직 문화

해결:
- 긴급도 ≠ 중요도 구분
- 정해진 기준으로 평가
- 긴급 요청 처리 프로세스 분리
```

### ❌ 안티패턴 3: 모든 것이 P1

```
현상: "이것도 중요, 저것도 중요"

문제:
- 진짜 P1이 무엇인지 불명확
- 리소스 분산, 아무것도 완료 안 됨

해결:
- P1은 동시에 2-3개로 제한
- 강제 순위화 (Stack Ranking)
- "이것을 위해 무엇을 포기할까?"
```

### ❌ 안티패턴 4: 분석 마비 (Analysis Paralysis)

```
현상: 완벽한 데이터가 있어야 결정

문제:
- 의사결정 지연
- 시장 기회 놓침

해결:
- 70% 확신에서 결정
- 작게 실험, 빠르게 학습
- "완벽한 결정 < 빠른 결정 + 빠른 수정"
```

## 8. 체크리스트

### 우선순위 결정 시
- [ ] 이번 분기 전략 목표와 정렬되는가?
- [ ] 비즈니스 임팩트가 명확한가?
- [ ] 구현 비용(시간, 인력)을 파악했는가?
- [ ] 기회 비용을 고려했는가?
- [ ] 이해관계자와 합의했는가?

### 요청 검토 시
- [ ] 해결하려는 문제가 명확한가?
- [ ] 요청이 아닌 니즈를 파악했는가?
- [ ] 대안은 없는가?
- [ ] 거절한다면 이유를 명확히 전달했는가?

### MVP 결정 시
- [ ] 검증하려는 가설이 명확한가?
- [ ] 가설 검증에 필수적인 것만 포함했는가?
- [ ] 학습 후 확장 계획이 있는가?

## 9. 프레임워크 선택 가이드

### 9.1 상황별 프레임워크 추천

| 상황 | 추천 프레임워크 | 이유 |
|------|----------------|------|
| 여러 기능 중 개발 순서 결정 | **RICE** | 정량적 비교, 객관적 근거 제시 |
| 릴리즈 범위 협의 (Must/Should) | **MoSCoW** | 이해관계자 합의에 용이 |
| 빠른 직관적 판단 필요 | **Impact-Effort Matrix** | 시각화, 빠른 의사결정 |
| 사용자 가치 중심 평가 | **Kano Model** | 사용자 만족도 관점 |
| 긴급 이슈 대응 | **P0-P4 레벨링** | 신속한 분류 |
| 기술 부채 vs 신규 기능 | **RICE + 리스크 가중치** | 장기적 관점 반영 |

### 9.2 프레임워크 혼용 전략

```markdown
[Phase 1: 초기 분류]
→ P0-P4 레벨링으로 긴급/필수 항목 분리

[Phase 2: 상세 평가]
→ P1-P2 항목에 RICE 적용하여 순서 결정

[Phase 3: 범위 합의]
→ MoSCoW로 이해관계자와 릴리즈 범위 확정

[Phase 4: 실행]
→ Impact-Effort Matrix로 스프린트 내 순서 결정
```

## 10. RICE 점수 산정 상세 가이드

### 10.1 Reach (도달) 산정 기준

**정의**: 일정 기간 내 이 기능의 영향을 받는 사용자/이벤트 수

| 수준 | 기준 | 예시 |
|------|------|------|
| 매우 높음 | 전체 MAU의 80% 이상 | 앱 홈 화면 개선 |
| 높음 | 전체 MAU의 50~80% | 검색 기능 개선 |
| 중간 | 전체 MAU의 20~50% | 결제 화면 개선 |
| 낮음 | 전체 MAU의 5~20% | 특정 카테고리 기능 |
| 매우 낮음 | 전체 MAU의 5% 미만 | 니치 기능 |

**산정 방법**
```markdown
Reach = [해당 기능을 경험할 사용자 수] × [기간 (분기)]

예시:
- 결제 화면 개선
- 월 결제 시도 사용자: 200,000명
- 분기 Reach: 200,000 × 3 = 600,000
```

### 10.2 Impact (영향) 산정 기준

**정의**: 개별 사용자에 대한 영향의 크기

| 점수 | 수준 | 기준 | 예시 |
|------|------|------|------|
| 3 | Massive | 핵심 지표 20%+ 개선 | 간편결제 도입 |
| 2 | High | 핵심 지표 10~20% 개선 | 결제 UX 개선 |
| 1 | Medium | 핵심 지표 5~10% 개선 | 로딩 속도 개선 |
| 0.5 | Low | 핵심 지표 2~5% 개선 | 마이너 UI 개선 |
| 0.25 | Minimal | 핵심 지표 2% 미만 | 부가 기능 |

**판단 기준**
```markdown
질문 1: 이 기능이 핵심 전환 퍼널에 직접 영향을 미치는가?
- 예 → 2~3점
- 아니오 → 0.5~1점

질문 2: 유사 기능/경쟁사 사례에서 측정된 효과가 있는가?
- 검증된 효과 있음 → 점수 유지
- 추정만 있음 → 점수 1단계 낮춤
```

### 10.3 Confidence (확신도) 산정 기준

**정의**: Reach, Impact 추정에 대한 확신 수준

| 점수 | 수준 | 근거 수준 |
|------|------|----------|
| 100% | 높은 확신 | 데이터 + 사용자 리서치 + 유사 사례 |
| 80% | 중간 확신 | 데이터 또는 리서치 (둘 중 하나) |
| 50% | 낮은 확신 | 직감, 추정만 있음 |

**확신도 판단 질문**
```markdown
- [ ] 관련 데이터 분석을 했는가?
- [ ] 사용자 인터뷰/리서치를 했는가?
- [ ] 경쟁사 또는 내부 유사 사례가 있는가?
- [ ] 실험(A/B 테스트)으로 검증한 적 있는가?

체크 개수:
- 3~4개: 100%
- 2개: 80%
- 1개: 50%
- 0개: 50% (리서치 먼저 권장)
```

### 10.4 Effort (노력) 산정 기준

**정의**: 구현에 필요한 리소스 (인월 단위)

| 규모 | 인월 | 특징 |
|------|------|------|
| XS | 0.5 | 1명 × 1~2주 |
| S | 1 | 1명 × 1개월 |
| M | 2 | 2명 × 1개월 |
| L | 4 | 2명 × 2개월 또는 4명 × 1개월 |
| XL | 8+ | 대규모 프로젝트 |

**산정 시 고려사항**
```markdown
- 기획/디자인 시간 포함
- QA/테스트 시간 포함
- 불확실성 버퍼 (+20~30%)
- 의존성 (외부 연동, 다른 팀 협업)
```

## 11. RICE 평가 워크시트

```markdown
# [프로젝트/기능명] 우선순위 평가

## 평가 대상 목록

| ID | 기능 | 설명 |
|----|------|------|
| A | 간편결제 도입 | 토스/카카오페이 연동 |
| B | 검색 자동완성 | 인기 검색어 기반 |
| C | 리뷰 정렬 개선 | 최신순/평점순 필터 |

---

## RICE 상세 평가

### A. 간편결제 도입

**Reach**
- 월 결제 시도 사용자: 200,000명
- 분기 Reach: **600,000**

**Impact**
- 결제 전환율 예상 개선: +15%
- 근거: 경쟁사 도입 후 13~18% 개선 사례
- 점수: **2 (High)**

**Confidence**
- [x] 데이터 분석 (현재 이탈 지점 파악)
- [x] 사용자 인터뷰 (결제 불편 사유 확인)
- [x] 경쟁사 사례 (배민, 쿠팡 간편결제 효과)
- 점수: **100%**

**Effort**
- 기획: 0.5인월
- 개발: 2인월
- QA: 0.5인월
- 총: **3인월**

**RICE Score**: (600,000 × 2 × 1.0) / 3 = **400,000**
```

## 12. 도메인 특화 예시: 배달/물류

### 예시 1: 배달 앱 백로그 RICE 평가

```markdown
## Q1 개발 백로그 우선순위 평가

### 평가 대상

| ID | 기능 | 요청자 |
|----|------|--------|
| F-01 | 간편결제 도입 | 사업팀 |
| F-02 | 배달 시간 예측 개선 | CS팀 |
| F-03 | 단골 가게 추천 | 마케팅팀 |
| F-04 | 라이더 실시간 위치 | CS팀 |

### RICE 평가 결과

| ID | Reach | Impact | Confidence | Effort | RICE | 순위 |
|----|-------|--------|------------|--------|------|------|
| F-01 | 600K | 2 | 100% | 3 | 400,000 | 1 |
| F-02 | 1M | 1 | 80% | 4 | 200,000 | 2 |
| F-04 | 500K | 1 | 100% | 2 | 250,000 | 3 |
| F-03 | 300K | 0.5 | 50% | 2 | 37,500 | 4 |

### 평가 근거

**F-01 간편결제 (1위)**
- 전환율 직접 영향 (핵심 퍼널)
- 경쟁사 대비 열세 상황
- 이번 분기 "전환율 개선" 전략과 정렬
- 결정: **P1 - 즉시 착수**

**F-02 배달 시간 예측 (2위)**
- CS 문의 30%가 배달 시간 관련
- 사용자 NPS 주요 불만 요인
- 결정: **P1 - 간편결제 후 착수**

**F-04 실시간 위치 (3위)**
- CS 문의 감소 기대
- 사용자 불안감 해소
- 결정: **P2 - 다음 분기 검토**

**F-03 단골 추천 (4위)**
- 효과 불확실 (Confidence 50%)
- MVP로 가설 검증 선행 필요
- 결정: **P3 - 리서치 후 재검토**
```

### 예시 2: 라이더 앱 MVP 범위 결정

```markdown
## 라이더 앱 리뉴얼 MVP 범위 결정

### 핵심 가설
"라이더 앱 UX 개선으로 콜 수락률이 15% 상승할 것이다"

### 기능 목록 분류 (MoSCoW)

**Must Have (필수)**
- 배달 정보 한눈에 보기 (거리, 배달료, 예상 시간)
- 콜 수락/거절 간소화 (원터치)
- 목적지 네비게이션 연동

**Should Have (중요)**
- 일별/주별 수익 대시보드
- 푸시 알림 커스터마이징

**Could Have (있으면 좋음)**
- 배달 경로 최적화 제안
- 라이더 커뮤니티

**Won't Have (이번 제외)**
- 실시간 채팅 기능
- 라이더 랭킹 시스템

### MVP 범위 합의
- Must Have만 MVP 포함 (4주)
- Should Have는 출시 후 1개월 내 추가
- MVP 출시 후 콜 수락률 측정하여 확장 결정
```

### 예시 3: 가맹점 기능 요청 우선순위

```markdown
## 가맹점 앱 기능 요청 처리

### 이번 달 접수된 요청

| 요청 | 빈도 | 긴급도 | 비즈니스 영향 |
|------|------|--------|--------------|
| 정산 내역 상세화 | 120건 | 중 | 이탈 방지 |
| 메뉴 일괄 수정 | 85건 | 하 | 운영 효율 |
| 실시간 주문 알림 | 50건 | 상 | 주문 누락 방지 |
| 프로모션 직접 등록 | 30건 | 하 | 자율성 |

### 우선순위 결정

**P1: 실시간 주문 알림**
- 빈도는 낮지만 주문 누락 시 매출 직접 영향
- 긴급도 높음, 경쟁사 기본 제공 기능
- 1인월 소요 예상

**P2: 정산 내역 상세화**
- VOC 빈도 가장 높음
- 가맹점 이탈 방지 효과
- 2인월 소요 예상

**P3: 메뉴 일괄 수정**
- 운영 편의 기능
- 다음 분기 검토
```
