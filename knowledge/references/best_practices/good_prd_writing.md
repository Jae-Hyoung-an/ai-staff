# 좋은 PRD 작성의 원칙

## 핵심 요약

좋은 PRD는 **비즈니스 로직과 정책의 명확성**과 **변경 이력 최신화**를 동시에 갖춘다.

> *"PRD는 '무엇을', '왜'를 정의한다. '어떻게'는 개발/디자인 문서의 영역이다."*

---

## PRD의 역할과 범위

### PRD가 다뤄야 할 것
- **비즈니스 로직**: 서비스가 어떤 규칙으로 동작해야 하는가
- **정책**: 각 상황에서 어떤 결과가 기대되는가
- **요구사항**: 사용자와 이해관계자가 기대하는 바
- **성공 기준**: 이 기능이 성공했는지 어떻게 판단할 것인가

### PRD가 다루지 않아야 할 것 (별도 문서로 분리)
| 내용 | 담당 문서 |
|------|----------|
| API 스펙, 데이터 스키마 | 개발 기술 문서 |
| UI 상호작용, 로딩/애니메이션 | 디자인 스펙 문서 |
| 구현 아키텍처, 기술 선택 | 개발 설계 문서 |
| 테스트 시나리오 상세 | QA 문서 |

### 유사 문서와의 구분

| 문서 | 초점 | 주요 독자 |
|------|------|----------|
| **MRD** (Market Requirements Document) | 시장 기회, 경쟁사 분석, 비즈니스 케이스 | 경영진, 사업팀 |
| **PRD** (Product Requirements Document) | 사용자 문제 해결, 기능 요구사항, 정책 | PM, 개발, 디자인, QA |
| **SRS** (Software Requirements Specification) | 상세 구현 명세, 기술 아키텍처 | 개발팀 |

## 1. 비즈니스 로직과 정책의 명확성

### 원칙
> 각 도메인 담당자가 "이 기능이 비즈니스적으로 어떻게 동작해야 하는가"를 명확히 이해할 수 있어야 한다.

### 왜 중요한가
- 개발자는 비즈니스 규칙을 스스로 판단해선 안 된다
- 정책이 불명확하면 구현마다 해석이 달라진다
- 나중에 "왜 이렇게 동작하지?"라는 질문에 답할 수 있어야 한다

### 도메인별로 명시해야 할 정책

| 관점 | 명시해야 할 내용 |
|------|-----------------|
| **사용자 정책** | 누가 이 기능을 사용할 수 있는가? 권한 조건은? |
| **주문/결제** | 주문 생성/취소/환불 조건, 금액 계산 규칙 |
| **재고/수량** | 재고 차감 시점, 품절 처리 정책 |
| **시간/기한** | 마감 시간, 유효 기간, 만료 처리 |
| **알림/커뮤니케이션** | 알림 발송 조건, 대상, 내용 |
| **예외 상황** | 실패/오류 시 비즈니스 처리 방향 |

### 좋은 정책 정의 vs 나쁜 정책 정의

| 나쁜 예시 ❌ | 좋은 예시 ✅ |
|-------------|-------------|
| "결제 실패 시 적절히 처리" | "결제 실패 시: 주문 상태를 '결제대기'로 유지, 30분 내 재결제 가능, 30분 초과 시 자동 취소" |
| "재고가 부족하면 알림" | "주문 시점에 재고 < 주문수량이면 주문 불가. 장바구니 담기는 허용하되, 결제 직전 재검증" |
| "적절한 시점에 푸시 발송" | "주문 완료 즉시 푸시 발송. 배송 시작 시 추가 발송. 하루 최대 5건 제한" |
| "VIP 사용자 우대" | "최근 3개월 주문금액 50만원 이상 → VIP 등급. 무료배송 + 5% 추가 할인 적용" |

### 정책 정의 체크리스트
- [ ] 각 조건(if)에 대한 결과(then)가 명확한가?
- [ ] 숫자/기준이 구체적인가? (적절히 → 3회, 빠르게 → 30분 이내)
- [ ] 예외 상황의 비즈니스 처리 방향이 정의되었는가?
- [ ] 경계값(boundary)이 명확한가? (이상/초과, 이하/미만)

## 2. 사용자 스토리와 페르소나

### 원칙
> 단순 정책 나열이 아닌, "누가, 왜 이 기능을 사용하는가"에 대한 맥락을 포함해야 한다.

### 사용자 스토리 형식

```
[사용자 유형]로서, [목적]을 위해 [기능]을 원한다.
```

**예시:**
- "바쁜 직장인으로서, 퇴근 후 빠르게 저녁을 해결하기 위해 예상 배달 시간을 미리 확인하고 싶다."
- "처음 주문하는 고객으로서, 안심하고 결제하기 위해 취소/환불 정책을 명확히 알고 싶다."

### 사용자 시나리오 구성

| 시나리오 유형 | 설명 | 필수 여부 |
|-------------|------|----------|
| **Happy Path** | 정상적인 사용 흐름 | ✅ 필수 |
| **Edge Case** | 경계 조건에서의 동작 | ✅ 필수 |
| **Error Flow** | 오류/실패 상황에서의 동작 | ✅ 필수 |
| **Alternative Path** | 대안 경로 (선택적 기능 사용) | 선택 |

## 3. 성공 지표의 계층화

### 원칙
> 단순한 KPI 나열이 아닌, 지표를 계층화하여 기능의 성공을 다각도로 측정한다.

### 지표 유형

| 지표 유형 | 목적 | 예시 |
|----------|------|------|
| **핵심 지표 (Core)** | 비즈니스 가치 창출 여부 | 결제 전환율 +5%, 객단가 +10% |
| **보조 지표 (Usage)** | 기능 활용도 측정 | 기능 사용률, DAU, 재사용률 |
| **가드레일 지표 (Guardrail)** | 기존 지표 훼손 방지 | 앱 크래시율 유지, CS 문의 증가 없음 |

### 지표 설정 예시

```markdown
## 성공 지표

### 핵심 지표
- 신규 기능을 통한 주문 전환율: 기존 대비 **+3%p**

### 보조 지표  
- 기능 노출 대비 클릭률: **15% 이상**
- 기능 사용 후 재사용률: **40% 이상**

### 가드레일 지표
- 앱 크래시율: 기존 수준 **유지** (0.1% 이하)
- 결제 관련 CS 문의: **증가 없음**
- 기존 결제 플로우 전환율: **하락 없음**
```

## 4. 우선순위 관리 (MoSCoW)

### 원칙
> 모든 요구사항을 동일하게 취급하지 않고, 명확한 우선순위로 개발 범위를 관리한다.

### MoSCoW 프레임워크

| 분류 | 의미 | 예시 |
|------|------|------|
| **Must have** | 없으면 출시 불가. MVP 필수 기능 | 결제 기능, 주문 취소 |
| **Should have** | 중요하지만 당장 없어도 운영 가능 | 주문 내역 필터링, 즐겨찾기 |
| **Could have** | 여유가 있으면 추가할 부가 기능 | 리뷰 이미지 확대, 애니메이션 |
| **Won't have** | 이번 버전에서 명확히 제외 | 소셜 공유, 다국어 지원 |

### 왜 중요한가
- **Scope Creep 방지**: "이것도 넣으면 좋겠다"는 요청에 명확한 기준 제시
- **일정 관리**: Must만 완료되면 최소 출시 가능
- **팀 합의**: 무엇을 안 할지 명확히 합의

## 5. 비기능적 요구사항

### 원칙
> 구현 방식은 개발 문서의 영역이지만, 비즈니스 관점에서 보장되어야 할 품질 기준은 PRD에 명시한다.

### 비기능적 요구사항 유형

| 영역 | PRD에 명시할 내용 | 예시 |
|------|------------------|------|
| **성능** | 사용자 경험 기준 | "결제 완료까지 3초 이내" |
| **가용성** | 서비스 운영 기준 | "월 99.9% 가용성" |
| **보안** | 데이터 보호 정책 | "개인정보 암호화 필수, 결제 정보 미저장" |
| **규제** | 법적 준수 사항 | "전자상거래법 준수, 7일 이내 환불 보장" |
| **접근성** | 사용자 접근성 | "스크린 리더 호환 필수" |

### 예시

```markdown
## 비기능적 요구사항

### 성능
- 상품 목록 로딩: 2초 이내
- 결제 처리 완료: 5초 이내

### 보안
- 결제 정보: PCI-DSS 준수
- 개인정보: 암호화 저장, 최소 수집 원칙

### 규제
- 미성년자 주류 구매 차단 (본인인증 필수)
- 통신판매업 신고 정보 노출
```

## 6. 변경 이력과 Living Document

### 원칙
> PRD는 배포 후 끝나는 문서가 아니라, 제품 생애주기 동안 지속 업데이트되는 '살아있는 문서'다.

### 변경 이력 관리 체계

```markdown
## 변경 이력

| 버전 | 날짜 | 변경 내용 | 변경 사유 | 담당자 |
|------|------|----------|----------|--------|
| v1.3 | 2026-01-20 | 자동 취소 시간 30분→1시간 변경 | CS 피드백: 재결제 시도 시간 부족 | @jaehyoung |
| v1.2 | 2026-01-18 | VIP 기준 30만원→50만원 상향 | 사업팀 요청: 수익성 검토 결과 | @biz-manager |
| v1.1 | 2026-01-15 | 하루 푸시 제한 3건→5건 변경 | 마케팅 요청: 캠페인 효과 증대 | @marketing |
| v1.0 | 2026-01-10 | 최초 작성 | - | @jaehyoung |
```

### 최신화 트리거

다음 상황에서는 반드시 PRD를 업데이트해야 한다:

1. **정책 변경**: 비즈니스 규칙/조건이 바뀐 경우
2. **요구사항 변경**: 이해관계자 요청으로 스펙이 달라진 경우
3. **운영 피드백**: CS/운영 이슈로 정책 수정이 필요한 경우
4. **외부 요인**: 법규 변경, 파트너사 정책 변경 등

### 사후 분석 기록 (Post-Release)

배포 후 성과를 PRD에 기록하여 학습 자산으로 활용:

```markdown
## 사후 분석 (2026-02-15 추가)

### 지표 달성 여부
| 지표 | 목표 | 실제 | 달성 |
|------|------|------|------|
| 전환율 | +3%p | +2.1%p | ❌ 미달 |
| 기능 사용률 | 15% | 22% | ✅ 초과 |
| CS 문의 | 유지 | +5% | ⚠️ 소폭 증가 |

### Learnings
- 전환율 미달 원인: 버튼 위치가 눈에 띄지 않음 → v2에서 UI 개선 예정
- CS 증가 원인: 정책 안내 부족 → FAQ 추가 완료
```

## 7. 개발 준비/완료 기준 (DoR & DoD)

### Definition of Ready (DoR) - 개발 시작 가능 기준

개발팀이 작업을 시작하기 전 PRD가 갖춰야 할 조건:

- [ ] 사용자 스토리와 시나리오가 정의되어 있다
- [ ] 비즈니스 정책이 구체적 숫자/조건으로 명시되어 있다
- [ ] 성공 지표(핵심/보조/가드레일)가 정의되어 있다
- [ ] 우선순위(MoSCoW)가 합의되어 있다
- [ ] 디자인 문서 링크가 연결되어 있다
- [ ] 이해관계자 리뷰가 완료되어 있다

### Definition of Done (DoD) - 기능 완료 기준

기능이 "완료"되었다고 판단하는 기준:

- [ ] 모든 Must have 요구사항이 구현되었다
- [ ] 정의된 Happy Path / Edge Case / Error Flow가 동작한다
- [ ] 가드레일 지표가 악화되지 않았다
- [ ] PRD의 정책과 실제 동작이 일치한다
- [ ] 변경 사항이 PRD에 반영되어 있다

## 8. PRD 구조 템플릿

### 권장 구조

```markdown
# [기능명] PRD

## 메타 정보
- 작성자: 
- 최종 수정일: 
- 버전: 
- 상태: [Draft / In Review / Approved / In Development / Released]

## 1. 개요
### 1.1 배경 및 문제 정의
### 1.2 목표
### 1.3 성공 지표 (핵심/보조/가드레일) ⭐

## 2. 사용자
### 2.1 대상 사용자 (페르소나)
### 2.2 사용자 스토리

## 3. 범위
### 3.1 In-Scope (MoSCoW 분류) ⭐
### 3.2 Out-of-Scope

## 4. 사용자 시나리오
### 4.1 Happy Path
### 4.2 Edge Cases
### 4.3 Error Flows

## 5. 비즈니스 정책 ⭐
### 5.1 핵심 비즈니스 규칙
### 5.2 권한 및 접근 정책
### 5.3 예외 처리 정책

## 6. 비기능적 요구사항
### 6.1 성능
### 6.2 보안/규제

## 7. 연관 문서
- 디자인 스펙: [링크]
- 개발 기술 문서: [링크]
- QA 테스트 케이스: [링크]

## 8. 일정 및 마일스톤

## 9. 변경 이력 ⭐

## 10. 사후 분석 (배포 후 작성)
```

## 9. 비즈니스 정책 작성 예시

### 예시: 주문 취소 정책

```markdown
## 5. 비즈니스 정책

### 5.1 주문 취소 정책

#### 취소 가능 조건
| 주문 상태 | 취소 가능 여부 | 환불 정책 |
|----------|---------------|----------|
| 결제 대기 | ✅ 가능 | 환불 없음 (미결제) |
| 결제 완료 | ✅ 가능 | 전액 즉시 환불 |
| 상품 준비중 | ✅ 가능 | 전액 환불 (1-3영업일) |
| 배송 시작 | ❌ 불가 | 반품 절차 안내 |
| 배송 완료 | ❌ 불가 | 반품 절차 안내 |

#### 자동 취소 규칙
- 결제 대기 상태에서 **1시간 경과** 시 자동 취소
- 자동 취소 **10분 전** 리마인드 알림 발송

#### 취소 제한
- 동일 상품 **3회 연속 취소** 시 해당 상품 7일간 주문 제한
- 월 **10회 이상 취소** 시 고객센터 연결 필요
```

## 10. 피해야 할 안티패턴

| 안티패턴 | 설명 | 결과 |
|----------|------|------|
| **모호한 정책** | "적절히 처리", "상황에 맞게" | 개발자마다 다르게 해석 |
| **숫자 없는 기준** | "빈번한 취소", "장기간 미접속" | 기준이 사람마다 다름 |
| **예외 미정의** | 정상 케이스만 기술 | 예외 상황 처리가 제각각 |
| **변경 미반영** | 구두 합의만 하고 문서 미수정 | PRD와 실제 동작 불일치 |
| **과도한 상세화** | How까지 PRD에서 정의 | 개발/디자인 자율성 저해, 유지보수 어려움 |
| **Scope Creep** | 개발 중 요구사항 계속 추가 | 일정 지연, 품질 저하 |
| **이해관계자 합의 누락** | 기획자 단독 확정 | 출시 직전 정책 번복 |
| **전문 용어 남발** | 기술/기획 용어만 사용 | 타 부서 이해 불가, 소통 단절 |
| **솔루션 중심 작성** | "왜"보다 "무엇을 만든다"에 집중 | 문제 해결이 아닌 기능 구현에 그침 |

---

## 11. 문서 간 역할 분리

```
┌─────────────────────────────────────────────────────────┐
│                        PRD                              │
│  "무엇을, 왜" - 비즈니스 로직, 정책, 요구사항           │
└─────────────────────────────────────────────────────────┘
                          │
          ┌───────────────┼───────────────┐
          ▼               ▼               ▼
┌─────────────────┐ ┌─────────────┐ ┌─────────────┐
│   디자인 문서    │ │  개발 문서   │ │   QA 문서   │
│ UI/UX 상세 스펙  │ │ API, 아키텍처│ │ 테스트 케이스│
│ 인터랙션, 플로우 │ │ 데이터 스키마 │ │ 시나리오 상세│
└─────────────────┘ └─────────────┘ └─────────────┘
```

### 문서 간 연결
- PRD에서 정책이 변경되면 → 관련 문서에 변경 필요 여부 공유
- 개발/디자인 문서 작성 시 → PRD의 정책을 기준으로 상세화
- PRD에는 각 문서의 링크를 "연관 문서" 섹션에 정리

---

## 12. PRD 작성 전 자문

1. 비즈니스 규칙과 정책이 **구체적인 숫자/조건**으로 정의되었는가?
2. **사용자 스토리**로 "누가, 왜" 사용하는지 설명되는가?
3. **성공 지표**가 핵심/보조/가드레일로 계층화되어 있는가?
4. **우선순위(MoSCoW)**가 명확히 정의되고 합의되었는가?
5. 예외/에러 상황의 **비즈니스 처리 방향**이 정의되었는가?
6. 구현 상세(API, UI 인터랙션)가 **별도 문서로 분리**되었는가?
7. **이해관계자 리뷰**가 완료되었는가?
8. **DoR 조건**을 충족하는가?

---

## 좋은 PRD 공식

```
좋은 PRD = 명확한 비즈니스 정책 + 계층화된 성공 지표 + 우선순위 합의 + 최신화된 변경 이력
```

| 요소 | 질문 |
|------|------|
| 비즈니스 정책 | 개발자가 비즈니스 판단 없이 정책을 구현할 수 있는가? |
| 성공 지표 | 기능의 성공/실패를 다각도로 판단할 수 있는가? |
| 우선순위 | 무엇을 먼저 하고, 무엇을 안 할지 합의되었는가? |
| 변경 이력 | 최신 문서인지 신뢰할 수 있는가? |

---

*"좋은 PRD는 비즈니스 의사결정을 문서화하고, 나쁜 PRD는 의사결정을 개발자에게 떠넘긴다."*
