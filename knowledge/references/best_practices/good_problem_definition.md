# 좋은 문제 정의의 원칙

## 핵심 요약

좋은 문제 정의는 **현상이 아닌 원인**에 집중하고, **측정 가능한 형태**로 표현된다.

> *"문제를 잘 정의하면 절반은 해결한 것이다."* — Charles Kettering

---

## 문제 정의가 중요한 이유

### 문제 정의 없이 시작하면

- 솔루션이 문제를 해결하지 못함 (솔루션 먼저 생각하는 오류)
- 팀마다 다른 문제를 풀고 있음 (정렬 실패)
- 성공/실패 판단 기준이 없음 (측정 불가)
- 리소스 낭비 후 "왜 만들었지?" 질문

### 문제 정의가 명확하면

- 솔루션 방향이 자연스럽게 도출됨
- 팀 전체가 같은 목표를 향해 정렬됨
- 성공 기준이 명확해짐
- 불필요한 기능 추가를 방지함

## 1. 문제 vs 현상 vs 솔루션 구분

### 정의

| 구분 | 설명 | 예시 |
|------|------|------|
| **현상 (Symptom)** | 관찰되는 결과, 표면적 증상 | "결제 전환율이 낮다" |
| **문제 (Problem)** | 현상의 근본 원인, 해결해야 할 것 | "결제 단계가 복잡해서 이탈이 발생한다" |
| **솔루션 (Solution)** | 문제를 해결하는 방법 | "간편결제 도입" |

### 흔한 실수: 현상을 문제로 착각

| 현상 ❌ | 문제 ✅ |
|---------|---------|
| "MAU가 감소하고 있다" | "신규 가입 후 7일 내 이탈률이 60%로, 온보딩 과정에서 핵심 가치를 경험하지 못한다" |
| "고객 문의가 많다" | "배송 상태 확인이 어려워 고객이 직접 문의해야 한다" |
| "매출이 줄었다" | "재구매율이 20%에서 12%로 하락, 첫 구매 후 재방문 유인이 부족하다" |

### 흔한 실수: 솔루션을 문제로 포장

| 솔루션 포장 ❌ | 문제 ✅ |
|---------------|---------|
| "푸시 알림 기능이 필요하다" | "사용자가 중요한 업데이트를 놓쳐 기회를 잃는다" |
| "검색 필터를 추가해야 한다" | "원하는 상품을 찾는 데 평균 5분 이상 소요된다" |
| "다크모드를 지원해야 한다" | "야간 사용 시 눈의 피로로 이용 시간이 단축된다" |

## 2. 문제 정의 프레임워크

### 2.1 5 Whys (도요타 방식)

현상에서 출발해 "왜?"를 반복하여 근본 원인에 도달한다.

**예시: 결제 전환율 하락**

```
현상: 결제 전환율이 68%에서 62%로 하락
↓ Why?
1차: 결제 페이지에서 이탈이 증가했다
↓ Why?
2차: 결제 정보 입력에 시간이 오래 걸린다
↓ Why?
3차: 카드 정보를 매번 새로 입력해야 한다
↓ Why?
4차: 카드 정보 저장 기능이 없다
↓ Why?
5차: 보안 우려로 기능 개발을 미뤄왔다

→ 근본 문제: 보안과 편의성의 균형을 맞춘 카드 정보 저장 방식이 없다
```

**5 Whys 활용 팁**
- 5번이 정답이 아님. 3번이면 충분할 수도, 7번이 필요할 수도 있음
- "왜?"에 대한 답이 여러 개면 각각 분기하여 탐색
- 사람/프로세스 탓으로 끝나면 시스템적 원인을 더 파고들기

---

### 2.2 Problem Tree (문제 나무)

문제를 원인(roots)과 결과(branches)로 구조화한다.

```
                    [결과/현상]
                 고객 이탈률 증가
                        │
        ┌───────────────┼───────────────┐
        │               │               │
   [직접 원인]     [직접 원인]     [직접 원인]
   서비스 불만족   경쟁사 이동     가격 민감도
        │               │               │
   ┌────┴────┐     ┌────┴────┐     ┌────┴────┐
 [근본원인]  [근본원인]  ...        ...        ...
 느린 응답   기능 부족
```

**활용 시점**
- 복잡한 문제의 구조를 파악할 때
- 여러 원인이 얽혀 있을 때
- 팀과 함께 문제를 분석할 때

---

### 2.3 Jobs-to-be-Done (JTBD)

사용자가 "해결하려는 작업(Job)"에 집중한다.

**JTBD 문장 구조**
```
[상황]에서 [사용자]는 [동기/목적]를 위해 [해결하려는 작업]을 하고 싶다.
```

**예시**
- "바쁜 아침에 직장인은 시간을 절약하기 위해 빠르게 아침 식사를 해결하고 싶다"
- "처음 방문한 사이트에서 신규 고객은 안심하고 결제하기 위해 환불 정책을 확인하고 싶다"

**JTBD가 유용한 이유**
- 솔루션이 아닌 사용자의 근본적 니즈에 집중
- 경쟁자를 다르게 정의하게 됨 (밀크셰이크 vs 바나나 vs 베이글)
- "왜 이 기능을 원하는가"의 답을 제공

---

### 2.4 Problem Statement Canvas

문제를 구조화된 형식으로 정리한다.

| 항목 | 질문 | 예시 |
|------|------|------|
| **Who** | 누가 이 문제를 겪는가? | 첫 구매 고객 (신규 가입 7일 이내) |
| **What** | 무엇이 문제인가? | 결제 과정에서 이탈 |
| **Where** | 어디서 발생하는가? | 결제 정보 입력 페이지 |
| **When** | 언제 발생하는가? | 카드 정보 입력 단계 |
| **Why** | 왜 문제인가? (비즈니스 임팩트) | 전환율 하락 → 월 매출 X억 손실 |
| **How much** | 얼마나 심각한가? | 전환율 68%→62%, 일 Y건 이탈 |

## 3. 좋은 문제 정의 기준

### 3.1 측정 가능성 (Measurable)

| 측정 불가 ❌ | 측정 가능 ✅ |
|-------------|-------------|
| "사용자 경험이 나쁘다" | "상품 상세 페이지 로딩 시간이 평균 4.2초로, 업계 기준(2초) 대비 느리다" |
| "만족도가 낮다" | "NPS 점수가 -15로, 비추천 비율(45%)이 추천(30%)보다 높다" |
| "검색이 불편하다" | "검색 결과 첫 페이지에서 원하는 상품을 찾는 비율이 35%에 불과하다" |

### 3.2 구체성 (Specific)

| 추상적 ❌ | 구체적 ✅ |
|----------|----------|
| "고객이 이탈한다" | "결제 페이지에서 카드 정보 입력 시 32%가 이탈한다" |
| "앱이 느리다" | "홈 화면 최초 로딩에 3.5초 소요, 3초 초과 시 이탈률 40% 증가" |
| "리텐션이 낮다" | "첫 구매 후 30일 내 재구매율이 12%로 목표(25%) 대비 미달" |

### 3.3 행동 유발성 (Actionable)

문제 정의가 솔루션 방향을 시사해야 한다.

| 행동 불명확 ❌ | 행동 명확 ✅ |
|--------------|-------------|
| "경쟁이 치열하다" | "경쟁사 대비 배송 소요 시간이 1.5일 길어 고객이 이탈한다" |
| "시장이 어렵다" | "신규 고객 획득 비용(CAC)이 LTV의 80%로 수익성이 낮다" |

### 3.4 비즈니스 임팩트 연결

문제가 비즈니스에 미치는 영향이 명확해야 우선순위를 판단할 수 있다.

**임팩트 연결 형식**
```
[문제]로 인해 [현상]이 발생하고, 이는 [비즈니스 지표]에 [수치적 영향]을 미친다.
```

**예시**
> "결제 정보 입력 단계의 복잡함으로 인해 32%의 이탈이 발생하고, 이는 월 매출 2억 원 손실에 해당한다."

### 3.5 지표의 3요소 (Value-Usage-Guardrail)

문제 정의 시 성공을 단일 지표로만 보지 않고, 세 가지 관점에서 정의한다.

| 지표 유형 | 목적 | 예시 |
|----------|------|------|
| **핵심 지표 (Value)** | 문제 해결 시 창출되는 본질적 가치 | 결제 전환율 +5%p |
| **보조 지표 (Usage)** | 기능/솔루션의 활용도 측정 | 간편결제 선택률 40% |
| **가드레일 지표 (Guardrail)** | 문제 해결 과정에서 악화되면 안 되는 지표 | 결제 오류율 유지, 앱 로딩 속도 유지 |

**가드레일 지표가 중요한 이유**
- 전환율은 올랐는데 취소율도 폭등하는 상황 방지
- 속도 개선했는데 안정성이 떨어지는 상황 방지
- 단기 성과에 매몰되어 장기 지표를 희생하는 것 방지

## 4. 가설 수립

### 가설이란

문제에 대한 잠정적 답변으로, 검증을 통해 참/거짓을 판단할 수 있어야 한다.

### 가설 구조화 (변경-대상-결과)

```
만약 [변경 사항]을 하면, [대상 사용자]의 [기대 결과/지표]가 [방향 및 크기]로 변할 것이다.
```

**좋은 가설 예시**
- "만약 간편결제를 도입하면, 신규 고객의 결제 전환율이 7%p 상승할 것이다"
- "만약 배송 추적 알림을 제공하면, 배송 관련 CS 문의가 30% 감소할 것이다"
- "만약 온보딩 튜토리얼을 3단계로 줄이면, 신규 가입자의 7일 리텐션이 20% 개선될 것이다"

### 가설 품질 체크리스트

- [ ] 검증 가능한가? (실험/데이터로 확인 가능)
- [ ] 반증 가능한가? (틀릴 수 있는 형태)
- [ ] 구체적인가? (변경, 대상, 결과가 명확)
- [ ] 측정 가능한가? (성공/실패 기준이 수치로 표현)

### 가설 우선순위

| 기준 | 설명 |
|------|------|
| **임팩트** | 성공 시 비즈니스 가치가 큰가? |
| **확신도** | 성공할 것이라는 근거가 있는가? |
| **검증 비용** | 실험/검증에 필요한 리소스는? |

### 사전 학습 정의 (Pre-mortem Lessons)

가설 수립 시 "맞을 때"와 "틀릴 때" 모두에 대한 다음 행동을 미리 정의한다.

```
가설: 간편결제를 도입하면 결제 전환율이 7%p 상승할 것이다.

✅ 가설이 맞을 경우 (Next Step)
→ "고객이 결제 편의성에 민감하다는 것이 검증됨. 
   결제 수단 다양화(토스, 카카오페이 등)로 확장한다."

❌ 가설이 틀릴 경우 (Pivot)
→ "결제 편의성이 이탈의 주원인이 아닐 수 있음.
   결제 전 단계(상품 탐색, 장바구니)의 이탈 원인을 분석한다."
```

**왜 중요한가**
- 실험 결과에 감정적으로 반응하지 않고 계획된 다음 행동 실행
- "실패"도 학습으로 전환
- 팀의 기대치 정렬

## 5. 문제-가설-솔루션 연결

### 연결 흐름

```
[현상 관찰] → [문제 정의] → [가설 수립] → [솔루션 설계] → [검증]
                                ↑                              │
                                └──────────(학습 반영)─────────┘
```

### 연결 예시

| 단계 | 내용 |
|------|------|
| **현상** | 첫 구매 후 재구매율이 12%로 낮다 |
| **문제** | 첫 구매 경험이 기대에 미치지 못해 재방문 동기가 없다 |
| **가설** | 첫 구매 시 다음 구매 할인 쿠폰을 제공하면 재구매율이 20%로 상승할 것이다 |
| **솔루션** | 첫 구매 완료 시 10% 할인 쿠폰 자동 발급 |
| **검증** | A/B 테스트로 쿠폰 제공군의 30일 내 재구매율 측정 |

## 6. 유저 문제와 비즈니스 문제 연결

### 두 관점의 균형

| 관점 | 질문 | 위험 |
|------|------|------|
| **유저 문제** | 사용자가 겪는 불편/고통은? | 비즈니스 가치 연결 부족 |
| **비즈니스 문제** | 매출/비용/성장에 미치는 영향은? | 사용자 니즈 무시 |

### 연결 프레임워크

```
[유저의 고통] ──causes──→ [유저의 행동 변화] ──leads to──→ [비즈니스 영향]
```

**예시**
```
배송 상태를 알 수 없다 → 고객이 CS에 문의한다 → CS 비용 증가 + 고객 만족도 하락
         │
         └→ 재구매를 꺼린다 → 리텐션 감소 → LTV 하락
```

### 연결 질문

1. 이 유저 문제가 해결되면 사용자 행동이 어떻게 바뀌는가?
2. 그 행동 변화가 어떤 비즈니스 지표에 영향을 미치는가?
3. 그 영향의 크기는 얼마인가?

## 7. 서비스 성숙도별 접근법

서비스의 단계에 따라 문제 정의의 데이터 소스와 접근 방식이 달라진다.

| 구분 | 접근 방식 | 주요 데이터 소스 |
|------|----------|-----------------|
| **신규 프로젝트** | 유저 시나리오, 가상 페르소나 기반 | 인터뷰, 유저 저니 맵, FGI, 타사 벤치마킹 |
| **운영 중 서비스** | 정량적 지표 분석, 유저 행동 로그 기반 | 퍼널 이탈률, 검색어 빈도, CTR, CS 분석 |

**신규 프로젝트의 경우**
- 데이터가 없으므로 가설의 불확실성이 높음
- 빠른 MVP/프로토타입으로 가설 검증 우선
- "우리가 모르는 것"을 명확히 정의

**운영 중 서비스의 경우**
- 기존 데이터에서 문제의 크기와 원인 파악 가능
- 정량 데이터 + 정성 피드백을 결합
- 실험을 통한 인과관계 검증 가능

## 8. 문제 정의 프로세스

### Step 1: 현상 수집

- 데이터 분석 (퍼널, 리텐션, 전환율)
- 사용자 피드백 (VOC, 리뷰, 인터뷰)
- 운영 이슈 (CS 문의, 버그 리포트)
- 비즈니스 리포트 (매출, 비용, KPI)

### Step 2: 문제 구조화

- 5 Whys로 근본 원인 탐색
- Problem Tree로 원인-결과 구조화
- JTBD로 사용자 관점 정리

### Step 3: 문제 명확화

- Problem Statement Canvas 작성
- 측정 가능한 형태로 정제
- 비즈니스 임팩트 수치화

### Step 4: 조기 검증 및 정렬 (Early Alignment)

솔루션(How/What)으로 넘어가기 전, 문제 정의(Why) 단계에서 반드시 합의를 거친다.

- 이해관계자와 문제 정의 합의
- 데이터로 문제 크기 검증
- 우선순위 결정
- **개발/디자인과 조기 싱크** (실현 가능성 피드백)

> **왜 조기 피드백이 중요한가?**
> - 문제 정의 단계에서 방향이 잘못되면 이후 모든 작업이 낭비됨
> - 기술적 제약이나 비용을 미리 파악하여 문제 범위 조정 가능
> - "다 만들고 나서 안 된다고요?" 상황 방지

### Step 5: 가설 수립

- 변경-대상-결과 구조로 가설 작성
- 검증 방법 설계 (실험, MVP, 파일럿)
- 성공/실패 기준 정의

## 9. 피해야 할 안티패턴

| 안티패턴 | 설명 | 결과 |
|----------|------|------|
| **솔루션 역행** | 솔루션을 먼저 정하고 문제를 끼워맞춤 | 진짜 문제를 해결하지 못함 |
| **현상 = 문제 혼동** | 표면적 현상을 문제로 정의 | 근본 원인 미해결, 재발 |
| **측정 불가** | "사용자 경험 개선" 같은 모호한 정의 | 성공/실패 판단 불가 |
| **범위 과대** | 너무 큰 문제를 한 번에 정의 | 실행 불가, 우선순위 혼란 |
| **데이터 무시** | 직감으로만 문제 정의 | 잘못된 가정에 기반한 솔루션 |
| **합의 누락** | 기획자 혼자 문제 정의 확정 | 팀 정렬 실패, 후반 번복 |
| **비즈니스 단절** | 유저 문제만 정의, 임팩트 미연결 | 우선순위 정당화 불가 |
| **가설 = 사실 착각** | 가설을 검증 없이 사실로 간주 | 잘못된 방향으로 리소스 투입 |

## 10. 문제 정의 작성 전 자문

1. 현상이 아닌 **근본 원인**을 정의했는가?
2. 솔루션을 먼저 생각하지 않았는가?
3. **측정 가능한 형태**로 표현되었는가?
4. **누가, 언제, 어디서** 겪는 문제인지 명확한가?
5. **비즈니스 임팩트**가 수치로 연결되었는가?
6. 이 문제를 해결하면 **사용자 행동**이 어떻게 바뀌는가?
7. **데이터/근거**가 있는가, 아니면 가정인가?
8. **이해관계자들이 동의**하는가?
9. 가설이 **검증 가능한 형태**인가?
10. 문제의 **크기/범위**가 실행 가능한 수준인가?

---

## 좋은 문제 정의 공식

```
좋은 문제 정의 = 근본 원인 + 구체적 맥락 + 측정 가능한 크기 + 비즈니스 임팩트
```

| 요소 | 질문 |
|------|------|
| 근본 원인 | 현상이 아닌 "왜"를 설명하는가? |
| 구체적 맥락 | 누가, 언제, 어디서 겪는지 명확한가? |
| 측정 가능한 크기 | 문제의 크기가 숫자로 표현되는가? |
| 비즈니스 임팩트 | 해결 시 기대 효과가 명확한가? |

### 간단 버전

```markdown
## 문제 정의

### 현상
[관찰된 현상/데이터]

### 문제
[누가] [언제/어디서] [무엇]을 겪고 있으며, 이는 [근본 원인] 때문이다.

### 비즈니스 임팩트
[지표]가 [현재 수치]로, 이는 [비즈니스 영향]을 의미한다.

### 가설
만약 [변경]을 하면, [대상]의 [지표]가 [기대 변화]할 것이다.
```

### 상세 버전

```markdown
## 문제 정의서

### 1. 배경
- 현재 상황 및 맥락
- 문제 발견 경위 (데이터/VOC/운영이슈)

### 2. 현상
- 관찰된 증상들
- 지표 변화 추이

### 3. 문제 분석
- 5 Whys / Problem Tree 결과
- 근본 원인 정리

### 4. 문제 정의
[Problem Statement Canvas 형식]

### 5. 비즈니스 임팩트
- 영향받는 지표
- 예상 손실/기회비용

### 6. 가설
- 가설 목록
- 각 가설의 검증 방법

### 7. 범위
- In-Scope / Out-of-Scope

### 8. 합의
- 이해관계자 리뷰 기록
```

## 11. 도메인 특화 예시: 배달/물류

### 예시 1: 배달 주문 취소율 문제

**현상**
> 주문 후 15분 이내 취소율이 8%에서 12%로 상승

**5 Whys 분석**
```
현상: 주문 취소율 상승
↓ Why?
1차: 고객이 주문 직후 취소를 요청한다
↓ Why?
2차: 예상 배달 시간이 기대보다 길다
↓ Why?
3차: 주문 확인 전 예상 시간과 주문 후 실제 시간이 다르다
↓ Why?
4차: 실시간 라이더 상황을 반영하지 않은 예측 시간을 표시한다
↓ Why?
5차: 주문 전 예상 시간은 평균값 기반, 주문 후에야 실제 라이더 매칭 결과 반영

→ 근본 문제: 주문 전 예상 배달 시간의 정확도가 낮아 고객 기대와 실제 사이에 괴리 발생
```

**Problem Statement Canvas**

| 항목 | 내용 |
|------|------|
| **Who** | 배달 앱 주문 고객 (전 연령대) |
| **What** | 주문 후 예상 배달 시간이 크게 늘어나 취소 |
| **Where** | 주문 완료 → 라이더 매칭 사이 |
| **When** | 주문 후 5분 이내, 특히 피크 시간대 |
| **Why** | 취소율 상승 → CS 비용 증가 + 가맹점 신뢰 하락 |
| **How much** | 취소율 8%→12%, 월 추정 손실 X억 원 |

**가설**
> 만약 주문 전 예상 배달 시간을 실시간 라이더 가용 상황 기반으로 계산하면, 주문 후 15분 이내 취소율이 12%에서 8%로 감소할 것이다.

**사전 학습 정의**
- ✅ 가설이 맞으면: 예상 시간 정확도가 핵심임 → 예측 모델 고도화에 투자
- ❌ 가설이 틀리면: 시간 외 다른 원인(메뉴 오류, 가격 재확인 등) 탐색

---

### 예시 2: 라이더 콜 수락률 하락

**현상**
> 점심 피크 시간(11:30~13:00) 콜 수락률이 75%에서 62%로 하락

**JTBD 관점**
```
[점심 피크 시간에] [배달 라이더]는 [수익을 최대화하기 위해] 
[짧은 거리, 높은 배달료의 콜만 선택]하고 싶다.

→ 문제: 배달료 책정이 거리/난이도를 충분히 반영하지 않아,
        먼 거리 콜은 기피되고 특정 지역 배달 지연 발생
```

**비즈니스 임팩트 연결**
```
[라이더가 먼 거리 콜 기피] 
    → [해당 지역 배달 지연] 
    → [고객 불만 및 재주문율 하락]
    → [해당 지역 가맹점 이탈 위험]
```

### 예시 3: 가맹점 정산 문의 증가

**현상**
> 정산 관련 CS 문의가 월 500건에서 800건으로 증가

**Problem Tree**
```
                    [결과/현상]
               정산 문의 800건/월
                       │
       ┌───────────────┼───────────────┐
       │               │               │
  [직접 원인]     [직접 원인]     [직접 원인]
  정산 내역      정산 금액        정산 일정
  이해 어려움    불일치 의심      예측 어려움
       │               │               │
  ┌────┴────┐    ┌────┴────┐    ┌────┴────┐
항목별 분류   전문 용어   수수료 계산   프로모션   정산 주기   공휴일
불명확       많음       공식 미공개   반영 시점   복잡       처리 안내 X
```

**솔루션 방향**
- 항목별 분류 불명확 → 정산 내역 UI 개편 (항목별 그룹핑)
- 수수료 계산 공식 미공개 → 정산 시뮬레이터 제공
- 정산 주기 복잡 → 예상 정산일 캘린더 제공

### 예시 4: 세트 메뉴 주문 오류

**현상**
> 세트 메뉴 주문의 12%에서 "구성품 누락" 클레임 발생

**문제 정의**
```
[세트 메뉴 주문 고객]이 [주문 완료 후 수령 시점에] 
[구성품 일부가 누락된 것]을 발견하며,
이는 [가맹점이 세트 구성을 정확히 파악하기 어렵기] 때문이다.

비즈니스 임팩트: 클레임 처리 비용 + 재배달 비용 + 고객 신뢰 하락
월 추정 손실: X천만 원
```

**가설**
> 만약 가맹점 주문 접수 화면에 세트 구성품을 체크리스트 형태로 표시하면, 세트 메뉴 구성품 누락 클레임이 12%에서 5% 이하로 감소할 것이다.

**지표 설계**
| 지표 유형 | 지표 | 목표 |
|----------|------|------|
| **핵심 (Value)** | 세트 메뉴 구성품 누락 클레임율 | 12% → 5% |
| **보조 (Usage)** | 체크리스트 사용률 | 90% 이상 |
| **가드레일** | 주문 처리 시간 | 현재 수준 유지 (증가 X) |
