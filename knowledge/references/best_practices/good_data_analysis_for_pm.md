# [Best Practice] PM을 위한 데이터 분석

> **핵심 메시지**: 데이터는 직관을 대체하는 것이 아니라 보완한다.  
> PM의 역할은 기술적 분석 능력이 아니라, 데이터를 통해 **올바른 질문을 던지고 의사결정을 내리는 것**이다.

---

## 핵심 요약

PM에게 데이터 분석 역량이란 "데이터 사이언티스트 수준의 분석 능력"이 아니다. **비즈니스 질문을 데이터 질문으로 변환하고, 분석 결과를 의사결정에 연결하는 능력**이다. 현상과 원인을 구분하고, 상관관계와 인과관계를 구분하며, 데이터로 가설을 검증하는 사고방식이 핵심이다.

---

## 1. PM의 데이터 분석 역할

### 1.1 PM이 직접 해야 하는 것 vs 협업해야 하는 것

| PM이 직접 해야 하는 것 | 데이터팀과 협업 |
|----------------------|----------------|
| 비즈니스 질문 정의 | 복잡한 데이터 추출 |
| 분석 방향 설정 | 통계적 유의성 검증 |
| 인사이트 해석 | 데이터 파이프라인 구축 |
| 의사결정 연결 | 고급 분석/모델링 |

### 1.2 데이터 분석의 목적

```
데이터 분석 ≠ 숫자 보고
데이터 분석 = 더 나은 의사결정

"이 숫자가 의미하는 것은 무엇인가?"
"그래서 우리는 무엇을 해야 하는가?"
```

## 2. 좋은 데이터 분석 접근 방식

### 2.1 질문에서 시작하라

```
❌ 나쁜 접근: "데이터를 보니 이런 게 있네요"
✅ 좋은 접근: "이 질문에 답하기 위해 데이터를 봅시다"

[질문 예시]
비즈니스 질문: "왜 매출이 떨어졌는가?"
           ↓
데이터 질문: "어떤 세그먼트의 전환율이 하락했는가?"
           ↓
구체적 질문: "신규 vs 기존, 모바일 vs 웹, 결제수단별로 나눠보자"
```

### 2.2 전체 → 세부 (Top-down)

```
[분석 순서]

1단계: 전체 그림 (숲)
└── 핵심 지표의 전체 트렌드 확인
    "전환율이 떨어졌다"

2단계: 세그먼트 분해 (나무)
└── 어떤 그룹에서 문제가 발생하는가?
    "기존 사용자에서만 떨어졌다"

3단계: 근본 원인 (뿌리)
└── 왜 그 그룹에서 문제가 발생하는가?
    "간편결제 사용자의 이탈이 원인이다"

4단계: 액션 도출
└── 무엇을 해야 하는가?
    "토스 결제 연동 오류를 수정하자"
```

### 2.3 비교 기준을 명확히 하라

```
[나쁜 예]
"전환율이 12%입니다."
→ 높은 건지 낮은 건지 모름

[좋은 예]
"전환율이 12%로,
- 지난 달(15%) 대비 3%p 하락
- 경쟁사 평균(14%) 대비 2%p 낮음
- 목표(18%) 대비 6%p 부족"
```

**비교 기준 유형**

| 비교 유형 | 설명 | 예시 |
|----------|------|------|
| **시계열** | 이전 기간 대비 | 전주, 전월, 전년 |
| **목표** | 설정한 목표 대비 | KPI, OKR 목표치 |
| **벤치마크** | 업계/경쟁사 대비 | 업계 평균, 경쟁사 |
| **세그먼트** | 다른 그룹 대비 | 신규 vs 기존 |

### 2.4 상관관계 ≠ 인과관계

```
[관찰]
"푸시 알림을 받은 사용자의 구매율이 높다"

[잘못된 해석]
"푸시를 더 보내면 구매율이 오른다"

[올바른 사고]
Q1: 푸시가 구매를 유발한 것인가?
Q2: 구매 성향이 높은 사람이 푸시를 허용한 것인가?
Q3: 다른 변수(앱 사용 빈도)가 둘 다 영향을 주는가?

→ 인과관계 확인: A/B 테스트 필요
```

### 2.5 맥락(Context)을 함께 보라

```
[데이터만 보면]
"주문 취소율이 2% → 5%로 상승"
→ 문제 발생?

[맥락과 함께 보면]
"무료 취소 정책 도입 후 취소율 상승"
"하지만 주문 전환율이 10% → 18%로 상승"
"순 주문 수는 20% 증가"
→ 오히려 긍정적 변화
```

## 3. 핵심 분석 프레임워크

### 3.1 퍼널 분석 (Funnel Analysis)

**목적**: 사용자 여정에서 이탈 지점 파악

```
[음식 배달 앱 결제 퍼널]

앱 실행     → 검색      → 상품 조회   → 장바구니   → 결제 완료
  100%     →   70%    →    45%     →    20%    →    12%
         (-30%)     (-25%)      (-25%)      (-8%)

▶ 최대 이탈: 검색 → 상품 조회 (25%p)
  → 검색 결과 품질 개선 필요
```

**퍼널 분석 시 고려사항**

| 고려사항 | 설명 |
|---------|------|
| **역순 분석** | 결제부터 앞으로 (비즈니스 임팩트 큰 단계부터) |
| **세그먼트** | 신규 vs 기존, 모바일 vs 웹 |
| **시계열** | 시간에 따른 변화 트렌드 |
| **절대값** | 비율뿐 아니라 절대 수치도 확인 |

### 3.2 코호트 분석 (Cohort Analysis)

**목적**: 시간에 따른 사용자 행동 변화 추적

```
[월별 리텐션 코호트]

가입월    Week0   Week1   Week2   Week3   Week4
─────────────────────────────────────────────
1월       100%    35%     22%     18%     15%
2월       100%    38%     25%     20%     17%
3월       100%    42%     28%     23%     19%
4월       100%    45%     32%     26%     22%

▶ 4월 가입자 리텐션 개선
  → 3월에 도입한 온보딩 개선 효과 확인
```

**코호트 분석이 중요한 이유**

```
[일반 리텐션]
"전체 리텐션 30%"
→ 신규 유입이 많으면 리텐션이 낮아 보임
→ 개선 효과인지 유입 변화인지 구분 불가

[코호트 리텐션]
"4월 가입자의 1주차 리텐션이 3월 대비 +7%p"
→ 특정 시점 가입자의 행동 변화 추적 가능
→ 제품 개선 효과 명확히 측정
```

### 3.3 세그먼트 분석

**목적**: 사용자 그룹별 차이 발견

```
[사용자 세그먼트별 전환율]

세그먼트        전환율    주문빈도    객단가
────────────────────────────────────────
신규 사용자      8%       1.2회      15,000원
복귀 사용자      15%      1.8회      18,000원
충성 사용자      35%      4.5회      22,000원

▶ 인사이트: 신규→복귀 전환 촉진이 ROI 높음
▶ 액션: 첫 주문 후 재방문 유도 캠페인
```

## 4. 지표 설계 원칙

### 4.1 좋은 지표의 조건

| 조건 | 설명 | 나쁜 예 → 좋은 예 |
|------|------|-----------------|
| **측정 가능** | 명확히 수치화 | "만족도" → "NPS 점수" |
| **행동 연결** | 액션으로 이어짐 | "총 방문자" → "검색 전환율" |
| **비교 가능** | 기준과 비교 | "매출 10억" → "목표 대비 85%" |
| **적시성** | 빠른 피드백 | "연간 매출" → "주간 GMV" |

### 4.2 지표 위계 (Metric Hierarchy)

```
                    ┌─────────────────┐
                    │  북극성 지표     │
                    │ (North Star)    │
                    │ 월간 총 주문 수  │
                    └────────┬────────┘
                             │
        ┌────────────────────┼────────────────────┐
        ▼                    ▼                    ▼
  ┌───────────┐        ┌───────────┐        ┌───────────┐
  │  활성 유저 │        │  전환율   │        │  주문 빈도│
  │   (MAU)   │        │  (CVR)    │        │           │
  └─────┬─────┘        └─────┬─────┘        └─────┬─────┘
        │                    │                    │
  ┌─────┴─────┐        ┌─────┴─────┐        ┌─────┴─────┐
  │ 신규 가입 │        │ 검색→조회 │        │ 재주문율  │
  │ DAU/WAU  │        │ 조회→결제 │        │ 주문 간격 │
  └───────────┘        └───────────┘        └───────────┘
```

### 4.3 지표 정의의 중요성

```
[같은 이름, 다른 정의]

"DAU(Daily Active Users)"

정의 A: 앱을 실행한 사용자 → 100,000명
정의 B: 로그인한 사용자 → 70,000명
정의 C: 1개 이상 액션을 한 사용자 → 50,000명

→ 지표 정의가 명확하지 않으면 소통 오류 발생
→ 지표 정의서 필수
```

**지표 정의서 항목**

| 항목 | 설명 |
|------|------|
| 지표명 | 공식 명칭 |
| 정의 | 무엇을 측정하는지 |
| 계산식 | 분자/분모 명확히 |
| 측정 주기 | 일간, 주간, 월간 |
| 데이터 소스 | 어느 테이블/이벤트 |
| 제외 조건 | 테스트 계정, 봇 등 |

## 5. 실무 분석 예시

### 5.1 문제 발견 → 원인 분석 → 액션

```
[상황]
금주 결제 전환율이 58% → 52%로 하락

[분석 단계]

Step 1: 전체 vs 세그먼트
├── 신규 사용자: 45% → 44% (변화 없음)
├── 기존 사용자: 65% → 55% (급락 ⚠️)
└── → 기존 사용자에서 문제 발생

Step 2: 기존 사용자 심층 분석
├── 결제 수단별
│   ├── 카드: 68% → 67% (변화 없음)
│   ├── 간편결제: 70% → 50% (급락 ⚠️)
│   └── → 간편결제에서 문제 발생
│
└── 간편결제 상세
    ├── 카카오페이: 72% → 70% (정상)
    ├── 토스: 65% → 30% (급락 ⚠️)
    └── → 토스 연동에서 문제 발생

Step 3: 원인 확인
└── 토스 API 응답 시간 증가 확인
    → 기술팀에 이슈 전달

Step 4: 액션
└── 토스 API 연동 수정 후 모니터링
```

### 5.2 가설 기반 분석

```
[가설]
"장바구니 페이지 리뉴얼 후 결제 전환율이 하락한 것은
쿠폰 적용 버튼의 가시성이 낮아졌기 때문이다"

[검증 설계]
1. 쿠폰 적용률 변화 확인
   - 리뉴얼 전: 45%
   - 리뉴얼 후: 28% (↓17%p) ← 변화 발생

2. 쿠폰 적용자 vs 미적용자 전환율
   - 적용자 전환율: 75% (변화 없음)
   - 미적용자 전환율: 40% (변화 없음)

3. 결론
   - 쿠폰 적용률 하락이 전체 전환율 하락의 원인
   - 쿠폰 버튼 가시성 개선 필요
```

## 6. 데이터 기반 의사결정

### 6.1 의사결정 프레임워크

```
[질문]
"간편결제를 도입해야 하는가?"

[데이터 수집]
1. 현재 결제 단계 이탈률: 32%
2. 이탈 사용자 설문: 결제 수단 불편 45%
3. 경쟁사 간편결제 도입 후 전환율 개선: +8%p
4. 예상 개발 비용: 2개월, 3명

[분석]
- 예상 전환율 개선: 32% × 45% × 0.5 = 7.2%p
- 예상 추가 매출: 월 2억원
- ROI: 3개월 내 회수 가능

[결정]
→ 도입 진행 (우선순위 P1)
```

### 6.2 데이터 vs 직관

| 상황 | 접근 방법 |
|------|---------|
| **데이터 충분 + 확실** | 데이터 기반 결정 |
| **데이터 부족 + 리스크 낮음** | 빠른 실험으로 검증 |
| **데이터 부족 + 리스크 높음** | 추가 리서치 후 결정 |
| **데이터와 직관이 충돌** | 가설을 명확히 하고 실험 |

### 6.3 불확실성을 인정하라

```
[나쁜 예]
"전환율이 5%p 상승합니다" (확정적)

[좋은 예]
"전환율이 3~7%p 상승할 것으로 예상합니다.
최악의 경우 1%p, 최선의 경우 10%p.
가장 가능성 높은 시나리오는 5%p입니다."

→ 의사결정자가 리스크를 판단할 수 있게 함
```

## 7. 피해야 할 안티패턴

### ❌ 안티패턴 1: 숫자 나열 = 분석

```
나쁜 예: "DAU 10만, MAU 50만, 전환율 12%입니다"
문제: 그래서 뭘 해야 하는지 불명확

좋은 예: "전환율이 12%로 경쟁사 대비 5%p 낮습니다.
        결제 단계 이탈이 주원인이며, 간편결제 도입으로
        3%p 개선을 목표합니다"
```

### ❌ 안티패턴 2: 상관관계 = 인과관계

```
관찰: "푸시 알림 받은 사용자의 구매율이 높다"
잘못된 결론: "푸시를 더 보내면 구매율이 오른다"

실제 원인: 충성 고객이 푸시를 허용하는 경향
→ A/B 테스트로 인과관계 검증 필요
```

### ❌ 안티패턴 3: Vanity Metrics 집착

```
허영 지표: 총 다운로드 수, 총 가입자 수
문제: 실제 비즈니스 성과와 무관

유의미한 지표: DAU, 리텐션, 전환율
→ "활성화된" 사용자 기준으로 측정
```

### ❌ 안티패턴 4: 데이터 없이 의사결정

```
문제: "제 경험상 이렇습니다" 만으로 결정
결과: 편향된 의사결정, 반복적 실패

해결: 
- 최소한의 데이터 확인 습관화
- 빠른 실험으로 검증
- "모르겠으면 측정한다"
```

### ❌ 안티패턴 5: 평균의 함정

```
[데이터]
"평균 결제 금액 25,000원"

[숨겨진 현실]
- 상위 10% 고객: 평균 80,000원
- 나머지 90%: 평균 18,000원

→ 평균만 보면 고객 행동 왜곡
→ 분포를 함께 확인해야 함
```

## 8. 체크리스트

### 분석 시작 전
- [ ] 명확한 비즈니스 질문이 있는가?
- [ ] 이 질문에 답하면 어떤 의사결정이 가능한가?
- [ ] 지표 정의가 명확한가?

### 분석 중
