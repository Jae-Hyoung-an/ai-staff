# PRD 자가 평가 체크리스트

> 오더수행 PRD 평가·개선 경험에서 도출한 실전 체크리스트
> 출처: projects/260121_work-review/reviews/ 및 improvement/
> 참조: good_prd_writing.md (이론적 기준)
> 최종 업데이트: 2026-04-02

---

## 사용법

PRD 작성 완료 후, 아래 체크리스트로 자가 평가를 수행한다.
"X" 항목이 2개 이상이면 개발팀 전달 전에 보완이 필요하다.

---

## 1. DoR (Definition of Ready) 체크

> 이 항목들을 모두 통과해야 개발 시작이 가능하다.

| # | 항목 | 체크 | 기준 |
|---|------|------|------|
| 1 | 사용자 스토리와 시나리오가 정의되어 있다 | | As a.../I want.../So that... 형식 |
| 2 | 비즈니스 정책이 구체적 숫자/조건으로 명시되어 있다 | | "논의 필요", "N회" 같은 미확정 값이 0건 |
| 3 | 성공 지표(핵심/보조/가드레일)가 정의되어 있다 | | 최소 핵심 1개, 가드레일 1개 |
| 4 | 우선순위(MoSCoW)가 합의되어 있다 | | Must/Should/Could/Won't 분류 |
| 5 | 디자인 문서 링크가 연결되어 있다 | | 피그마 또는 목업 |
| 6 | 이해관계자 리뷰가 완료되어 있다 | | 리뷰 코멘트 반영 완료 |

---

## 2. 구조 체크

| # | 항목 | 체크 | 흔한 실수 |
|---|------|------|-----------|
| 1 | PRD 본문과 회의록이 분리되어 있다 | | 회의 받아쓰기가 본문에 혼재 |
| 2 | 메타 정보(Epic, 작성자, 버전, 상태, 날짜)가 상단에 있다 | | 메타 정보 누락 |
| 3 | 변경 이력이 관리되고 있다 | | 날짜+변경내용 테이블 |
| 4 | 용어 정의 섹션이 있다 | | 도메인 특화 용어 미설명 |

## 3. 내용 품질 체크

### 3-1. 문제 정의 및 배경

| # | 항목 | 체크 | 기준 |
|---|------|------|------|
| 1 | 현재 상황(As-Is)이 데이터로 뒷받침되어 있다 | | 정성적 서술만 있으면 X |
| 2 | 핵심 문제가 표로 정리되어 있다 (문제-영향-근거) | | |
| 3 | "왜 만드는가"가 명확하다 | | 기능 나열만 있으면 X |

### 3-2. 성공 지표

| # | 항목 | 체크 | 기준 |
|---|------|------|------|
| 1 | 핵심 지표(Core): 이 기능이 성공했는지 판단하는 지표 | | 예: 수락률 65%→75% |
| 2 | 보조 지표(Usage): 사용 패턴을 추적하는 지표 | | 예: On 유지 시간 4시간+ |
| 3 | 가드레일 지표(Guardrail): 악화되면 안 되는 지표 | | 예: CS 문의 증가 없음 |

### 3-3. 기능 명세

| # | 항목 | 체크 | 기준 |
|---|------|------|------|
| 1 | 각 조건(if)에 대한 결과(then)가 명확하다 | | 모호한 표현 0건 |
| 2 | 숫자/기준이 구체적이다 | | "적절히"→3회, "빠르게"→30분 이내 |
| 3 | Edge Case와 Error Flow가 정의되어 있다 | | Happy Path만 있으면 X |
| 4 | 우선순위가 Must/Should/Could/Won't로 분류되어 있다 | | 전부 P0이면 X |

### 3-4. 비기능적 요구사항

| # | 항목 | 체크 | 기준 |
|---|------|------|------|
| 1 | 성능 요구사항 (응답 시간, 처리량) | | |
| 2 | 보안 요구사항 (개인정보 처리, 암호화) | | 위치/전화번호 다루면 필수 |
| 3 | 규제 요구사항 (개인정보보호법, 위치정보법) | | |

## 4. 흔한 안티패턴

> 오더수행 PRD 평가에서 발견된 실제 안티패턴 목록

| 안티패턴 | 설명 | 해결 |
|----------|------|------|
| **미확정 값 방치** | "논의 필요", "N회", "N미터" 등이 6주 이상 방치 | 임시 값이라도 확정하고 근거 명시 |
| **회의록 혼재** | 회의 받아쓰기가 PRD 본문에 섞임 | 회의록은 별도 파일로 분리 |
| **기능 나열 문서** | "무엇을"은 있지만 "왜"와 "성공 기준"이 없음 | 사용자 스토리 + 성공 지표 필수 |
| **우선순위 포기** | 모든 기능이 P0 | MoSCoW로 재분류, MVP 범위 명확화 |
| **개발자에게 판단 위임** | 정책 결정을 "구현 시 협의"로 미룸 | PO가 비즈니스 판단을 문서화 |

---

## 5. 평가 결과 해석

| 통과 항목 수 | 상태 | 액션 |
|-------------|------|------|
| 17개 이상 / 20개 | 개발 전달 가능 | 리뷰 후 Handoff |
| 13~16개 | 보완 필요 | 미통과 항목 보완 후 재평가 |
| 12개 이하 | 개발 시작 불가 | 전면 재작성 검토 |

---

> *"좋은 PRD는 비즈니스 판단을 문서화하고, 나쁜 PRD는 판단을 개발자에게 떠넘긴다."*
