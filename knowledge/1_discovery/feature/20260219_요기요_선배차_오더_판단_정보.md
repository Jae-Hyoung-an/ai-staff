## 핵심 요약

특정 오더가 **요기요 선배차 오더**인지 판단하는 방법:

1. **`order_assignment_policies` 테이블**의 `priority` 컬럼 확인
2. **`dispatch_type`** 필드 확인 (API 레벨)

## 테이블/컬럼 정보

### 1. `order_assignment_policies` 테이블

**테이블명**: `order_assignment_policies`  
**스키마**: `RAW_LASTMILE` 또는 `ebdb`

**핵심 컬럼**:
- `order_id`: 오더 ID (Orders 테이블과 조인)
- `priority`: 배정 정책 우선순위 (INT 또는 ENUM)

**판단 방법**:
```sql
-- 요기요 선배차 오더 확인
SELECT o.*
FROM orders o
INNER JOIN order_assignment_policies oap 
    ON o.id = oap.order_id
WHERE oap.priority = 'RIDER_ASSIGN_FIRST_BEFORE_STORE_SUBMIT'
   OR oap.priority = 1  -- 숫자로 저장될 수 있음
```

**참고**:
- `OrderAssignmentPolicyPriority` enum에 `RIDER_ASSIGN_FIRST_BEFORE_STORE_SUBMIT` 값이 추가됨
- 일부 데이터에서는 `priority = 1`로 저장될 수 있음 (Jira 이슈 ITSMCHG-22495 참고)

### 2. `dispatch_type` 필드 (API 레벨)

**필드명**: `dispatch_type`  
**위치**: 오더 생성 API 요청/응답

**값**:
- `DISPATCH_FIRST`: 라이더 선배차 (요기요 선배차)
- `SEND_TO_VENDOR_FIRST` 또는 `null`: 상점 선전달 (일반 오더)

**API 예시**:
```json
{
  "assignmentPolicies": [
    {
      "priority": "RIDER_ASSIGN_FIRST_BEFORE_STORE_SUBMIT"
    }
  ],
  "dispatch_type": "DISPATCH_FIRST"
}
```

## 기능 개요

### 요기요 라이더 선배차란?

요기요 오더의 라이더 미배차 해결책 중 하나로, **오더 생성 후 라이더가 먼저 오더를 잡으면 상점에 오더를 노출**하는 방식입니다.

**프로세스**:
1. 오더 생성 시 라이더에게 먼저 노출
2. 라이더가 오더 수락 → 상점에 오더 전달
3. 상점에서 조리 시간 추가 요청 가능
4. 특정 조건에서 배차 취소 정책 적용

### 두 가지 타입 비교

| 타입 | `dispatch_type` | 설명 |
|------|----------------|------|
| **상점 선전달** | `null` 또는 `SEND_TO_VENDOR_FIRST` | 일반적인 케이스로 오더 생성 시 상점에 오더 노출 |
| **라이더 선배차** | `DISPATCH_FIRST` | 오더 생성 후 라이더가 오더를 잡으면 상점에 오더 노출 |

## 주요 특징

### 1. 픽업 요청 시간 관리

- **초기 픽업 요청 시각**: 오더 생성 시 STORE의 픽업 요청 시각
- **조리 시간 추가**: 오더 당 1회만 전달 가능
  - 두 번째 들어온 STORE 픽업 요청 시각을 조리 시간 추가 요청으로 간주
- **픽업 시간 조정**: 조리 시간 추가 요청 전까지 기사가 픽업 요청 시각 변경 불가

### 2. 배차 취소 정책

**조리 시간 추가 요청 전**:
- 무료 취소 가능

**조리 시간 추가 요청 후 3분 이내**:
- 픽업 요청 시간 차이가 **15분 이상** → 무료 취소
- 픽업 요청 시간 차이가 **15분 미만** → 일반 취소 정책 적용
  - 2분 이내: 24시간 기준 2번 무료 취소 가능
  - 5분 이내: 500원
  - 5분 이상: 1000원

**조리 시간 추가 요청 후 3분 초과**:
- 기사 배차 취소 불가 (매니저는 가능)

### 3. 제외 조건

- "Friends-only" 지점은 제외

## 데이터 확인 방법

### SQL 쿼리 예시

```sql
-- 방법 1: order_assignment_policies 테이블 조인
SELECT 
    o.id AS order_id,
    o.ordernumber AS order_number,
    oap.priority,
    CASE 
        WHEN oap.priority = 'RIDER_ASSIGN_FIRST_BEFORE_STORE_SUBMIT' 
             OR oap.priority = 1 
        THEN '요기요 선배차'
        ELSE '일반 오더'
    END AS order_type
FROM orders o
LEFT JOIN order_assignment_policies oap 
    ON o.id = oap.order_id
WHERE o.createdat >= '2025-01-01'
  AND (oap.priority = 'RIDER_ASSIGN_FIRST_BEFORE_STORE_SUBMIT' 
       OR oap.priority = 1);
```

### 주의사항

1. **데이터 누락 가능성**: 
   - 일부 오더는 `order_assignment_policies` 테이블에 데이터가 없을 수 있음
   - Jira 이슈 ITSMCHG-23570에서 확인된 바와 같이, 삭제 로직에 의해 제거될 수 있음

2. **Priority 값**:
   - Enum 값: `RIDER_ASSIGN_FIRST_BEFORE_STORE_SUBMIT`
   - 숫자 값: `1` (일부 데이터에서 사용)

3. **데이터 정합성**:
   - `priority = 1`인 데이터는 삭제 대상으로 검토되었음 (ITSMCHG-22495)
   - 현재 사용 중인지 확인 필요
