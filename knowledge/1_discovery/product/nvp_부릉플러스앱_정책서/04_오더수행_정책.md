## 0. 용어 정리

### 오더 유형

| 용어 | 정의 |
|------|------|
| NVP 오더 | 오더 생성 상점의 관제 지점이 NVP 지점인 오더 |
| 일반 오더 | 기본 배송 유형. 여러 오더를 동시 수행 가능 |
| 단건 오더 | 한 번에 하나의 주문만 수행. 수행 중 다른 오더 동시수행 불가 |
| 묶음 오더 | 여러 건을 하나로 묶어 동시 제안. 묶음 내 오더를 순서대로 수행 |
| 과적 오더 | 물건의 무게/양이 많아 추가 운임이 필요한 경우 원오더에 추가로 묶여 생성 |
| 원오더 | 과적 오더의 기준이 되는 원래 오더. 과적 오더는 원오더에 종속 |
| 화주사 연동 오더 | 외부 화주사와 시스템 연동되어 생성 (올리브영, 스타벅스 등) |
| 동시제안 오더 | AI 배차가 하나의 오더를 여러 기사(이륜+프렌즈)에게 동시 제안 |

### 기사 액션 - 배정 및 제안

| 용어 | 정의 |
|------|------|
| 오더 제안 | 시스템이 기사에게 오더 수행을 제안 |
| 오더 수락 | 기사가 제안된 오더를 수락하여 배정받는 행위 |
| 오더 거절 | 기사가 제안된 오더를 거절하는 행위 |
| 강제 배차 | 관리자가 기사에게 오더를 강제로 배정 |

### 기사 액션 - 수행 흐름

| 용어 | 정의 |
|------|------|
| 수행 목록 | 현재 배정되어 수행해야 할 오더 목록 |
| 수행 Task | 기사가 처리해야 할 개별 작업 단위 (픽업, 배달 등) |
| 상점 도착 | 기사가 상점에 도착했음을 앱에서 알리는 행위 |
| 픽업 완료 | 상점에서 물건을 수령 후 앱에서 완료 처리 |
| 배달 사진 촬영 | 선불 오더 배달 완료 시 증빙을 위해 사진 촬영·등록 |
| 후불결제 | 배달 완료 시점에 기사가 고객으로부터 직접 결제를 받는 방식 |
| 배달 완료 | 고객에게 물건을 전달 후 앱에서 완료 처리 |

### 예외 처리

| 용어 | 정의 |
|------|------|
| 재이동 | 픽업 중 or 배송 완료 후 도착지 변경 시 신규 수행지로 재배송 |
| 반납 | 배송 불가 판단 시 물건을 상점으로 돌려보내는 처리 |

## 1. 기본 오더 수행 플로우 (해피패스)

```
오더 추천 발생 → 수락/거절 제안 (30초 타이머) → 수락 → 오더 배정 완료
  → 상점 도착 클릭 → 픽업 완료 클릭
    → [선불] 배달 사진 촬영 → 배달 완료
    → [후불] 결제하기 → 배달 완료
```

| 항목 | Given | When | Then |
|------|-------|------|------|
| 오더 수락 | 신규 오더 제안 on 상태인 기사 | 30초 이내에 수락 버튼 클릭 | 오더 수락 처리 |
| 오더 거절 | 신규 오더 제안 on 상태인 기사 | 거절 버튼 클릭 / 30초 무응답 | 오더 거절 처리 |

## 2. 신규 오더 제안 On/Off

NVP 기사가 신규 오더 제안을 수신할지 여부를 선택하는 정책.

### 2-1. On/Off 기본 동작

| 항목 | 정책 |
|------|------|
| Default 값 | **off** |
| On 전환 시 | AI 추천배차를 통한 신규배차 제안 수신 가능 |
| Off 전환 시 | AI 추천배차 후보에서 제외 |
| Off 시, 기배정 오더 | On→Off로 전환해도 기배정 오더에는 영향 없음 |

### 2-2. 자동 Off 전환 조건

| 항목 | 정책 | 예외 |
|------|------|------|
| 무응답 거절 3회 연속 | 오더 수행 중에도 3회 연속 무응답 시 Off | - |
| 2시간 동안 오더 미수행 | 근무 시작 후 2시간 경과 미수행 / 마지막 완료 후 2시간 경과 미수행 | 현재 수행 중인 주문이 있으면 정지 안 함 |

### 2-3. 최대 동시 수행 가능 오더 건수

| 항목 | 정책 |
|------|------|
| 최대 동시 수행 가능 건수 | **3건** [논의 중] |

## 3. 오더 정보 노출

오더 제안 → 픽업지 이동 → 배달지 이동 전 과정에서 기사에게 노출하는 정보.

### 3-1. 오더 제안 시 바텀시트

| 제안 유형 | 노출 항목 |
|-----------|-----------|
| 일반오더 (1건) | 상점명, 총 원가, 픽/도착지간 직선 거리 |
| 묶음오더 | '묶음' 배지, 각 오더 원가의 총합, 제안 오더 목록 (상점명, 원가, 직선거리) |
| 동시제안 오더 | '다른 기사가 먼저 잡을 수 있어요' 안내, '프로모션' 배지, 상점명, 총 원가, 직선거리 |

### 3-2. 픽업지 이동 시 바텀시트

**접힘 상태:** 상점명, baseAddress (도로명/지번 + 상세주소)

**펼침 상태:**

| 항목 | 정의 | 비고 |
|------|------|------|
| 상점명 | 인트라 상점 정보의 '상점명' | - |
| baseAddress | 고객이 입력한 주소 | 도로명/지번 + 상세주소 |
| 상품픽업번호 | 요기요(숏코드 4자리), 홈플러스/CU(자체주문번호) | 없는 경우 '고객 주소로 확인' 또는 배달지 주소 노출 |
| 오더 정보 | 상품 총 금액, 결제 유형(선불/후불카드/후불현금), 각 상품명·금액 | - |
| 배송메세지 | 고객이 남긴 원문 | 맥도날드/KFC/올리브영: 배송메세지 내 주문번호로 소통 |
| 부릉오더번호 | VCC 소통용 | - |

### 3-3. 배달지 이동 시 바텀시트

**접힘 상태:**

| 항목 | 정의 | 비고 |
|------|------|------|
| 상점명 | 인트라 상점 정보의 '상점명' | - |
| roadName + buildingNo | 고객 주소 좌표의 역지오코딩 도로명 | 도로명+빌딩번호 |
| baseAddress | 고객이 입력한 주소 | 도로명/지번 + 상세주소 |
| fallback | 도로명이 없는 지역 | 지번 주소 + 위경도 노출 |

**펼침 상태:** 픽업지와 동일한 구조 (상점명, baseAddress, 상품픽업번호, 오더정보, 배송메세지, 부릉오더번호)

### 3-4. 픽업지 지도 핀 안내 문구

| 조건 | 판단 기준 | 노출 문구 | 갱신 주기 |
|------|-----------|-----------|-----------|
| 픽업 요청시간 5분 초과 전 | 현재 < 픽업요청시간 - 5분 | - | - |
| 픽업 요청시간 5분 이내 | 픽업요청시간-5분 ≤ 현재 < 픽업요청시간 | "N분 후 픽업" | 1분 단위 차감 |
| 상점 조리완료 처리 | 상점이 조리완료 버튼 누름 | "준비완료" | 고정 |
| 픽업 요청시간 경과 | 현재 ≥ 픽업요청시간 | "픽업 N분 지남" | 1분 단위 증가 |

### 3-5. 안심번호

- 별도 정책 문서 참조

### 3-6. 고객 정보 만료

**만료 트리거:**

| 항목 | 만료 시점 |
|------|-----------|
| 기본 만료 | 상점 기본 정보 > 기사앱 내 고객정보 표시 시간(분)에 설정된 시간 경과 |
| 주문 취소 | 주문 취소 직후 |
| 오더 완료 | 오더 완료 후 30분 |

- 고객 정보 파기: 오더 생성 후 30일 / SPC 등 일부 요청에 따라 10일 후 파기

**만료 시 처리:**

| 항목 | 정책 |
|------|------|
| 배달지 baseAddress | '고객 정보 만료'로 표시 |
| 배송메세지 | 미노출 |
| 상품메뉴 | 미노출 |
| 고객 전화, 문자 | 전송 불가 |
| 고객 결제내역 | 조회 가능 |

### 3-7. 기상할증 표시

기상할증 적용 3가지 조건 모두 충족 시:
1. 인트라 > 할증 관리 > 기상 할증 설정
2. 인트라 > 상점 관리 > 지점 기상할증 허용 여부: 허용
3. 매니저 PC/M > 기상 할증 ON

→ 메인 화면에 `기상할증 적용중` 배지 노출

## 4. 수행 순서 안내

기사가 수락한 오더의 수행 Task와 순서를 지도·목록으로 안내.

### 4-1. 지도 표시

| 항목 | 정책 | 비고 |
|------|------|------|
| 지도 표시 범위 | 기사 위치 ↔ 다음 이동 지점만 표시 | 전체 경로 미표시 |
| 연결 방식 | 직선으로 자동 연결 | - |
| 다음 목적지=상점 | 핀 상단에 픽업 잔여/경과 시간 표시 | - |
| 다음 목적지=고객 | 핀 상단에 상점명 표시 | 최대 10글자 + ... + 고객 |

### 4-2. 수행 목록

| 항목 | 정책 |
|------|------|
| 접근 경로 | 오더 수행 중일 때 메인화면에 별도 버튼 |
| 수행 목록 | 상점명 + 오더 번호 |
| 수행 task 상세 정보 | 이전 단계 완료 후 표시됨 |

### 4-3. 수행 완료/취소 오더 목록 노출

- 완료 후 **30분** 경과 시 수행 목록에서 제거

| Task 상태 | 노출 유지 기준 |
|-----------|----------------|
| 픽업 완료 | 완료 후 30분 |
| 배달 완료 | 완료 후 30분 |
| 취소됨 | 이전 task 완료 후 30분 |

- **예외 (잔여 결제금액 존재 시):** 고객 정보 만료 처리 안 함, 경고 문구 표시, 결제하기 버튼 표시 → 분할결제 바텀시트 노출

## 7. 배송 완료 / 사진 촬영 검증

### 7-1. 기본 정책

| 항목 | 정책 |
|------|------|
| 사진 촬영 대상 | 선불 오더 배송 완료 시 |
| 촬영 화면 진입 | 배송 완료 시도 시 자동 이동 |
| 촬영 건너뛰기 | 가능 |

### 7-2. 위치 기반 완료 검증 (선완료 방지)

| 항목 | 정책 |
|------|------|
| 검증 대상 | 위치 기반 검증 필요 오더 |
| 대상 화주사 | 올리브영 |
| 검증 범위 | 100m |
| bypass 가능 여부 | 가능 (`계속하기` 버튼) |

### 7-3. 문자 발송

배달 완료 문자를 요청한 고객에게 전송 가능.

**일반 문자 발송:**

| 항목 | 내용 |
|------|------|
| 입력 방식 | 라디오 선택 또는 직접 입력 |
| 발신 번호 | 기사 번호 → 고객 번호로 직접 발송 |
| UX | 문자 보내기 버튼 tap → 내용 선택 → 전송 |

일반 문자 선택지: (1) 문 앞 배달 완료 (2) 위탁 장소 배달 완료 (3) 고객 연락 불가 (4) 배달 도착 예정 (5) 직접 입력

**스타벅스 문자 발송:**

| 항목 | 내용 |
|------|------|
| 입력 방식 | 라디오 선택만 가능 (직접 입력 없음) |
| 발신 주체 | 부릉 시스템 메시지 |
| UX | 배달 완료 시 바텀시트 자동 노출 |

스타벅스 문자 선택지: (1) 문 앞 배달 완료 (2) 위탁 장소 배달 완료 (3) 고객 연락 불가

### 7-4. 배송수수료 입금

- 배달완료 직후, 배송수수료 M캐시 입금

## 8. 후불결제

### 8-1. 결제 방식

| 항목 | 정책 |
|------|------|
| 결제 수단 | 카드기 없이 현금/카드 결제 가능 |
| 결제 완료 시 | 자동으로 배송 완료 처리 |

### 8-2. 카드 결제 에러케이스

| 사유 | 액션 |
|------|------|
| KIS Pay 앱 미설치 | KIS Pay 앱 설치 유도 |
| KIS Pay 앱 실행 불가 | VCC 전화연결 |
| PG 결제 상점 | VCC 전화연결 |
| 가맹점 미확인 | VCC 전화연결 |
| 다수 가맹점 | VCC 전화연결 |
| 가맹점 해지 | VCC 전화연결 |

결제 완료 후 에러케이스 (KIS 거래 에러, 가맹점 정보 없음, 중복 결제, 서버 기록 실패) → 모두 VCC 전화연결

### 8-3. 분할결제

| 항목 | 정책 |
|------|------|
| 분할결제 가능 여부 | 가능 |
| 분할결제 조합 | 현금+카드, 현금+현금, 카드+카드 |
| 최대 횟수 | 없음 (잔여 금액 없을 때까지) |
| 컵 보증금 | 300원 단위 입력 |

분할결제 에러:
- 보증금 and 상품가액 '0'원/입력값 없음 → 결제 버튼 비활성화
- 상품가액이 잔여상품가액 초과 → 결제 비활성화 + `잔여 상품가액 보다 작게 입력하세요` (최대 999,999,999원)

### 8-4. 후불오더 재결제 시 기사 제한

### 8-5. 기사매입

- **현행: NVP 오더는 기사매입을 불허용**

## 10. 인트라 상태 변경

인트라에서 오더/수행 task 상태가 변경될 때의 처리 정책.

### 10-1. 수행 목록 TASK 공통 규칙

| 항목 | 정책 |
|------|------|
| Task 완료 후 노출 | 수행 완료 후 30분까지 |
| 최대 노출 건수 | 최대 3건 |
| 재이동/반납 태그 | 해당 Task 발생 시만 표시 |
| Task 생성 기준 | 완료 이력 기준으로 필요한 Task 생성 |

### 10-2. 오더 완료 → 픽업 역방향 변경

| 항목 | 정책 |
|------|------|
| 처리 주체 | 인트라 (관리자) |
| 처리 방식 | '기사앱 적용' / '기사앱 미적용' 선택 |
| 앱 반영 | 배달완료 → 픽업 상태로 되돌아감 |
| 수행 task 처리 | 최상단 Task로 노출 |

**기사앱 적용 vs 미적용:**

| 인트라 액션 | 기사앱 동작 |
|-------------|-------------|
| 기사앱 적용 | 기사에게 노출되는 오더 상태 변경 |
| 기사앱 미적용 | 앱 미노출 + 배달내역 반영 + 배달료 변경 시 M캐시 출금/입금 재처리 |

**역방향 변경 주의사항:**
- 기사앱 미적용 + 주소 변경 → 배송료 취소(출금) 후 재입금, 배달내역에 반영
- 기사앱 미적용 + 결제수단/금액 변경 → 지도 화면에서 배달 Task 미노출. 잔여 결제금액 존재 + 오더 완료 후 24시간 이내 시 배달내역에서 재결제 가능

> 상세: `projects/260402_prd_화주사_과적오더_묶음배차/02_기능명세_화주사_과적오더_묶음배차.md`

**핵심 정책:**

| 항목 | 정책 |
|------|------|
| AI배차 대상 | 원오더만 배차 시도. 화주사 과적 오더는 AI배차 대상 제외 |
| 과적 오더 갯수 제한 | 없음. 원오더에 종속되는 과적 오더 갯수에 제한을 두지 않음 |
| 배정 전 과적 존재 시 | 원오더와 과적 오더를 묶음 형태로 기사에게 제안. 합산 원가 표기. 1회 수락으로 전체 배차 |
| 배정 후 과적 생성 시 | 원오더 배정 기사에게 자동 강제 배정. 앱 내 안내 (추가 건수, 추가 배송비) |

| 묶음 유지 원칙 | 원오더가 취소되지 않는 한 묶음은 유지. 배차 취소/접수 전환 시에도 묶음 해제 안 됨 |
| 묶음 해제 조건 | **원오더 취소 시에만** 묶음 해제 → 과적 오더가 별개 일반 오더로 전환 |
| 배차 취소 시 | 묶음 유지한 채 다른 플러스앱 기사 및 프렌즈에게 묶음 오더로 재제안 |
| 접수 상태 전환 시 | 묶음 유지 + 독점 제안 지속 + 플러스앱 셀프배차 노출(원오더만, 합산 원가) + 2륜 미노출 + 프렌즈 셀프배차 미노출 |
| VCC 배차 | 묶음 오더 존재 시 원오더+과적 오더 모두 VCC 배차 대상 제외. 목록 미노출. 묶음 해제 시 개별 VCC 배차 대상 |
| 최대 동시 수행 건수 | 원오더만 카운트 (과적 오더 미포함) |
| 배차 취소 카운트 | 원오더 기준 1건만 카운트 |
| 배달 완료 사진 | 원오더 촬영 사진을 종속 과적 오더에도 적용 |
| 배달내역 | 원오더와 과적 오더 각각 개별 건으로 표시. 오더별 배송비 확인 가능 |

## 12. 재이동

픽업 중 또는 배송 완료 후 도착지 변경 시 기사가 신규 수행지로 재이동하는 처리.

### 12-1. 재이동 발생 사유

| # | 사유 | 상세 |
|---|------|------|
| 1 | 결제금액/결제수단 변경 | 고객 결제 관련 변경으로 재배송 필요 |
| 2 | 기사 과실 오배송/정정배송 | 기사 실수로 잘못 배송, 정정 필요 |
| 3 | 상점 과실 정정배송 | 상점 실수로 잘못된 상품, 정정 필요 |

### 12-2. 케이스 분류

- 재이동 요금제 없음 (요금제 미적용 상점)
- 재이동 요금제 있음 - 기사가 상품을 들고 있는 경우
- 재이동 요금제 있음 - 배달지에 상품이 배달 완료된 경우

### 12-3. 요금제 미적용 상점

| 항목 | 정책 |
|------|------|
| 처리 주체 | VCC |
| 기사 수행 흐름 | 픽업지 → 배달지A(원래) → 상점(복귀) → 배달지B(신규) |
| 재이동 태그 | 표기되지 않음 |

### 12-4. 요금제 적용 - Task 처리 규칙

**공통 규칙:**

| 항목 | 정책 |
|------|------|
| Task 삽입 위치 | 최상단 (수행 목록 맨 위에 추가) |
| Task 생성 기준 | 완료 이력 기준으로 필요한 Task 생성 |
| 재이동 도착지 삭제 시 | 기존 배달지 Task로 원복 |
| 재이동 → 반납 전환 | 재이동 중/후에 반납으로 변경 가능 |

**시나리오별 Task 처리:**

| # | 선행 상태 | 트리거 | 완료된 TASK | 삭제 TASK | 추가 TASK | 태그 |
|---|-----------|--------|-------------|-----------|-----------|------|
| 1 | 상점 픽업 완료, 배달 전 | 재이동 | 상점→픽업 | 배달지A→배달 | 배달지B→배달 | 재이동 |
| 2 | 배달지A 배달 완료 | 재이동 | 상점→픽업, 배달지A→배달 | - | 배달지A→픽업, 배달지B→배달 | 재이동 |
| 3 | #1 후 (배달지B 배달 전) | 반납 전환 | 상점→픽업 | 배달지B→배달 | 상점→반납 | 반납 |
| 4 | #2 후, 배달지A 픽업 완료 | 반납 전환 | 상점→픽업, 배달지A→배달, 배달지A→픽업 | 배달지B→배달 | 상점→반납 | 반납 |
| 5 | #2 후, 배달지A 픽업 전 | 반납 전환 | 상점→픽업, 배달지A→배달 | 배달지A→픽업, 배달지B→배달 | 배달지A→픽업, 상점→반납 | 반납 |
| 6 | #2 후, 배달지B 배달 완료 | 반납 전환 | 상점→픽업, 배달지A→배달, 배달지A→픽업, 배달지B→배달 | - | 배달지B→픽업, 상점→반납 | 반납 |

## 14. 배송비 재계산

### 14-1. 행위별 재계산 방식

| 행위 | 재계산 방식 |
|------|-------------|
| 상점 취소 | 상점에 설정된 수수료 비율에 따라 결정 |
| 재이동 | 기본 원가는 이전 도착지 기준, 재이동 추가 비용은 그대로 지급 |
| 도착지 수정 | 지역성 할증만 새 도착지 기준 재계산, 나머지 할증 유지 |
| 반납 | 상점 설정된 반납 요율(보통 50%), 원가 지급은 VCC 수기 지급 |

### 14-2. 배송 완료 후 상태 변경 시 M캐시 정산

| 행위 | 정산 처리 |
|------|-----------|
| 배송 완료 후 상태 변경 + 주소 변경 | 기존 수수료 M캐시 역이체(취소) → 변경 주소 기준 재계산 → 재완료 시 새 금액 재정산 |
| 배송 완료 후 상태 변경 + 주소 변경 없음 | 기존 수수료 M캐시 역이체(취소) → 재완료 시 동일 금액 재정산 |
