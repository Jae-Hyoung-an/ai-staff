## 1. 제품관리란?

제품관리는 **제품이 고도화됨에 따라 고객가치와 비즈니스 가치가 함께 상승할 수 있도록 하는 관리방식**이다.

### 제품 고도화 사이클

```
고객가치
    ▲
    │                    Cycle3
    │               ┌─────────────┐
    │    Cycle2     │ 문제정의    │
    │   ┌────────┐  │  ↓         │
    │   │문제정의 │  │ 가설       │         제품
Cycle1 │  ↓      │  │  ↓         │    ──▶ 로드맵
┌────┐ │ 가설    │  │ 배포       │
│문제 │ │  ↓     │  │  ↓         │
│정의 │ │ 배포   │  │ 문제해결   │
│ ↓  │ │  ↓     │  │ (지표달성) │
│가설 │ │ 검증   │  │  ↓         │
│ ↓  │ └────────┘  │ 검증       │
│배포 │             └─────────────┘
│ ↓  │
│검증 │
└────┘─────────────────────────────────────▶ 비즈니스 가치
```

**핵심**: 각 사이클은 **문제정의 → 가설수립 → 배포 → 문제해결(지표달성) → 검증**의 순환을 통해 점진적으로 고도화된다.

## 2. 제품관리 프로세스 개요

제품의 목표 설계부터 **'제품 기획 → 배포 → 유저 임팩트 분석 → 회고'**의 순환과정을 아우르는 전반적인 프로세스

### 프로세스 구조

| 단계 | 주요 활동 | 참여자 |
|------|----------|--------|
| **사전 논의** | Top 레벨 목표 협의(상위지표), TF Kick-Off | 경영진, PO |
| **스프린트** | 제품 기획(with UX), 제품 개발(Maker) | PO, UX, 개발팀 |
| **검증** | 운영배포, 스프린트 종료 | QA, 운영팀 |
| **회고** | 제품 리뷰(이해관계자), 실무 구성원 피드백 | 전체 |

> **측정 필요**: 스프린트가 반복될수록 제품조직의 생산성이 향상되는지 (스토리 포인트 등)

## 3. 제품기획 산출물 간 연계

### 산출물 흐름

```
스토리 → 고객여정(CJM) → Flow Chart → 와이어프레임(스토리보드) → 디자인
```

### 영역별 역할

| 영역 | 담당 | 주요 산출물 |
|------|------|------------|
| **PO 영역** | Product Owner | 지라 스토리 생성, 스프린트 플래닝, 후행지표 설정 |
| **UX 영역** | UX 디자이너 | 고객여정, Flow Chart, 와이어프레임, 아키텍처 설계 |
| **개발 영역** | 개발팀 | 개발설계, API 개발, 서버 구축, 디자인 리뷰 |

### 지표 체계

| 유형 | 설명 |
|------|------|
| **후행지표 (Key Result)** | 최종 결과를 측정하는 지표 |
| **선행지표 (Input Metric)** | 과정을 측정하는 지표 + 인과관계 설정 |

## 4. 제품개발 협의 프로세스

유관부서의 제품 요구사항 협의 프로세스를 수립하고, **우선순위 기준에 따라 백로그를 관리**한다.

### 요구사항 처리 흐름

```
┌─────────────────────────────────────────────────────────────────┐
│ 현업 (사업/운영조직)                                             │
│   요구사항 발생 → 인터뷰/이메일 질의 → WF 상신                   │
├─────────────────────────────────────────────────────────────────┤
│ 제품팀                                                          │
│   요구사항 접수 → Type 분류 (New/Sustaining)                    │
│        ↓                                                        │
│   1차 분석 → 2차 분석(상세) → PRD 작성 → 과제선별위원회(승인)   │
├─────────────────────────────────────────────────────────────────┤
│ 각 업무 Part                                                    │
│   개발 진행 → 과제 착수 → R&R/일정 수립 → 과제 관리 → 과제 종료 │
└─────────────────────────────────────────────────────────────────┘
```

### 우선순위 결정 기준

- OKR 기준 비즈니스 임팩트
- Top Level Initiative
- Hot fix 건

## 5. 제품 품질 관리

### 품질(Quality) 정의

| 관점 | 정의 |
|------|------|
| **일반적** | 목적에 대한 적합성. 제품/서비스가 의도한 목적을 모두 만족시키면 품질 확보 |
| **S/W 방법론** | 주어진 기간에 정해진 예산으로 목적의 만족도를 달성하며 생산되는 것 |
| **QA** | 프로덕트 품질의 기준을 제시하고 달성하는 것을 목표로 활동하는 역할 |

### 품질 관리 방안

#### 방안 A: 기획과 테스트 케이스의 거리 좁히기

| AS-IS (기존) | TO-BE (개선) |
|-------------|-------------|
| 스토리작성 → UX기획 → TC작성 → 개발 → 검증(QA) | 스토리작성 + TC작성 **병행** → UX기획 → 개발 → 검증(QA) |
| 기획의도 파악 **불분명** | 기획의도를 포함한 TC 기반 검증 진행 **가능** |

#### 방안 B: 테스트 순환 횟수 늘리기

```
개발완료 70%  →  개발완료 80%  →  개발완료 90%  →  개발완료 100%
     ↓               ↓               ↓               ↓
   검증(QA)       검증(QA)        검증(QA)        검증(QA)
```

- 개발이 완료되는 부분부터 **빠르게 테스트 시작**
- 테스트 순환이 될수록 TC 수행 **Know-how 확보** → 엄밀한 검증 가능

#### 방안 C: 기술부채 줄이기

- 비즈니스 임팩트 과제와 기술부채/운영부채 해결 과제 **분리 운영**
- 회원/결제/Push 시스템/Data Infra/리팩토링 등 공통 영역 고도화
- 예시: 우아한형제들 **'피트스탑'** 제도

## 6. 제품배포 프로세스

개발 환경에서 작성된 코드가 최종적으로 **운영 환경에서 실행되기까지의 과정**

### 배포 흐름

```
제품기획 완료
    ↓
개발설계 (API 설계, DB 모델링 등)
    ↓
개발 (DB 구축, 코드작성 등)
    ↓
코드 빌드 & 자체테스트
    ↓
코드리뷰
    ↓
코드 충돌해결
    ↓
코드 저장소 반영
    ↓
인수요청 → 인수확인 → QA
```

### 제품관리자의 역할 (배포 기간)

1. 개발 진행 중 **누락/수정**되어야 하는 제품기획 내용 대응
2. 배포 전/후 필요한 내용들을 정리하여 **배포 준비**

## 7. 릴리즈 노트 (배포노트)

소프트웨어/서비스가 출시/업데이트될 때마다 **변경 사항을 체계적으로 정리하여 제공하는 문서**

| 유형 | 목적 |
|------|------|
| **대외용** | 사용자에게 업데이트/수정 사항 공유, 지속적 피드백 반영 |
| **대내용** | 배포 관련 사항 정리, 내부 이해관계자 협조사항 관리 |

## 8. 회고 (Retrospective)

### 회고의 의미

- **사전적 의미**: 지나간 일을 돌이켜 생각하는 것
- **목적**: 좋았던 부분을 유지(Keep)하고, 아쉬웠던 점/장애물을 제거(Problem)하여, 전체적인 퍼포먼스를 개선(Try)

### 회고의 효과

- 문제를 빠르게 발견하고 **실천 과제(Action)**를 빠르게 결정
- 팀원 각자가 느끼는 바를 공유하여 **팀 내부 소통 활성화**
- **점진적 발전** 달성

### KPT 회고 방식

| 구분 | 내용 |
|------|------|
| **Keep** | 현재 만족하고 있는 부분, 계속 이어갔으면 하는 부분 |
| **Problem** | 불편하게 느끼는 부분, 개선이 필요하다고 생각되는 부분 |
| **Try** | Problem에 대한 해결책, 다음 회고 때 판별 가능한 것, 당장 실행 가능한 것 |

**핵심 포인트**: 실행 가능한 **Action Item 도출**

### 회고 예시 (실제 Action Item)

- 스토리 중복을 없애고 웹/서버 개발자 간 **하나의 스토리로 통합**
- UX/QA 직군의 **업무집중 기간 운영**
