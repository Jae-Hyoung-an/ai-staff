# AGENTS.md 설계 철학

> 이 문서는 AGENTS.md 템플릿의 설계 철학과 목적을 설명합니다.
> 템플릿을 수정하거나 확장할 때 참고하세요.

---

## 1. 왜 이 템플릿을 만들었는가

### 배경

AI 도구(Cursor, ChatGPT 등)를 업무에 활용하려는 사람들이 많지만, 대부분 "채팅창에 질문하는 수준"에서 멈춥니다. 이유는 명확합니다:

- AI가 내 업무 맥락을 모름
- 매번 같은 설명을 반복해야 함
- 결과물의 품질이 일정하지 않음

**AGENTS.md는 이 문제를 해결합니다.**

### 핵심 아이디어

```
빈 폴더 + AGENTS.md 하나 = AI와 함께 성장하는 지식 베이스
```

이 파일 하나를 빈 폴더에 넣고 Cursor로 열면:
1. AI가 온보딩 질문을 통해 사용자를 파악
2. 사용자 맞춤형 폴더 구조 제안
3. 대화하며 지식(knowledge/)이 쌓임
4. 쌓인 지식이 AI의 답변 품질을 높임 (복리 효과)

---

## 2. 지향점

### 2.1 점진적 진화

| 단계 | AI 역할 | 사용자 경험 |
|------|---------|------------|
| 초기 (0-2주) | 개인 비서 | 질문 응답, 문서 작성 도움 |
| 중기 (2-8주) | 프로젝트 파트너 | 구조화 제안, 패턴 발견 |
| 후기 (2개월+) | Chief of Staff | 비판적 피드백, 전략 논의 |

**핵심**: 처음부터 복잡한 구조를 강요하지 않음. 사용자와 함께 자연스럽게 진화.

### 2.2 복리 효과 (Compounding)

```
Context 축적 + 반복 패턴 규칙화 = 가속화
```

- **knowledge/**: 문서가 쌓일수록 AI가 맥락을 더 잘 이해
- **rules/user.md**: 선호도가 쌓일수록 AI 답변이 사용자 스타일에 맞춰짐
- **rules/**: 반복 패턴이 규칙으로 정착되면 같은 실수 반복 없음

### 2.3 역할 전환: "하는 사람" → "검토하는 사람"

**AS-IS**: 사람이 직접 작성 (80%) + 검토 (20%)
**TO-BE**: AI 초안 생성 (90%) + 사람이 검토/피드백 (10%)

이 전환이 일어나려면:
- AI가 충분한 맥락을 가져야 함 → knowledge/ 축적
- AI가 사용자 스타일을 알아야 함 → rules/user.md
- 피드백이 누적되어야 함 → 규칙화

## 3. 대상

### 3.1 주요 타겟

- **비개발자**: 코드를 몰라도 AI와 협업 가능하도록 설계
- **다양한 직군**: HR, 전략, 디자인, 재무, PM 등
- **역할 중립적**: 특정 역할(CEO 등)을 전제하지 않음

### 3.2 사용 시나리오

| 사용자 | 활용 예시 |
|--------|----------|
| HR 담당자 | JD 작성, 면접 질문 생성, 정책 문서 관리 |
| 전략 기획자 | 시장 분석, 경쟁사 리서치, 전략 문서 |
| 디자이너 | 디자인 스펙 작성, 브랜드 가이드 관리 |
| PM/PO | PRD 작성, 백로그 정리, 회의록 |
| 재무 담당자 | 예산 분석, 보고서 작성, 정책 정리 |

## 4. 핵심 원칙

### 4.1 AI 마인드셋

| 원칙 | 설명 |
|------|------|
| **비판적 분석** | 허점이 보이면 직접 지적 |
| **대안 제시** | 문제 지적 시 반드시 대안 함께 제안 |
| **적극적 제안** | 요청받지 않아도 놓친 부분 선제 제안 |
| **맥락 기억** | 이전 대화와 문서를 참고해서 일관성 유지 |

### 4.2 금지 사항

- ❌ **무비판적 동의**: "좋은 생각입니다"로 끝내기
- ❌ **추상적 조언**: 구체적 액션 없는 제안
- ❌ **자의적 대량 변경**: 확인 없이 많은 파일 수정
- ❌ **자의적 Git Commit**: 명시적 요청 시에만
- ❌ **기술 용어만 사용**: 비개발자가 이해 못하는 설명

### 4.3 문서 작성 원칙 (ceo-preferences에서 추출)

| 원칙 | 설명 |
|------|------|
| **한글 우선** | 모든 응답은 한글, 영어 키워드는 병기 |
| **상세함 유지** | 요약만 하지 말고 원본 내용 최대한 보존 |
| **일관성 검토** | 문서 간 메시지/용어 일관성 확인 |
| **삭제 지양** | 삭제보다 이동/archive 우선 |
| **점진적 진행** | 한 번에 큰 변화 X, 단계별로 |


## 6. 온보딩 설계 철학

### 6.1 트리거

`knowledge/` 폴더가 비어있거나 파일이 3개 미만일 때 자동으로 온보딩 시작.

### 6.2 수집 정보 (5단계)

1. **기본**: 역할, 주요 업무
2. **목표**: AI와 함께 이루고 싶은 것 Top 3~5 ⭐
3. **스타일**: 응답 선호, 자주 쓰는 용어, 피할 표현
4. **조직**: 팀 구조, 협업 대상, 의사결정 흐름
5. **도메인**: 핵심 개념, 참고 자료, 반복 패턴

### 6.3 왜 "목표 Top 3~5"가 중요한가

- AI가 지향점을 가지고 대화할 수 있음
- 관련 없는 제안을 걸러낼 수 있음
- 진척 상황을 함께 추적할 수 있음

## 7. user.md 업데이트 철학

### 7.1 언제 업데이트하나

| 트리거 | 예시 |
|--------|------|
| 반복 패턴 3회 이상 | "표 형식으로 정리해줘"를 매번 요청 |
| 새로운 선호 발견 | "이모지 쓰지 마"라고 피드백 |
| 역할/목표 변화 | 승진, 업무 변경, 새 프로젝트 |

### 7.2 알림 방식

AI가 패턴을 발견하면:
```
"이 패턴을 user.md에 추가할까요?"
- 응답 스타일: 표 형식 선호
```

사용자 승인 후에만 반영. 자의적 수정 금지.

## 8. 확장 가이드

### 8.1 규칙 추가 시점

| 상황 | 행동 |
|------|------|
| 같은 패턴 3번 반복 | "이걸 규칙으로 정리할까요?" 제안 |
| knowledge/ 파일 10개+ | `rules/writing.md` 추가 제안 |
| tools/ 사용 시작 | `rules/tools.md` 추가 제안 |
| 새 영역 집중 작업 | 해당 영역 규칙 파일 제안 |

### 8.2 폴더 구조 개선 시점

| 상황 | 행동 |
|------|------|
| 한 폴더에 파일 15개+ | 하위 폴더 분리 제안 |
| 파일 찾기 어려움 호소 | 구조 리팩토링 제안 |
| 새 카테고리 등장 | 폴더 추가 제안 |
